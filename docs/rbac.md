# TIDE RBAC Reference

**Audience:** Cluster operators and security reviewers auditing TIDE's
Kubernetes permissions — what the orchestrator can do in your cluster, what
the subagent pods can do, what the dashboard can do, and how to scope all
three down to specific Project namespaces.

**Status:** v1.0. **AUTH-02** (per-namespace RoleBinding template, namespace-per-project
tenancy) and **AUTH-03** (no cluster-wide wildcards in the orchestrator
ServiceAccount) are locked. Helm chart pair `tide` + `tide-crds` ships the
RBAC manifests described below.

**Scope of this doc:**

- Per-Kind verbs the orchestrator's ServiceAccount holds across the six TIDE CRDs
- Auxiliary ServiceAccounts (subagent, dashboard) and what they're allowed to do
- The opt-in per-namespace RoleBinding template (D-X4, AUTH-02 catch-up) for multi-Project installs
- Conversion-webhook RBAC posture (D-X7 — scaffolded but no-op in v1.0)
- Auditing recipes for verifying any of the above with `kubectl` and `make` targets

---

## Per-Kind verbs

TIDE's RBAC posture is **namespace-scoped per Project, no cluster-wide
wildcards**. The CI lock `make verify-no-rbac-wildcards` (Phase 1) walks
`config/rbac/` — generated by `controller-gen` from kubebuilder markers in
`internal/controller/*_controller.go` — and fails the build if any rule
contains `verbs: *` or `resources: *`. A second lock,
`make verify-rbac-marker-discipline`, refuses wildcard markers in source so
the violation never reaches the generated manifest in the first place.

The orchestrator runs as ServiceAccount **`tide-controller-manager`** in the
TIDE install namespace (`tide-system` by default). At install time the Helm
chart binds that SA to the **`manager-role`** ClusterRole
(`charts/tide/templates/manager-rbac.yaml`), which enumerates the verbs
below. This is the SA the reconcilers use for every reconcile call.

### Manager-role: explicit verbs the controller binds to

| Resource (apiGroup `tideproject.k8s`) | Body verbs | Status verbs | Finalizer verbs |
|---------------------------------------|------------|--------------|-----------------|
| `projects` / `projects/status` / `projects/finalizers` | create, delete, get, list, patch, update, watch | get, patch, update | update |
| `milestones` / `milestones/status` / `milestones/finalizers` | create, delete, get, list, patch, update, watch | get, patch, update | update |
| `phases` / `phases/status` / `phases/finalizers` | create, delete, get, list, patch, update, watch | get, patch, update | update |
| `plans` / `plans/status` / `plans/finalizers` | create, delete, get, list, patch, update, watch | get, patch, update | update |
| `tasks` / `tasks/status` / `tasks/finalizers` | create, delete, get, list, patch, update, watch | get, patch, update | update |
| `waves` / `waves/status` / `waves/finalizers` | create, delete, get, list, patch, update, watch | get, patch, update | update |

The controller also holds a small set of standard-API verbs the reconcilers
need at runtime (also in `manager-role`):

| Resource | apiGroup | Verbs | Why |
|----------|----------|-------|-----|
| `events` | `""` (core) | create, patch | Record CRD-level Events for operators (`kubectl describe`). |
| `persistentvolumeclaims`, `pods`, `secrets` | `""` (core) | get, list, watch | Mount the workspace PVC, read Secret credentials, observe dispatched Pods. |
| `jobs` | `batch` | create, delete, get, list, patch, update, watch | Dispatch subagent Jobs (Phase 2 dispatch contract). |

No `*` appears in any rule. No `apiGroups: ["*"]`, no `resources: ["*"]`.
The CI gate is the lock; this table is the human-readable reference.

### Per-Kind admin / editor / viewer ClusterRoles (kubebuilder convention)

In addition to `manager-role`, the chart ships **18 ClusterRole templates**
generated by the kubebuilder admin/editor/viewer scaffolding — three roles
per CRD Kind, one set per Kind:

| ClusterRole template file | Tier | Verbs on body | Verbs on `/status` |
|---------------------------|------|---------------|--------------------|
| `project-admin-rbac.yaml` | admin | `*` | get |
| `project-editor-rbac.yaml` | editor | create, delete, get, list, patch, update, watch | get |
| `project-viewer-rbac.yaml` | viewer | get, list, watch | get |
| `milestone-admin-rbac.yaml` | admin | `*` | get |
| `milestone-editor-rbac.yaml` | editor | create, delete, get, list, patch, update, watch | get |
| `milestone-viewer-rbac.yaml` | viewer | get, list, watch | get |
| `phase-admin-rbac.yaml` | admin | `*` | get |
| `phase-editor-rbac.yaml` | editor | create, delete, get, list, patch, update, watch | get |
| `phase-viewer-rbac.yaml` | viewer | get, list, watch | get |
| `plan-admin-rbac.yaml` | admin | `*` | get |
| `plan-editor-rbac.yaml` | editor | create, delete, get, list, patch, update, watch | get |
| `plan-viewer-rbac.yaml` | viewer | get, list, watch | get |
| `task-admin-rbac.yaml` | admin | `*` | get |
| `task-editor-rbac.yaml` | editor | create, delete, get, list, patch, update, watch | get |
| `task-viewer-rbac.yaml` | viewer | get, list, watch | get |
| `wave-admin-rbac.yaml` | admin | `*` | get |
| `wave-editor-rbac.yaml` | editor | create, delete, get, list, patch, update, watch | get |
| `wave-viewer-rbac.yaml` | viewer | get, list, watch | get |

**Important:** the `*` body verb in the **admin** tier is a kubebuilder
scaffolding convention intended for binding to **human cluster admins**
(via RoleBinding or ClusterRoleBinding chosen by the operator) — NOT for
the controller's ServiceAccount. The controller's SA never binds to any
`*-admin-role`; it binds only to `manager-role`, whose verbs are the
explicit per-Kind list above. The presence of `*` in the admin-tier
templates does NOT violate AUTH-03 — `make verify-no-rbac-wildcards`
deliberately scans only `config/rbac/` (the controller's own RBAC),
not the operator-facing admin/editor/viewer ClusterRole scaffolding.

If you bind a human user or external automation to a `*-admin-role`,
you are extending TIDE's RBAC surface — review the binding the same way
you'd review any cluster-admin grant.

## Auxiliary ServiceAccounts

TIDE ships **three** ServiceAccounts. Only the orchestrator holds
mutate-shaped verbs; the other two are zero-RBAC (subagent) or
read-only (dashboard).

### `tide-controller-manager` (orchestrator)

- **Namespace:** `tide-system` (the TIDE install namespace).
- **Template:** `charts/tide/templates/serviceaccount.yaml`.
- **Binds to:** `manager-role` (see verbs table above) via the
  ClusterRoleBinding in `charts/tide/templates/manager-rbac.yaml`.
- **Reconciles:** all six TIDE CRDs across all watched namespaces (the
  `--watch-namespace` flag on `cmd/manager` narrows the watch scope per
  Phase 1 D-A4 namespace-filter predicate).

### `tide-subagent` (subagent pods — zero K8s API verbs)

- **Namespace:** `tide-system`.
- **Template:** `charts/tide/templates/serviceaccount-subagent.yaml`.
- **Verbs:** zero. No Role, no RoleBinding. The subagent never calls the
  Kubernetes API — its only side effects are filesystem writes to
  `/workspace` and stdout/stderr captured by the dispatched Job.
- **Rationale:** Phase 2 D-A4 / threat T-02-12-04. Locking the subagent
  out of the K8s API at SA level (not just by Pod SecurityContext) is the
  load-bearing mitigation against "subagent escapes its task scope and
  tampers with other Project CRDs."

### `tide-dashboard` (dashboard server — read-only)

- **Namespace:** `tide-system`.
- **Template:** `charts/tide/templates/dashboard-rbac.yaml` (gated by
  `dashboard.enabled` Helm value; disabled by default).
- **Verbs:** `get`, `list`, `watch` on the six TIDE CRDs (no `/status`,
  no mutations) plus `get`, `list`, `watch` on `pods` and `get` on
  `pods/log` (for SSE log streaming in DASH-04).
- **CI lock:** `make helm-rbac-assert` (Phase 4 T-04-D2 mitigation) walks
  the rendered dashboard ClusterRole at chart-render time and exits
  non-zero if any verb falls outside `{get, list, watch}`. Defense in
  depth against silent privilege escalation in PRs.

## Per-namespace RoleBinding (AUTH-02 catch-up)

**One TIDE install per cluster; each Project runs in its own namespace
with namespace-scoped RBAC.** The orchestrator SA
(`tide-controller-manager` in `tide-system`) is the **subject**; the
per-namespace RoleBinding lives in the **Project's namespace** and grants
the SA the per-Kind ClusterRole verbs **scoped to that namespace only**.
This is the standard Kubernetes pattern for binding a central
ServiceAccount to multiple tenant namespaces — RoleBinding lives where
the resources are, subjects reach back to where the SA lives.

The chart ships a template that does this automatically when an
operator opts in by listing the Project namespaces in Helm values:

```yaml
# values.yaml override — opt-in by listing project namespaces
projectNamespaces:
  - tide-customer-acme
  - tide-customer-globex
```

Then install or upgrade:

```bash
helm upgrade --install tide oci://ghcr.io/jsquirrelz/tide-charts/tide \
  --version 1.0.0 \
  --namespace tide-system \
  --create-namespace \
  --set 'projectNamespaces={tide-customer-acme,tide-customer-globex}'
```

For each entry in `projectNamespaces`, the template
`charts/tide/templates/per-namespace-rolebinding.yaml` renders one
`RoleBinding` in that namespace. Each binding has:

- `metadata.namespace: <project-namespace>` (RoleBinding lives in the
  tenant namespace — that's what scopes the verbs down)
- `subjects[].kind: ServiceAccount` with
  `name: tide-controller-manager` and
  `namespace: tide-system` (subject points back to the operator
  namespace per Research Open Question Q6 RESOLVED — central SA
  pattern)
- `roleRef.kind: ClusterRole` binding to the appropriate per-Kind
  ClusterRole (Plan 05-13 finalizes the aggregation strategy — see the
  template for the exact ClusterRole reference)

**The default `projectNamespaces: []` means no per-namespace
RoleBindings are created out of the box.** This is opt-in by design
(D-X4); single-Project installs that use the default `tide-system`
namespace or that watch all namespaces from `manager-role`'s
ClusterRoleBinding need nothing more. The per-namespace template
exists for multi-Project installs that want each Project's reconcile
authority limited to that Project's own namespace.

The template ships in Plan 05-13 (Wave 3) along with the
`projectNamespaces` Helm value in `hack/helm/tide-values.yaml` (the
chart-values source-of-truth — per CLAUDE.md's chart-is-fixed-contract
rule, the value is added to the SOT then propagated to
`charts/tide/values.yaml` via `bash hack/helm/augment-tide-chart.sh`).

## Conversion webhook (D-X7 — no-op for v1.0)

The CRD conversion webhook is **scaffolded but no-op for v1.0** because
v1alpha1 IS the hub schema (Phase 1 D-A3 schema lock) and no spoke
version (`v1beta1`, `v1`) exists yet. CRD-05 / Pitfall 16's mitigation
IS the scaffold itself — having the conversion-webhook plumbing wired
end-to-end at install time means the v1.x activation work is "write
`ConvertTo`/`ConvertFrom` bodies in Go" rather than "add the chart
templates, the cert plumbing, and the webhook service."

**What the chart ships at v1.0:**

- A `webhook-service` (`charts/tide/templates/webhook-service.yaml`)
  exposes the controller manager's webhook port. The validating webhook
  for Plan and Wave (CEL-and-Go pre-admission validation, Phase 1
  CRD-04) uses this same service.
- A `serving-cert` (`charts/tide/templates/serving-cert.yaml`) — a
  cert-manager `Certificate` mounted into the controller manager Pod —
  serves TLS for the webhook port. Same cert serves the validating
  webhook and the (future) conversion webhook; cert-manager rotation
  applies to both.
- A `validating-webhook-configuration`
  (`charts/tide/templates/validating-webhook-configuration.yaml`) wires
  the Plan and Wave pre-admission validators.
- **No** `conversion-webhook-configuration` is shipped — at v1.0 each
  CRD's `spec.conversion.strategy` is `None` (the kubebuilder default
  for single-version CRDs), so no separate conversion webhook
  registration is needed.

**RBAC implications:**

- The controller manager SA (`tide-controller-manager`) already has the
  verbs the conversion webhook will eventually need — it's the same SA,
  the same `manager-role` binding. Activating v1.x conversion will
  require zero RBAC changes: just the Go `ConvertTo`/`ConvertFrom`
  bodies plus the per-CRD `spec.conversion.strategy: Webhook` flip
  (and a `conversion-webhook-configuration` template).
- The cert-manager `Issuer` and `Certificate` are already serving the
  controller's webhook endpoint; the conversion webhook will reuse
  them. No new `Certificate` is required.

When v1.x activates conversion with a real spoke schema, no chart-level
RBAC change is required — only Go code and the per-CRD
`spec.conversion.strategy` flip.

## Auditing TIDE's RBAC

The recipes below verify that what's installed matches what's
documented above. Run them after `helm install` against your live
cluster.

### Inspect the orchestrator SA's bindings

```bash
# Show the orchestrator ServiceAccount itself
kubectl describe sa tide-controller-manager -n tide-system

# List every (Cluster)RoleBinding subject-bound to the orchestrator SA
kubectl get clusterrolebindings.rbac.authorization.k8s.io -o json \
  | jq -r '.items[] | select(.subjects[]?.name=="tide-controller-manager"
                              and .subjects[]?.namespace=="tide-system")
                    | .metadata.name'
kubectl get rolebindings.rbac.authorization.k8s.io -A -o json \
  | jq -r '.items[] | select(.subjects[]?.name=="tide-controller-manager"
                              and .subjects[]?.namespace=="tide-system")
                    | "\(.metadata.namespace)/\(.metadata.name)"'
```

### Verify a specific permission

```bash
# Can the orchestrator create a Project in tide-system?
kubectl auth can-i create projects.tideproject.k8s \
  --as=system:serviceaccount:tide-system:tide-controller-manager \
  -n tide-system

# Can the orchestrator patch a Wave status in a per-namespace install?
kubectl auth can-i patch waves.tideproject.k8s/status \
  --as=system:serviceaccount:tide-system:tide-controller-manager \
  -n tide-customer-acme

# Verify the subagent SA has zero verbs on TIDE CRDs (expects "no")
kubectl auth can-i list tasks.tideproject.k8s \
  --as=system:serviceaccount:tide-system:tide-subagent \
  -n tide-system

# Verify the dashboard SA is read-only (expects "no" for create, "yes" for list)
kubectl auth can-i create projects.tideproject.k8s \
  --as=system:serviceaccount:tide-system:tide-dashboard \
  -n tide-system
kubectl auth can-i list projects.tideproject.k8s \
  --as=system:serviceaccount:tide-system:tide-dashboard \
  -n tide-system
```

### Verify no wildcards in the controller's generated RBAC

```bash
# CI gate — runs in PRs (AUTH-03 / Pitfall 15 lock)
make verify-no-rbac-wildcards

# Source-side variant — verifies kubebuilder markers in
# internal/controller/*_controller.go carry no wildcards
make verify-rbac-marker-discipline

# Dashboard read-only assertion (T-04-D2 lock)
make helm-rbac-assert
```

### Render the chart locally and inspect what RBAC ships

```bash
helm template charts/tide --namespace tide-system \
    --set 'projectNamespaces={tide-customer-acme}' \
  | yq 'select(.kind == "ClusterRole" or .kind == "Role"
                or .kind == "ClusterRoleBinding" or .kind == "RoleBinding")
       | {kind: .kind, name: .metadata.name,
          ns: .metadata.namespace, rules: .rules, roleRef: .roleRef,
          subjects: .subjects}'
```

This dumps every RBAC object the chart renders for a given
`projectNamespaces` set. Useful both as a sanity check before applying
to a cluster and as evidence in a security audit.
