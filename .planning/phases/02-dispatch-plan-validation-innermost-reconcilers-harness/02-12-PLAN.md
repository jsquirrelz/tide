`charts/tide/templates/signing-secret.yaml` (NEW Helm template; idempotent via lookup):

```yaml
{{- $secret := lookup "v1" "Secret" .Release.Namespace .Values.signingKey.secretName }}
{{- if not $secret }}
apiVersion: v1
kind: Secret
metadata:
  name: {{ .Values.signingKey.secretName | default "tide-signing-key" }}
  namespace: {{ .Release.Namespace }}
  annotations:
    helm.sh/resource-policy: keep
  labels:
    {{- include "tide.labels" . | nindent 4 }}
type: Opaque
data:
  # Data key MUST be env-friendly (no dashes) so `envFrom: [secretRef: ...]` auto-populates
  # the TIDE_SIGNING_KEY env var on the Manager Deployment + credproxy sidecar (Blocker #1 fix).
  TIDE_SIGNING_KEY: {{ randAlphaNum 64 | b64enc | quote }}
{{- end }}
```

The `lookup` + `if not` guard makes installation idempotent — Helm only generates a fresh key if the Secret doesn't already exist. `resource-policy: keep` ensures `helm uninstall` doesn't delete it (so reinstalls reuse the same key). The data key is `TIDE_SIGNING_KEY` (env-friendly — dashes invalid in env var names), so `envFrom: [{secretRef: {name: tide-signing-key}}]` on both the Manager Deployment and the credproxy sidecar (Plan 08 BuildJobSpec) populates the `TIDE_SIGNING_KEY` env var directly with zero extra wiring.
