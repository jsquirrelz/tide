<objective>
Build the subagent-side harness ŌĆö the in-pod orchestrator that wraps every Subagent runtime (stub in Phase 2, Claude Code in Phase 3). The harness is the enforcement layer for per-Task caps (HARN-02), secret-pattern redaction (HARN-04), and output-path validation (HARN-05). It exposes a Runtime interface that swappable backends (stub-subagent binary in Plan 04, Claude Code in Phase 3) plug into without harness code changes ŌĆö the HARN-06 seam.

Purpose: The harness is what makes the security guarantees real. Without it, a runaway agent loop (Pitfall 8) burns the budget; a stray API key in subagent stdout (Pitfall 18) gets logged forever; an artifact written outside declared paths (Pitfall 7) bleeds context across Tasks. All three defenses live here.

Output: A `internal/harness` Go package with five concerns separated into files: orchestrator (`harness.go`), caps (`caps.go`), outputs (`outputs.go`), envelope I/O (`envelope_io.go`), redaction (`redact/redact.go` sub-package). All tests are stdlib `testing` table-driven per leaf-package convention. The stub-subagent Plan 04 binary will be wrapped by `internal/harness/harness.go.Run` in the Phase 2 Job dispatch flow (Plan 08 jobspec invokes the harness; the harness invokes the runtime).

NOTE on Phase 2 packaging (Warning #6 ŌĆö controller-side validator deviation; RESEARCH.md Responsibility Map carries the note): The harness package compiles in Phase 2 but is NOT yet invoked by Plan 08's PodJobBackend (Plan 04's stub-subagent runs as its own binary; the harness wraps it conceptually but in practice the K8s Job spec runs stub-subagent directly). The harness becomes the wrapping binary in Phase 3 when Claude Code lands behind the Runtime interface ŌĆö at that point Plan 08's jobspec switches its container `command` from `stub-subagent` to `harness` with `Runtime=ClaudeCode`. For Phase 2: the harness package SHIPS its API + tests + unit-level guarantees so Phase 3 has zero code-shape surprises. **The output-path validator (`outputs.Validate`) ships as a library here and is INVOKED FROM PLAN 09'S CONTROLLER-SIDE `handleJobCompletion` (Warning #5 fix ŌĆö wires HARN-05 into the dispatch chain).** The Manager pod can see the per-Task workspace slice because the shared `tide-projects` PVC is mounted at `/workspaces` on Manager (Blocker #2/#3 architecture); `handleJobCompletion` computes `taskWorkspaceRoot := "/workspaces/{project.UID}/workspace"` and calls `outputs.Validate(taskWorkspaceRoot, runStart, declaredOutputPaths)`. The redactor is meanwhile a library-only concern in Phase 2 ŌĆö it'll be wired when the harness binary becomes the Pod's actual entrypoint in Phase 3. The wall-clock cap is enforced by the K8s `Job.spec.activeDeadlineSeconds` field set by Plan 08 in Phase 2 (the harness's `context.WithTimeout` enforcement layer comes online when the harness is the entrypoint ŌĆö Phase 3).

// RedactingWriter wraps an io.Writer; bytes are redacted before forwarding.
// Maintains a tail-keep buffer to handle pattern matches across Write calls.
// Pitfall A (RESEARCH.md) prevention.
type RedactingWriter struct {
    dst      io.Writer
    tail     []byte
    patterns []*regexp.Regexp
