<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External (helm operator) → cluster RBAC | tide-push SA is the bridge between the push Job pod and the git creds Secret. Least-privilege (`secrets get` only) means the SA cannot enumerate Secrets, cannot read other Secrets, cannot write any K8s object. |
| External user → docs | docs/git-hosts.md influences how PATs are scoped at the upstream host. Recommending "fine-grained" tokens with minimum scopes reduces blast radius if a PAT is exfiltrated (T-301). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-301 (PAT exfiltration / over-scoped token) | docs/git-hosts.md PAT recipes | mitigate | Docs recommend fine-grained PATs with `repo:write` (GitHub) or `write_repository` (GitLab) — minimum scope. Operator following the docs ends up with a least-privilege token; if exfiltrated, blast radius is bounded to the target repo. |
| T-304 (gitleaks override RBAC) | charts/tide/templates/push-rbac.yaml | mitigate | tide-push SA's Role grants `secrets get` only — no ConfigMap verbs. Reading the per-Project gitleaks ConfigMap happens via the push Job's PodSpec volume mount (K8s-native; doesn't need RBAC on the consuming SA). Plan 03-06's push Job volume mounts the ConfigMap; this plan's RBAC only covers the Secret read path. |
| (Cross-namespace Secret access) | Tampering / Elevation | tide-push SA scope | mitigate | tide-push SA is namespace-scoped (Role, not ClusterRole). It can only access Secrets in its own namespace. v1.0 deployment pattern: Projects run in tide-system OR the operator replicates the SA into each Project namespace. Document this in values.yaml + git-hosts.md. |
</threat_model>
