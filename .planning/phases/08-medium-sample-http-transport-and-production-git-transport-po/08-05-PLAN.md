## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cluster → git-http-server Service | Anonymous push capability; only in-cluster pods can reach the ClusterIP Service |
| push Job → GIT_PAT guard | Relaxed guard for http:// URLs; only affects anonymous in-cluster push |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-05-01 | Tampering | git-http server anonymous push from outside cluster | mitigate | Service type is ClusterIP (not NodePort/LoadBalancer); unreachable from outside the cluster; anonymous push intentional for demo-only in-cluster context |
| T-08-05-02 | Tampering | git-http server push from pods in OTHER namespaces | accept | ClusterIP is reachable from any pod in the cluster (not namespace-isolated); in-cluster DNS traverses namespace boundaries. For a demo repository this is acceptable — operators can add NetworkPolicy if isolation is required. |
| T-08-05-03 | Elevation of Privilege | GIT_PAT guard bypass for http:// URLs in production | mitigate | Guard only relaxes for http:// scheme; https:// and git@ still require PAT. Production TIDE installations use https:// (CEL enforced); the demo uses http://. The scheme-conditional guard makes the production contract explicit. |
| T-08-05-04 | Elevation of Privilege | git-http server running as root | mitigate | Dockerfile USER 1000 (nonroot user via adduser -D -u 1000 nonroot); Deployment securityContext runAsNonRoot: true, runAsUser: 1000 |
| T-08-05-05 | Denial of Service | demo-remote-pvc RWO mount conflict (init Job + server) | mitigate | Documented in apply sequence step 5: kubectl wait --for=condition=Complete job/demo-remote-init before applying git-http-server Deployment; README enforces ordering |
