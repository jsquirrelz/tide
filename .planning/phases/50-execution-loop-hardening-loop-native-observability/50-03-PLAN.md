<objective>
Extend the dual Prometheus cardinality guard (D-06 / OBS-02): grow the `metriccardinality` static analyzer's forbidden-label set from the single `"task"` literal to the full run-ID-shaped list, and extend `wave_label_test.go`'s runtime source-grep to match — proving run IDs can never enter a Prometheus label (ROADMAP criterion #5's label-cardinality proof).

**Decision (resolves RESEARCH Open Question 3):** Phase 50 adds NO new Prometheus metric — guard-hardening only. Per LOOP-03 and five-loop-model §77, run-native detail lives in traces; the first loop-scoped bounded-label metric waits for Phase 51's real consumer (`EvaluationSummary.Decision`/`LoopStatus.ExitReason`). This matches the repo's "never ship a speculative superset ahead of a real consumer" principle.

Purpose: an agent-controlled or unbounded string reaching a Prometheus label is a cardinality-explosion DoS. This guard is a security control (T-50-06), enforced at `go vet` time (analyzer) AND test time (source-grep) — the same two-layer discipline already proven for `"task"`.
Output: extended analyzer + fixtures + runtime guard, green under `go test` and `make lint`.
</objective>
