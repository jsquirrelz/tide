<objective>
Close WR-03 (16-REVIEW.md Warning; 16-VERIFICATION.md artifact row "⚠️ ORPHANED"): `internal/config/config.go` parses a `prometheusEndpoint` YAML key and applies a `PROM_ENDPOINT` env override, with three tests pinning the behavior — but no binary consumes `cfg.PrometheusEndpoint`. The dashboard (the endpoint's only consumer) reads `os.Getenv("PROM_ENDPOINT")` directly in `cmd/dashboard/main.go:156` and never calls `config.Load`; the manager calls `config.Load` but never reads the field.

Direction decision (evidence-based, per gap-closure directive — this supersedes the 16-CONTEXT TELEM-01 wiring note that introduced the field):
- The chart renders `prometheusEndpoint` into ZERO templates (`grep -rn prometheusEndpoint charts/tide/templates/` is empty) — the manager's `/etc/tide/config.yaml` ConfigMap never carries the key.
- The dashboard Deployment (charts/tide/templates/dashboard-deployment.yaml) mounts NO config file (`readOnlyRootFilesystem: true`, no volumes) — `config.Load("/etc/tide/config.yaml")` would error in the dashboard pod.
- The chart is a FIXED contract (no chart behavior changes), so the "wire the dashboard to load the config file" alternative is permanently unreachable.

Therefore the field genuinely serves no binary → DELETE the dead config surface and align the docs to the working env-only mechanism (review WR-03's option B). The Helm `prometheus.endpoint` value keeps working unchanged — it only ever worked via the env injection.

Purpose: TELEM-01's operator surface stops lying — `helm value → PROM_ENDPOINT env → proxy` is the one documented, tested, working path.
Output: dead field/override/tests removed; MILESTONE.md and the chart template comment corrected; manager and dashboard binaries behaviorally unchanged.
