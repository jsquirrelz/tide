### FLAG 2 — Planner caps default contradicts P1.3's structural assumption (Plan 05 vs Plan 03)

**Severity:** WARNING.

**Description:** Plan 03's `DefaultCaps(caps)` returns `WallClockSeconds = 300` (the executor floor) when caps is nil or zero. Plan 05's must_haves truth #6 says "Planner caps default to `{WallClockSeconds: 600, Iterations: 20}` when `Project.Spec.Caps` is nil." Plan 05's action body at Step 4 says:

> `plannerCaps := podjob.DefaultCaps(project.Spec.Caps) // 300s floor + planner default if nil — TBD with P1.3`

The `TBD with P1.3` is a deferred-decision marker. The plan body acknowledges that the "planner-specific caps required a new `DefaultPlannerCaps` helper" is a possibility, recommending the executor add a planner variant. But Plan 03 doesn't add one — Plan 03 only adds `DefaultCaps` (executor-floor 300s).

**Why it matters:** if executor runs Plan 03 first (Wave 1) then Plan 05 (Wave 2/3), Plan 05's call to `podjob.DefaultCaps(project.Spec.Caps)` will return 300s (executor floor), NOT 600s (planner default). Plan 05's must_haves truth #6 will fail because the planner caps will be 300s not 600s. The executor will either:

1. Discover this mismatch and add `DefaultPlannerCaps` mid-plan-05 (cascade risk; not budgeted)
2. Accept the 300s planner default silently (violates Plan 05's truth; planner Jobs get killed earlier than expected)
3. Hand-roll the 600s value inline at each up-stack reconciler call site (violates the "single source of truth" intent of Plan 03)

**Fix:** decide upfront in Plan 03 (or a small predecessor edit to Plan 03) to either:
- Add `DefaultPlannerCaps()` helper alongside `DefaultCaps()` in Plan 03's caps.go
- Or change `DefaultCaps` signature to accept a JobKind discriminator: `DefaultCaps(caps, kind JobKind)`
- Or accept that planner caps are different from executor caps and document why the single-helper pattern doesn't apply to planners (which weakens P1.3's structural goal)
