# Phase 04.1 Plan 09: Budget Rolling-Window Reset Summary

**One-liner:** `RollingWindowCapCents` enforcement implemented via `MaybeResetWindow` with configurable `RollingWindowDuration` field; WR-02 doc-only caveat deleted.

## What Was Built

### 1. Schema addition (`api/v1alpha1/project_types.go`)

`BudgetConfig` gains a new optional field:

```go
RollingWindowDuration *metav1.Duration `json:"rollingWindowDuration,omitempty"`
```

- Backward-compatible: `omitempty`, pointer type — existing Projects without the field default to 24h window in `MaybeResetWindow`.
- The WR-02 doc-only caveat block (formerly lines 90–96) is deleted. `RollingWindowCapCents` is now a live enforcement field.
- Updated `RollingWindowCapCents` docstring to describe the production enforcement path.
- `metav1.Duration` was already imported; no new imports needed.

**Codegen deviation:** `+kubebuilder:validation:Pattern` cannot be applied to `*metav1.Duration` (a struct, not a string type) — controller-gen rejects it with "must apply pattern to a textual value." The minimum-1h constraint is documented in the field comment and enforced semantically. This is the correct behavior per controller-gen's marker type rules.

### 2. `MaybeResetWindow` (`internal/budget/tally.go`)

```go
func MaybeResetWindow(ctx context.Context, c client.Client, project *tidev1alpha1.Project, now time.Time) (bool, error)
```

- Returns `(false, nil)` for no-ops: nil project, zero/negative `RollingWindowCapCents`, nil `WindowStart`, window not yet elapsed.
- When window has elapsed: zeros `CostSpentCents` + `TokensSpent`, advances `WindowStart` to `now`, patches via `client.MergeFrom` (PIT-4 race safety).
- Default window: 24h when `Spec.Budget.RollingWindowDuration` is nil.
- Returns `(true, nil)` on successful reset; `(false, err)` on patch failure.
- Idempotent: concurrent reconciles both resetting are harmless (MergeFrom+Patch).

### 3. `IsCapExceeded` extension (`internal/budget/cap.go`)

Extended to check `RollingWindowCapCents` after the existing `AbsoluteCapCents` check:

```go
if project.Spec.Budget.RollingWindowCapCents > 0 && project.Status.Budget.CostSpentCents > project.Spec.Budget.RollingWindowCapCents {
    return true
}
```

Backward compatible: zero/negative `RollingWindowCapCents` = unlimited (same pattern as `AbsoluteCapCents`).

### 4. `ProjectReconciler.handleBudgetGate` wiring (`internal/controller/project_controller.go`)

At the start of `handleBudgetGate`, before the bypass/cap checks:

```go
// Phase 04.1 P4.1: reset rolling window if elapsed. Failures are logged
// non-fatal (Pitfall C pattern) — never block dispatch on a tally op.
if _, err := budget.MaybeResetWindow(ctx, r.Client, project, now); err != nil {
    logger.Error(err, "budget window reset failed (non-fatal)")
}
```

The post-reset `CostSpentCents` value is what `IsCapExceeded` sees — correct ordering. Reset failures use `logger.Error` (not return-the-error) per Pitfall C: budget tally failures must never block dispatch.

### 5. Codegen (`config/crd/bases/tideproject.k8s_projects.yaml`, `api/v1alpha1/zz_generated.deepcopy.go`)

- `make manifests` updated the CRD yaml: `rollingWindowDuration` field schema present (type: string, format: duration).
- `make generate` updated deepcopy: `DeepCopyInto` for `BudgetConfig` handles nil pointer for `RollingWindowDuration`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Removed invalid `+kubebuilder:validation:Pattern` marker from `*metav1.Duration` field**
- **Found during:** Task 1, Step 5 (`make manifests` failure)
- **Issue:** `+kubebuilder:validation:Pattern` marker cannot be applied to struct types; controller-gen emits "must apply pattern to a textual value, found type" and exits non-zero.
- **Fix:** Removed the Pattern marker; documented the 1h minimum constraint in the field comment as a semantic constraint enforced by the reconciler.
- **Files modified:** `api/v1alpha1/project_types.go`
- **Impact:** CRD doesn't have schema-level pattern validation for `rollingWindowDuration`. This is acceptable — `metav1.Duration` fields in kubebuilder projects conventionally rely on semantic validation at the reconciler layer (the K8s API machinery validates the JSON → struct parse at admission time anyway).
