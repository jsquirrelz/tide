## Summary

Phase 04.1 is a closeout phase, not new architecture. The shape of the work is already locked by ROADMAP.md (13 success criteria) and `docs/audit-report-2026-05-20-merged.md` (4 P1 + 4 P2 + 2 P3 + 4 P4 + 15 UAT items). This research document tells the planner **how to land each item** with the smallest correct diff and the lowest cascade risk.

Three findings reshape the plan from what the audit body suggests:

1. **P1.2 (planner Job contract) is the only item with real architectural weight.** Every other P-item is a 5-50 line targeted fix. The planner-Job refactor is the single decision point in the phase. The recommended factoring is documented below in detail.

2. **Two Phase 03 UAT items (item 3 "mid-stack boundary push" and item 4 "leak counter wiring") are ALREADY CLOSED in Phase 4 source.** `project_controller.go:484-557` parses the push-result envelope and fires `SecretLeakBlockedTotal` on `reason=leak-detected`; `boundary_push.go` + `gates.BoundaryDetected` wires mid-stack pushes. These UAT items should be **re-verified now** and flipped to `pass`, not re-implemented. The verification report is stale.

3. **Wave 5/6/7 UAT cascade risk is real.** Phase 02.2 ran 12 plans to close one timing fix because each re-run surfaced a new harness/wiring gap. Plan the UAT runner waves with the same expectation — `/gsd-sdk query state.begin-phase` reset behavior, worktree cwd-drift, and the wave-N filter for closeout plans all apply here.

**Primary recommendation:** Structure the phase as the planner-hint suggests (Wave 1–7) but **add a Wave 0 "tooling readiness gate"** that confirms Docker + kind + Chrome DevTools MCP availability and pre-builds + loads images before Wave 5–7 UAT plans dispatch. This pre-empts the most common 02.2-style cascade class (missing prerequisite surfacing 3 plans in).

## Pitfall Map (Phase 02.2 lessons applied to Phase 04.1)

Phase 02.2 ran 12 plans to close a single timing fix. Eleven of those were cascades from missed prerequisites or unverified assumptions. The patterns below are direct ports from `STATE.md` Accumulated Context + CLAUDE.md Operating Notes. Each is high-impact for Phase 04.1.

### PIT-1: Chart values.yaml is FIXED contract — binary catches up to chart [VERIFIED: CLAUDE.md "Stack Anti-patterns" + STATE.md 02.2 cascade-10 note]

**Applies to:** P1.2 (planner Job contract), P4.2 (cred-proxy allowlist), P4.3 (PROD_OVERRIDE_REQUIRED comments).
**Trap:** Renaming or restructuring an existing `values.yaml` key to fit a new code shape.
**Discipline:** Add NEW keys only. If the cred-proxy allowlist needs a new `Spec.Providers[*].AllowedRoutes` field, the chart values addition is purely additive (`providers: []` already exists; the new sub-field is non-breaking). P4.3's `PROD_OVERRIDE_REQUIRED` change is comment-only and touches no values.

### PIT-2: `gsd-sdk query state.begin-phase` resets STATE.md body fields [VERIFIED: CLAUDE.md Operating Notes]

**Applies to:** Phase 04.1 kickoff.
**Trap:** Editing STATE.md "Current Position" or "Last activity" BEFORE `state.begin-phase` runs; the SDK then clobbers the edit.
**Discipline:** Plan ordering — when the planner authors Plan 04.1-01 it must call `state.begin-phase` FIRST and then any STATE.md body updates happen AFTER. The plans themselves never edit STATE.md body fields directly; the executor only edits source.

### PIT-3: "Wave Ready check" doesn't exist — don't fabricate runtime gate framings [VERIFIED: STATE.md cascade-5 debug note]

**Applies to:** Wave 5/6/7 UAT debug sessions.
**Trap:** When a UAT spec fails, hypothesizing a code path that "should" handle a runtime condition without grepping the codebase first.
**Discipline:** If a UAT cascades, the FIRST debugging move is `kubectl logs deploy/tide-controller-manager -n tide-system --tail=200` + `kubectl describe <crd>` + `kubectl get <crd> -o yaml`. THEN read source. The cascade-5 session lost 4 plans to "Wave Ready check" hypothesis before reading manager logs.

### PIT-4: ResourceVersion-conflict exponential backoff was real (cascade-8) [VERIFIED: STATE.md "Phase 02.2 cascade-8" note]

**Applies to:** P1.4 (first-Project fallback fix).
**Trap:** When removing the fallback, two concurrent reconciles (Task vs Plan) can both call `r.Get(Project)` and one of them races on the patch.
**Discipline:** The fix MUST use `client.MergeFrom(project.DeepCopy())` + `Patch` (not `Update`) to handle concurrent reconciles. If the `ParentUnresolved` condition is set during a race, the next reconcile retry will pick up the resolved Project and clear it.

### PIT-5: Worktree cwd-drift before merge [VERIFIED: CLAUDE.md Operating Notes]

**Applies to:** Every plan in this phase that runs in an executor worktree.
**Trap:** Executor agents occasionally write artifacts to main's working tree instead of the worktree's, producing duplicated commits at merge time.
**Discipline:** Pre-merge check `git status --porcelain | grep '^??' | grep "<plan>-"`. If non-empty, diff against worktree branch's commits; remove duplicates before `git merge`.

### PIT-6: `--wave N` filter when closeout plan is gated on BLOCKED artifact [VERIFIED: CLAUDE.md Operating Notes]

**Applies to:** Plan 04.1-final (ROADMAP/STATE closeout, gated on every prior plan's VERIFICATION).
**Trap:** `--auto` mode auto-approves against a BLOCKED gate; closeout fires falsely.
**Discipline:** Filter dispatch with `--wave <final-wave>` until upstream plans record APPROVED.

### PIT-7: Don't predict "chain terminator" [VERIFIED: STATE.md 02.2 Lessons]

**Applies to:** Wave 5/6/7 UAT plans.
**Trap:** Framing the UAT runner plan as "the final closeout."
**Discipline:** Frame UAT plans as iterations; surface "what we learned" not "what's done." Each UAT run plan should declare its predecessors but not predict its successor.

### PIT-8: Subagent prompt focus, not restatement [VERIFIED: CLAUDE.md Operating Notes]

**Applies to:** Planner agent prompts for Wave 5/6/7 plans.
**Trap:** Pasting the full audit + verification + cascade history into the plan body.
**Discipline:** Plans carry decision-only context — the specific UAT items being closed, the source files touched, the verification command. The agent's system prompt already covers anti-patterns.

## P1.2 — Planner Job contract refactor (HIGHEST research value)

**Current state evidence:**
- `internal/controller/planner_job_helpers.go:25-38` — 14-line `batchv1Template(name, image)` returns a `PodTemplateSpec` with one container, no PVC, no credproxy, no envelope writer, no signed token.
- `internal/controller/milestone_controller.go:403-426` `buildPlannerJob` calls `batchv1Template(jobName, image)` — the planner Job has NONE of the Phase 2 dispatch contract.
- `internal/controller/milestone_controller.go:212-216` builds an `EnvelopeIn` via `BuildPlannerEnvelope`, marshals it, then literally does `_ = envInJSON` (line 216) — the envelope is discarded.
- `internal/dispatch/podjob/jobspec.go:107-291` `BuildJobSpec(opts)` is the task-side full contract: PVC mount with subPath isolation, envelope-writer init container, credproxy native sidecar (RestartPolicy=Always), subagent container with SignedToken env, three SecurityContexts, ActiveDeadlineSeconds derived from caps, deterministic labels.
- `internal/dispatch/podjob/jobspec.go:55-94` `BuildOptions` struct carries every input.

**The delta between task dispatch and planner dispatch:**

| Aspect | Task dispatch | Planner dispatch |
|---|---|---|
| Identity | `Task.UID` + attempt | `Milestone/Phase/Plan.UID` + attempt |
| EnvelopeIn.Role | `"executor"` | `"planner"` |
| EnvelopeIn.Level | `"task"` | `"milestone"` / `"phase"` / `"plan"` |
| EnvelopeOut consumed | Status patch + budget rollup | `MaterializeChildCRDs` (dispatch_helpers.go:194) |
| PVC + credproxy + token | Required | Required (currently missing — the bug) |
| Caps source | `Task.Spec.Caps` | Currently hardcoded (e.g., milestone_controller.go:211 `caps := Caps{WallClockSeconds: 600, Iterations: 20}`) |
| Outputs.Validate | Required (task writes diffs) | Not required (planner outputs ChildCRDs via envelope) |

**Proposed approach: shared `BuildJobSpec` with `JobKind` discriminator.**

Refactor `internal/dispatch/podjob/jobspec.go` BuildOptions to accept a `Kind` field:

```go

// container/volume/security shape but differs in EnvelopeIn payload, deterministic
// job name format, and label set.
type JobKind string

const (
    JobKindExecutor JobKind = "executor" // Phase 2 task dispatch
    JobKindPlanner  JobKind = "planner"  // Phase 3 milestone/phase/plan dispatch
)

type BuildOptions struct {
    Kind JobKind  // NEW — required
    
    // Identity
    ParentObj metav1.Object  // REPLACES Task — accepts Milestone/Phase/Plan/Task
    ParentUID string         // derived from ParentObj.GetUID() but explicit for tests
    Level     string         // "milestone"|"phase"|"plan"|"task" — drives Job name + labels
    Attempt   int
    
    // Existing fields — all preserved
    Project        *tidev1alpha1.Project
    SignedToken    string
    EnvelopeInJSON []byte
    SubagentImage  string
    CredproxyImage string
    SecretUID      string
    PVCName        string
    ProjectUID     string
    
    // NEW — caps applied with defaultCaps() helper (P1.3 shared with token mint)
    Caps *tidev1alpha1.Caps  // nil → applies defaults via defaultCaps()
}
```

Then refactor `BuildJobSpec(opts)` to:
1. Compute `jobName` per Kind: executor uses `JobName(parentUID, attempt)` = `tide-task-<uid>-<attempt>`; planner uses `tide-<level>-<parentUID>-<attempt>` (matches the existing `milestone_controller.go:176` shape `tide-milestone-<uid>-1`).

The `_, envInJSON, err := BuildPlannerEnvelope(...)` line stays; `_ = envInJSON` becomes `opts := podjob.BuildOptions{Kind: JobKindPlanner, ParentObj: ms, Level: "milestone", Attempt: 1, Project: project, EnvelopeInJSON: envInJSON, SignedToken: token, ...}` then `job := podjob.BuildJobSpec(opts)`. ## P1.3 — Shared `defaultCaps(caps)` helper for 300s floor

**Current state evidence:**
- `internal/controller/task_controller.go:64-68` — `const defaultWallClockFloorSeconds int32 = 300`. The "Must outlive image pull + scheduler delay + init container startup" comment is correct.
- `internal/controller/task_controller.go:379-385` — when `task.Spec.Caps` is nil or `WallClockSeconds <= 0`, the token mint uses `defaultWallClockFloorSeconds + DefaultWallClockGraceSeconds` = 360s.
- `internal/dispatch/podjob/jobspec.go:108-113` — `BuildJobSpec` reads `opts.Task.Spec.Caps.WallClockSeconds` directly (no floor), so when Caps is nil it falls through to `wallClockSeconds = 0` and `activeDeadline = 0 + 60 = 60` seconds. **The bug is right there.**

**Proposed approach:**

Extract a single helper to `internal/dispatch/podjob/caps.go` (new file):

```go
package podjob

import (
    tidev1alpha1 "github.com/jsquirrelz/tide/api/v1alpha1"
)

// DefaultWallClockFloorSeconds is the minimum wall-clock budget applied when
// caps is nil or caps.WallClockSeconds <= 0. Sized to outlive image pull +
// scheduler delay + init container startup on a cold cluster (Phase 2 WR-01).
//
// MUST match the value used by both the controller's token mint and the
// Job's activeDeadlineSeconds derivation, OR the token expires before the
// Pod can use it (token validity < deadline) or the Pod is killed mid-call
// (deadline < token validity).
const DefaultWallClockFloorSeconds int32 = 300

// DefaultCaps returns a *Caps with the wall-clock floor applied. If caps is
// non-nil and WallClockSeconds > 0, returns caps unchanged. Otherwise returns
// a copy with WallClockSeconds = DefaultWallClockFloorSeconds and any other
// nil/zero fields populated with reasonable defaults.
//
// Used by:
//   - internal/controller/task_controller.go:reconcileDispatch (token mint)
//   - internal/dispatch/podjob/jobspec.go:BuildJobSpec (activeDeadlineSeconds)
//
// A nil-caps unit test asserts the two derived deadlines match within
// DefaultWallClockGraceSeconds (60s grace already applied to deadline only).
func DefaultCaps(caps *tidev1alpha1.Caps) *tidev1alpha1.Caps {
    if caps != nil && caps.WallClockSeconds > 0 {
        return caps
    }
    // Copy + floor; preserve any non-zero Iterations / TokenBudget.
    out := tidev1alpha1.Caps{
        WallClockSeconds: DefaultWallClockFloorSeconds,
    }
    if caps != nil {
        if caps.Iterations > 0 {
            out.Iterations = caps.Iterations
        }
        // ... preserve other fields
    }
    return &out
}
```

Then update both call sites:
- `task_controller.go:379-386` → `caps := podjob.DefaultCaps(task.Spec.Caps); wallClock := caps.WallClockSeconds; token, err := credproxy.Sign(r.SigningKey, ..., time.Duration(wallClock + podjob.DefaultWallClockGraceSeconds)*time.Second)`
- `jobspec.go:108-113` → `caps := DefaultCaps(opts.Task.Spec.Caps); wallClockSeconds := int64(caps.WallClockSeconds); activeDeadline := wallClockSeconds + DefaultWallClockGraceSeconds`

After P1.2 lands the planner refactor, `BuildJobSpec` takes `BuildOptions.Caps *tidev1alpha1.Caps` directly (not `Task.Spec.Caps`), so `DefaultCaps(opts.Caps)` works for both Kinds.

**Test approach:**
- Unit `internal/dispatch/podjob/caps_test.go`: table-driven test with cases `{caps=nil, expect 300}`, `{caps={WallClockSeconds:0}, expect 300}`, `{caps={WallClockSeconds:60}, expect 60}`, `{caps={WallClockSeconds:600}, expect 600}`.
- Unit `task_controller_test.go` (new test or extend existing): construct a Task with nil Caps, drive `reconcileDispatch`, assert `tokenValidity` derived from `credproxy.Sign` matches `Job.spec.activeDeadlineSeconds - DefaultWallClockGraceSeconds`. The audit's "match within a grace window" requirement is exactly the 60s grace already in `DefaultWallClockGraceSeconds`.

**Known pitfalls:**
- The constant `defaultWallClockFloorSeconds` currently lives in `internal/controller/task_controller.go:68` (lowercase, unexported). The new home is `internal/dispatch/podjob/caps.go` (exported as `DefaultWallClockFloorSeconds`). Delete the old constant + update the import.
- Don't rename `DefaultWallClockGraceSeconds` (already exported, 60s, already used by `jobspec.go:113`). The new helper composes WITH it, not replaces it.
- The test that asserts "two derived deadlines match" should be a single function (not separate tests) because the bug class is "drift between two derivations" — co-located tests make drift obvious.

**Confidence:** HIGH — 30-line change, deterministic.

**Existing condition vocabulary:**
- `api/v1alpha1/shared_types.go` (per STATE.md decision: "4 condition types (Pending/Ready/Reconciling/Failed) + 4 reasons applied uniformly across all six Kinds"). The audit's `ParentUnresolved` is NOT in this set today.
- `task_controller.go` uses condition types `ConditionRunning`, `ConditionFailed`, `ConditionPending` (from `tideprojectv1alpha1`). New condition `ConditionParentUnresolved` would need adding to `shared_types.go`.

**Recommended approach:**

1. Add to `api/v1alpha1/shared_types.go`:

   ```go
       // ConditionParentUnresolved is set on a Task or Plan when its parent
       // Project cannot be resolved by label, owner-chain walk, or explicit
       // Spec.ProjectRef. Distinct from ConditionFailed because the resource
       // is not terminally failed — it's awaiting either a label stamp from
       // PlanReconciler (Task case) or an owner-ref addition (Plan case).
       ConditionParentUnresolved = "ParentUnresolved"
       
       // ReasonNoProjectLabel — no tideproject.k8s/project label.
       ReasonNoProjectLabel = "NoProjectLabel"
       
       // ReasonNoOwnerRef — no Project owner ref in the owner-ref chain.
       ReasonNoOwnerRef = "NoOwnerRef"
   )
   ```

2. Rewrite `resolveProject` in `task_controller.go`:
   ```go
   func (r *TaskReconciler) resolveProject(ctx context.Context, task *tideprojectv1alpha1.Task) (*tideprojectv1alpha1.Project, error) {
       // Fast path: label set by PlanReconciler.
       if projectName, ok := task.Labels["tideproject.k8s/project"]; ok && projectName != "" {
           var project tideprojectv1alpha1.Project
           if err := r.Get(ctx, client.ObjectKey{Namespace: task.Namespace, Name: projectName}, &project); err == nil {
               return &project, nil

       parent, err := r.walkOwnerChainToProject(ctx, task)

       return nil, ErrParentUnresolved
   ```

4. Caller change in `task_controller.go:reconcileDispatch`:

   ```go
   project, err := r.resolveProject(ctx, &task)
   ```

**Single-Project namespace fallback flag:**

The audit's suggestion to keep the fallback behind a feature flag is **NOT recommended**. The single-Project namespace case is already handled by the label fast-path (PlanReconciler stamps the label on every Task within ~1 reconcile of the Plan being applied). The fallback exists for a window of ~100ms during the first reconcile cycle, and removing it forces correct behavior. Adding a feature flag re-introduces the bug class.

**Test approach:**
- Unit: extend `task_controller_test.go` with a "no project label and no owner-ref" test asserting `ConditionParentUnresolved=True` and RequeueAfter=30s.
- Envtest: add `parent_unresolved_test.go` (Layer A) that creates a Task with no label and no owner-ref in a namespace with two Projects, asserts the Task gets `ConditionParentUnresolved` (not adopted by either Project).

**Known pitfalls:**

## P4.1 — Budget rolling-window reset

**Current state evidence:**
- `internal/budget/tally.go:42-44` — `WindowStart` set on first call to `RollUpUsage`, preserved thereafter. The comment at lines 25-30 documents the deferral: "A future ProjectReconciler change will compare (now - WindowStart) against BudgetConfig.RollingWindowCapCents' window and zero the tally on boundary crossings."
- `internal/budget/cap.go:18-23` — `IsCapExceeded` only checks `AbsoluteCapCents`, no rolling check.
- `internal/controller/project_controller.go:227` calls `r.handleBudgetGate(ctx, project, now)` — this is the natural site for the reset.
- `api/v1alpha1/project_types.go:81-99` — `BudgetConfig.RollingWindowCapCents int64` with the doc-only caveat at lines 90-94.

**Critical schema question:** `RollingWindowCapCents` is a single value but a rolling window needs TWO inputs: the cap AND the window duration. The current schema doesn't carry the window duration. **The planner must decide how the window is configured.**

Two options:

**Option A: window duration is a separate field.** Add `RollingWindowDuration metav1.Duration` to BudgetConfig. Cap + duration together fully specify the rolling cap.

**Option B: hardcode a 24-hour window.** Simpler, less expressive. Matches typical SaaS provider billing windows.

**Recommendation: Option A** for v1.0. The schema addition is backward-compatible (the field is `+optional`); operators who don't set it get a default (e.g., 24h). The audit's success criterion doesn't specify, but the audit's intent ("future production multi-day run") implies operators want configurability.

**Proposed implementation:**

1. Schema addition `api/v1alpha1/project_types.go`:
   ```go
   type BudgetConfig struct {
       AbsoluteCapCents int64 `json:"absoluteCapCents"`
       RollingWindowCapCents int64 `json:"rollingWindowCapCents,omitempty"`
       
       // RollingWindowDuration is the window length over which
       // RollingWindowCapCents applies. Default: 24h. Must be a positive
       // duration parseable by time.ParseDuration; values < 1h are rejected
       // at admission to prevent inadvertent denial-of-dispatch via tiny windows.
       // +optional
       RollingWindowDuration *metav1.Duration `json:"rollingWindowDuration,omitempty"`
   }
   ```

2. Reset logic — add to `internal/budget/tally.go`:
   ```go
   // MaybeResetWindow checks if the rolling window has elapsed since WindowStart
   // and zeros the tally + advances WindowStart if so. Idempotent: returns
   // false (no-op) when window hasn't elapsed or RollingWindow is unconfigured.
   //
   // Called by ProjectReconciler at the start of each reconcile (before
   // budget gate check) so the cap reflects current-window spend.
   func MaybeResetWindow(ctx context.Context, c client.Client, project *tidev1alpha1.Project, now time.Time) (bool, error) {
       if project.Spec.Budget.RollingWindowCapCents <= 0 { return false, nil }
       if project.Status.Budget.WindowStart == nil { return false, nil }
       
       windowDur := 24 * time.Hour
       if project.Spec.Budget.RollingWindowDuration != nil {
           windowDur = project.Spec.Budget.RollingWindowDuration.Duration
       }
       
       if now.Sub(project.Status.Budget.WindowStart.Time) < windowDur { return false, nil }
       
       patch := client.MergeFrom(project.DeepCopy())
       project.Status.Budget.CostSpentCents = 0
       project.Status.Budget.TokensSpent = 0
       project.Status.Budget.WindowStart = &metav1.Time{Time: now}
       if err := c.Status().Patch(ctx, project, patch); err != nil {
           return false, fmt.Errorf("budget: window reset: %w", err)
       }
       return true, nil
   }
   ```

3. Wire into `ProjectReconciler.handleBudgetGate` (project_controller.go:567):
   ```go
   func (r *ProjectReconciler) handleBudgetGate(ctx, project, now) (ctrl.Result, error) {
       if _, err := budget.MaybeResetWindow(ctx, r.Client, project, now); err != nil {
           // Log but don't fail — reset failures shouldn't halt dispatch.
           logger.Error(err, "budget window reset failed")
       }
       // ... existing budget cap + bypass logic
   }
   ```

4. Update `IsCapExceeded` to check rolling cap if configured:
   ```go
   func IsCapExceeded(project *tidev1alpha1.Project) bool {
       if project == nil { return false }
       if project.Spec.Budget.AbsoluteCapCents > 0 && project.Status.Budget.CostSpentCents > project.Spec.Budget.AbsoluteCapCents { return true }
       if project.Spec.Budget.RollingWindowCapCents > 0 && project.Status.Budget.CostSpentCents > project.Spec.Budget.RollingWindowCapCents { return true }
       return false
   }
   ```

5. Remove the WR-02 doc-only caveat from `project_types.go:90-94`.

**Test approach:**

- `.golangci.yml:25` — `logcheck` is enabled. `:28` settings block declares it as `type: "module"`.
- The k8s `logcheck` linter has multiple sub-checks: `check-deprecations`, `check-contextual`, `check-with-helpers`, etc. (per the k8s.io/component-base/logs/api/v1 docs). The default sub-check set is `check-contextual` only.
- Currently TIDE's logcheck likely has only the default subset enabled.

**Recommended approach:**

Step 1 — survey current violations: run `golangci-lint run --enable-only logcheck ./...` against `internal/controller/` and capture all warnings. Categorize:
- Terminal periods in log messages (`logger.Info("foo.")`)
- Unbalanced key-value pairs (`logger.Info("foo", "key1")` without value)
- Lowercase first word
- Passive voice

Step 2 — fix the violations in `internal/controller/*.go`. Mechanical: most are mechanical regex substitutions.

Step 3 — tighten the linter to enforce future compliance. Extend `.golangci.yml`:
```yaml
linters:
  settings:
    custom:
      logcheck:
        type: "module"
        description: Checks Go logging calls for Kubernetes logging conventions.
        settings:
          # Phase 04.1 P4.4: enable additional logcheck sub-checks.
          # check-contextual is default; these are opt-in.
          check-deprecations: true
          check-with-helpers: true
          check-key: true  # enforces key-value pair balance + casing
          check-format: true  # enforces no terminal periods, active voice
```

The exact knob names depend on logcheck version — verify against `go doc sigs.k8s.io/logtools/logcheck/pkg/logcheck` or by running `golangci-lint help linters | grep logcheck` after a `make tide-lint`.

**Test approach:**
- Verify zero `logcheck` warnings post-sweep: `golangci-lint run --enable-only logcheck ./internal/controller/... 2>&1 | grep -c "logcheck"` returns 0.
- CI gate: `ci.yaml` already runs `golangci-lint run` via the `lint` job — the tightened `.golangci.yml` propagates automatically.

**Known pitfalls:**
- The Kubernetes logcheck "active voice" rule is opinionated; some legitimate messages may resist conversion. Use exclusion comments (`//nolint:logcheck`) sparingly — prefer rewriting the message.
- Don't sweep test files (`_test.go`) — test log strings are debugging output, not user-facing.
- Don't sweep `cmd/tide/*.go` — CLI output strings are user-facing and follow different conventions.

**Confidence:** MEDIUM — the sweep is mechanical but the tightened linter knobs depend on logcheck version specifics that need verifying before commit.

### How does a UAT item's status flip from `human_needed` to `pass`?

Per inspection of `gsd-sdk` source (`bin/lib/audit.cjs:399-433`):

There is **no SDK verb that auto-flips status.** The audit scanner reads `status:` from VERIFICATION.md frontmatter and `result:` from each item. To flip an item from `human_needed` to `pass`:

1. Open the VERIFICATION.md file.
2. Edit the top-level frontmatter `status: human_needed` → `status: pass` (or `passed` — the scanner accepts both per the TERMINAL_UAT_STATUSES set).
3. For each `human_verification` item: change the item's documentation to reflect the verification outcome. The audit scanner reads the frontmatter status and the item's `result:` field if present.

The scanner ALSO looks at HUMAN-UAT.md (`04-HUMAN-UAT.md` example) where each test has `result: [pending]` — these need updating to `result: pass` for each verified item.

**Plan task shape:** the UAT runner plan's task per item should:
1. Run the verification command (e.g., `make test-int`).
2. Capture stdout/exit code.
3. On pass: edit VERIFICATION.md + HUMAN-UAT.md frontmatter/items to reflect `pass`.
4. On fail: keep status as `human_needed`, document the failure in the plan's SUMMARY, generate a follow-up fix plan in the same phase (Wave N+1).
