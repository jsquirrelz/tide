# Phase 04.1 Plan 04: P1.4 — Remove First-Project Fallback + ParentUnresolved Condition

JWT-style auth with refresh rotation using jose library — closed the silent multi-Project mis-routing bug class across all three production fallback sites: `TaskReconciler.resolveProject`, `PlanReconciler.resolveProjectName`, and `PodJobBackend.Run`; added `ConditionParentUnresolved` + owner-ref chain walk (depth 5) with MergeFrom+Patch race safety.

## What Was Done

### Three Sites Closed

**Site 1: `internal/controller/task_controller.go::resolveProject`** (audit-flagged)
- Removed `projectList.Items[0]` fallback at line 619 (original)
- Replaced with: label fast-path → owner-ref chain walk (`walkOwnerChainToProject`) → `ErrParentUnresolved`
- Caller in `reconcileDispatch` (Step 3) now sets `ConditionParentUnresolved=True` + requeues 30s on miss
- Caller in the Running-path job completion also handles the sentinel (no condition set there — task was already dispatched)

**Site 2: `internal/controller/plan_controller.go::resolveProjectName`** (audit-flagged)
- Removed `projectList.Items[0].Name` fallback at line 802 (original)
- Replaced with: label fast-path → `resolveProjectForPlan` (existing owner-chain walk via PhaseRef spec) → `ErrParentUnresolved`
- Return type changed from `string` to `(string, error)`
- Caller in `reconcileWaveMaterialization` handles sentinel with MergeFrom+Patch condition + 30s requeue
- Wave materialization runs BEFORE project resolution — waves are label-independent; only label-stamping is gated on project resolution

**Site 3: `internal/dispatch/podjob/backend.go::PodJobBackend.Run`** (RESEARCH.md-found; audit missed)
- Removed `projectList.Items[0]` fallback at lines 152-163 (original)
- Replaced with: label fast-path → `walkOwnerChain` (depth 5) → `ErrParentUnresolved`
- Backend propagates the sentinel error to its caller; backend cannot set conditions (not a reconciler)

### New Condition Vocabulary (`api/v1alpha1/shared_types.go`)

Added Phase 04.1 P1.4 const block following Phase 4 block-comment convention:
- `ConditionParentUnresolved = "ParentUnresolved"` — set on Task/Plan when Project not resolvable
- `ReasonNoProjectLabel = "NoProjectLabel"` — no `tideproject.k8s/project` label present
- `ReasonNoOwnerRef = "NoOwnerRef"` — no Project owner ref found in bounded chain walk

### Owner-Walk Implementation

`walkOwnerChainToProject` / `walkOwnerChain`:
- Bounded to depth 5 (Task→Plan→Phase→Milestone→Project)
- Checks each owner reference's Kind for `"Project"` with correct APIVersion
- Recursively fetches intermediate TIDE Kinds (Plan/Phase/Milestone/Wave/Task)
- Returns `nil, nil` on miss (not error) — caller decides if miss is terminal

### Race Safety

All condition mutations use `client.MergeFrom(x.DeepCopy()) + r.Status().Patch(...)` — PIT-4 pattern. Never `r.Status().Update(...)` which would lose concurrent label-stamp writes by PlanReconciler.

### New Envtest Spec

`test/integration/envtest/parent_unresolved_test.go`:

- Asserts `ConditionParentUnresolved=True` with `Reason=NoProjectLabel` within 30s
- Asserts NO Job is created for the unlabeled Task
- 131 lines, Layer A (in-process envtest, no cluster required)

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Reordered project resolution after wave materialization in PlanReconciler**

- **Found during:** Task 1 (controller test execution)
- **Issue:** After replacing `resolveProjectName` with the sentinel-error form, the `PlanReconciler` reconcile flow short-circuited BEFORE calling `materializeWaves`. The plan spec says "requeue on miss" but existing tests create Plans without Projects, and wave materialization must still proceed (it's DAG-only, independent of project label).
- **Fix:** Moved `materializeWaves` call to run BEFORE `resolveProjectName`. Waves materialize regardless; `ConditionParentUnresolved` is set only when label-stamping fails.
- **Files modified:** `internal/controller/plan_controller.go`
- **Impact:** Correct behavior — the spec's intent is to close silent mis-routing, not to block wave materialization.

**2. [Rule 1 - Bug] Updated existing tests that relied on the removed projectList.Items[0] fallback**

- **Fix:** 
  - `plan_controller_test.go`: Added `tideproject.k8s/project: "proj-labelstamp"` label to the Plan in `BeforeEach`
  - `backend_test.go`: Updated `testTask` to include `tideproject.k8s/project: "project-alpha"` label; updated 3 tests to use `project-alpha` as project name so the label fast-path works
- **Files modified:** `internal/controller/plan_controller_test.go`, `internal/dispatch/podjob/backend_test.go`
