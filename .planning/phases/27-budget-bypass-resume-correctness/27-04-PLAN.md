<objective>
Make a budget bypass acknowledge current spend as a durable baseline so the rolling-window cap does
not immediately re-halt the resume on the already-incurred amount, and improve the halt observability
to name WHICH cap fired. This is the operator-chosen root-cause behavior fix (D-04), NOT an
observability-only or TTL-bypass-docs change.

Purpose: BYPASS-04 — after a bypass clears the halt, the next reconcile re-evaluates BOTH caps via
`budget.IsCapExceeded`; if only the absolute cap was raised, the rolling-window cap still trips and
re-halts immediately. D-04 fixes this by recording spend-at-bypass as a baseline and re-halting only
on NEW post-bypass spend — so raising the absolute cap alone (or just resuming) makes the resume stick.
Output: A baseline set at bypass time, a `newSpendSinceBypass`-guarded re-halt scoped to
`handleBudgetGate`, a which-cap reason/message, an unchanged `IsCapExceeded` with a call-site audit, and
unit + envtest coverage.

NOTE: This plan OVERRIDES RESEARCH.md Pattern 4 / assumption A2 (the superseded "Fix A: TTL-bypass +
docs only" recommendation). The authoritative shape is CONTEXT.md D-04 and PATTERNS.md Change D / its
"⚠ CORRECTION" block.
</objective>
