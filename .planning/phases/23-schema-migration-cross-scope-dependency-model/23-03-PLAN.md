<objective>
Add the two controller-side validation guards that make the migration safe (D-09, D-10) and confirm the metric `wave` label carries the global wave (D-08, SCHEMA-02):

1. **Old-object fail-closed guard (SCHEMA-03):** at the head of ProjectReconciler.Reconcile, reject any Project whose v1alpha2 discriminator field is absent (an object authored under v1alpha1) with a `RequiresReinstall` status condition and `reconcile.TerminalError` so it never runs and never requeue-storms.
2. **Global cross-scope cycle gate (DEPS-03):** assemble the Project-wide task-level dependency graph and run `pkg/dag.ComputeWaves`; on `CycleError`, surface the involved nodes on a Project status condition and refuse to proceed — no run starts, no recovery. Coarse scope-level deps are conservatively skipped (RESEARCH OQ#3 — documented limitation; Phase 24 adds fan-out).
3. **Metric resemantics confirmation (SCHEMA-02):** confirm/adjust the `wave` metric label so it sources from the (now global) Wave the Task belongs to, with the locked `{project,phase,plan,wave}` label set unchanged and the `task` label still forbidden.

This plan does NOT build the global wave-derivation engine — the cycle gate validates the graph and refuses cycles; it does not derive or store a schedule (that would trip verify-no-aggregates).
</objective>
