## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Test BeforeSuite → kind cluster (in local Docker) | Test process runs helm + kubectl against the ephemeral kind cluster; no external network exposure. |
| Helm `--set` arguments → Helm template render | `--set` values are interpolated into chart templates. Values passed are static literals (image registry/tag/pullPolicy); no secret material in `--set` flags. |
| Helm-rendered Secret (tide-signing-key) → tide-system namespace | The chart's signing-secret.yaml template uses `randAlphaNum 64 | b64enc` to generate a fresh HMAC key per install. The key never leaves the test cluster. |
| ServiceAccount `tide-subagent` → per-test namespace | Helper `ensureSubagentSA(ns)` creates the SA in each test-fixture namespace with zero RBAC (D-A4). Tasks reference it via PodSpec; no token leakage outside the pod. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.1-02-01 | Information Disclosure | Helm `--set` flags echo to GinkgoWriter / CI logs | accept | All `--set` values are static literals (registry, tag, pullPolicy) — no real secrets. The chart's `tide-signing-key` is generated server-side by Helm's `randAlphaNum 64`; the value never appears in the `--set` arguments. |
| T-02.1-02-02 | Information Disclosure | `helm install` stdout (captured via `CombinedOutput`) may print rendered template values, potentially including the signing key | mitigate | Helm 3 by default does NOT print Secret resource data in install output (Secrets show as `# Source:` headers only, no `data:` values). Verify by inspecting one capture; if Secret data appears, filter via `sed -E '/^data:/,/^[^ ]/{/TIDE_SIGNING_KEY/{s/.*/  TIDE_SIGNING_KEY: <redacted>/}}'` before emitting to GinkgoWriter. The Phase 02.1 baseline (Plan 01) verifies this redaction discipline. |
| T-02.1-02-03 | Elevation of Privilege | `ensureSubagentSA` could be tricked into creating an SA with additional RBAC if the YAML template is mutated | mitigate | Helper YAML is a static `fmt.Sprintf` template with only `name` and `namespace` interpolated; the structural fields (no `Role`, no `RoleBinding`) are hardcoded. The function body MUST NOT accept any RBAC params. D-A4 enforced structurally. |
| T-02.1-02-04 | Tampering | The chart's `tide-signing-key` is created per-install (`randAlphaNum 64`); a test cluster persisting across runs (KEEP_KIND_CLUSTER=true) could leave a stale key in etcd that conflicts with new install attempts | mitigate | Chart uses `lookup` + `if not $secret` guard in signing-secret.yaml, so subsequent helm installs do NOT regenerate the key when the Secret already exists. `helm uninstall` honors the `resource-policy: keep` annotation on the Secret. Re-running the suite with KEEP_KIND_CLUSTER=true reuses the existing key — consistent with the Phase 2 design. |
| T-02.1-02-05 | Denial of Service | helm install --wait blocks indefinitely if the Deployment never becomes ready | mitigate | `--timeout 5m` flag bounds the wait; `ctx, cancel := context.WithTimeout(context.Background(), kindTestTimeout)` at the suite level (kindTestTimeout = 4*time.Minute) is the outer fence. If helm exceeds 5m it returns an error which the WR-09 gate handles. |
