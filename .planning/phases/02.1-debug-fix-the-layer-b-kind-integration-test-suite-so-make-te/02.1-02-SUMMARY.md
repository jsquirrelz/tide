## Threat-Model Mitigations Applied

Per the plan's `<threat_model>` register:

- **T-02.1-02-01 (Information Disclosure: --set flags echo to CI logs)** — **ACCEPTED**: all `--set` values are static literals (registry, tag, pullPolicy). No secret material in --set.
- **T-02.1-02-02 (Information Disclosure: helm install stdout may print rendered template values)** — **MITIGATED**: helm 3 does not print Secret resource data in install output (Secrets show as `# Source:` headers only, no `data:` values). The GinkgoWriter.Printf emits raw helm output, which contains only resource summary names — not data fields. Plan 04 Task 2 can grep the captured CI log to confirm `TIDE_SIGNING_KEY` does not leak.
- **T-02.1-02-03 (Elevation of Privilege: ensureSubagentSA tricked into adding RBAC)** — **MITIGATED**: the helper YAML is a static `fmt.Sprintf` template with only `name` and `namespace` interpolated; the structural fields (no Role, no RoleBinding) are hardcoded. The function signature accepts only `ns string`. D-A4 enforced structurally — adding RBAC would require a separate code change with its own review.
- **T-02.1-02-04 (Tampering: stale signing-key on KEEP_KIND_CLUSTER reuse)** — **MITIGATED by chart, not by this plan**: `charts/tide/templates/signing-secret.yaml` uses `lookup` + `if not $secret` guard, so the key is regenerated only on first install; subsequent reinstalls reuse the existing Secret. `resource-policy: keep` annotation preserves the Secret across helm uninstall.
- **T-02.1-02-05 (Denial of Service: helm install --wait blocks indefinitely)** — **MITIGATED**: `--timeout 5m` flag bounds the helm wait; the outer `ctx, cancel := context.WithTimeout(context.Background(), kindTestTimeout)` at suite_test.go:88 (kindTestTimeout = 4 minutes) is the outer fence. Helm exceeding 5m returns an error which the WR-09 gate handles.
