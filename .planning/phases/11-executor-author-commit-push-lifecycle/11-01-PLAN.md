<objective>
Implement the executor commit step (B3). After the claude CLI authors into the per-task linked worktree, the harness checks for changes, stages them, and commits under a TIDE Bot identity. Empty diff surfaces as an explicit task failure (not a silent success). HeadSHA is returned in the out envelope.

Purpose: This is the load-bearing seam between "claude wrote files" and "those files exist on a branch that integration can merge into the run branch." Without it the per-task worktrees are abandoned at the end of the task pod.

Output: internal/harness/commit.go (CommitWorktree), internal/harness/commit_test.go (SC-2 tests), cmd/claude-subagent/main.go updated.
</objective>

    Step 2 — stage all: run `git -C worktreeDir add -A`. On exec error, return (plumbing.ZeroHash, false, fmt.Errorf("CommitWorktree: git add -A: %w: %s", err, output)).

    Step 3 — commit with identity: read botName from os.Getenv("TIDE_BOT_NAME") with fallback "TIDE Bot"; read botEmail from os.Getenv("TIDE_BOT_EMAIL") with fallback "tide-bot@tideproject.k8s". <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| executor pod → worktree | claude CLI writes arbitrary file content into the linked worktree; CommitWorktree commits it as-is |
| CommitWorktree identity | git -c user.name/email flags set identity per-call; env vars TIDE_BOT_NAME/TIDE_BOT_EMAIL are read from the pod spec (set by the controller), not from the subagent's output |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01-01 | Tampering | CommitWorktree: task-authored content committed without content scanning | accept | Content scanning (gitleaks) runs at push time in tide-push, not at commit time in the executor pod (per existing D-B3 architecture). The executor pod has no remote credentials and cannot push, so commit-time scanning is not a required boundary. |
| T-11-01-02 | Injection | CommitWorktree: taskUID passed to fmt.Sprintf for commit message | mitigate | taskUID is a Kubernetes UID (UUID format, validated at CRD admission). Commit message is passed as a single `-m` arg to exec.Command, not via shell interpolation — exec.Command does not invoke a shell, so no shell injection path exists. |
| T-11-01-03 | Elevation of Privilege | executor pod attempting remote push via CommitWorktree | mitigate | CommitWorktree only calls `git add -A` and `git commit` — both are local operations. No `git push` or remote URL is involved. The executor pod spec has no GIT_PAT Secret bound (ART-04 enforcement; push Job is the only pod with creds). |
| T-11-01-04 | Information Disclosure | TIDE_BOT_EMAIL leaking committer info into logs | accept | Bot identity (tide-bot@tideproject.k8s) is not sensitive. Commit metadata is public in the pushed branch. |
</threat_model>
