<objective>
Implement IntegrateTaskBranches (B4) in pkg/git/integrate.go. This function merges per-task worktree branches into the run branch in provided order, using `git merge --no-ff` via exec.Command inside a provisioned run worktree. This satisfies SC-3: dependents see their dependencies' commits because wave-k task branches are on the run branch before wave-k+1 worktrees are created.

Purpose: go-git v5.19.0 Repository.Merge() only supports FastForwardMerge. Per-task branches authored by two tasks in the same wave need a three-way merge when they're both ahead of the run branch. Git CLI handles this correctly.

Output: pkg/git/integrate.go + pkg/git/integrate_test.go.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| task branch names → git merge CLI | taskBranch strings from TaskBranchName(taskUID) are passed as args to exec.Command; taskUIDs are K8s UUIDs |
| integration worktree provisioning | the run branch name is orchestrator-controlled (tide/run-<project>-<unix>); passed as arg to git worktree add |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-02-01 | Injection | IntegrateTaskBranches: taskBranch passed to exec.Command | mitigate | taskBranch comes from TaskBranchName(taskUID) where taskUID is a K8s UID validated at CRD admission (UUID format). exec.Command does not invoke a shell, so no shell injection path. Args are positional elements in a slice, not a shell string. |
| T-11-02-02 | Tampering | Merge conflict silently discarded | mitigate | `git merge --no-ff` exits non-zero on conflict; CombinedOutput captures stderr; error is returned and surfaces to the caller. The controller marks the push job failed, blocking the push. |
| T-11-02-03 | Tampering | Integration worktree reused across projects | accept | The integration worktree path uses runBranch (tide/run-<project>-<unix>) which is project-scoped and stored in Project.Status.Git.BranchName. Cross-project collision is structurally prevented by the unique per-project branch name. |
</threat_model>
