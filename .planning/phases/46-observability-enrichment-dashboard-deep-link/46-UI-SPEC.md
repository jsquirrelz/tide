# Phase 46 — UI Design Contract

> Visual and interaction contract for OBS-04's dashboard deep link — the ONLY UI surface in this phase. Generated by gsd-ui-researcher (--auto mode: all open items resolved from CONTEXT.md D-09..D-12, RESEARCH.md findings, and the existing dashboard's established conventions; every resolution logged with rationale). Verified by gsd-ui-checker.

> **This is an addition to an established design system, not a new design.** The token system (`dashboard/web/src/index.css`), typography scale, color contract, and accessibility conventions were locked by the Phase 04 UI-SPEC (`.planning/milestones/v1.0.0-phases/04-gates-observability-dashboard-cli/04-UI-SPEC.md`) and extended in Phases 37/38. This spec declares the existing contract for checker completeness and prescribes exactly one new affordance against it. **Do not restyle, re-token, or redesign anything outside the Phoenix link.**

---

## Scope

**In scope (the entire UI surface of OBS-04):**

- One external deep-link affordance ("View trace in Phoenix") rendered in TWO mount points:
  1. `NodeDetailPanel` — the shared detail panel for `project | milestone | phase | plan` nodes (content composed in `App.tsx` `nodePanelContent`, ~line 615).
  2. `TaskDetailDrawer` — the SEPARATE Task detail component (`dashboard/web/src/components/TaskDetailDrawer.tsx`). **RESEARCH.md Pitfall 1 corrects CONTEXT.md D-09: Task nodes do NOT render in NodeDetailPanel.** A plan touching only `NodeDetailPanel.tsx` has silently dropped Task-level deep links.
- Visibility gating: link renders ONLY when `phoenixBaseURL` (from `GET /api/v1/config`) is non-empty AND the node has trace identity. Absent either → **render nothing** — no dead button, no disabled state, no placeholder row (OBS-04 requirement letter; mirrors the Telemetry view's disabled-by-config handling).
- Link copy, states, accessibility, and the two-mount-point consistency contract.

**Out of scope (do not spec, do not build):**

- Link affordances on DAG node shells (deferred per CONTEXT.md — NodeDetailPanel/TaskDetailDrawer only).
- Any bespoke trace-viewer UI (REQUIREMENTS.md Out of Scope — link out, don't rebuild).
- Loading/skeleton states — config is fetched once per page load; until it resolves the link is simply absent, which is visually identical to the hidden state.
- Backend plumbing (chart value → env → ConfigHandler → API payloads) — that is D-10/D-11 planner territory; this contract covers only what the browser renders.

## Spacing Scale

Existing 8-point scale from `index.css` (multiples of 4, declared as `--spacing-*` tokens). Restated for checker completeness; **unchanged this phase**:

| Token | Value | Usage |
|-------|-------|-------|
| `--spacing-1` | 4px | Icon-to-label gap |
| `--spacing-2` | 8px | Compact element spacing |
| `--spacing-3` | 12px | Card/node internal padding |
| `--spacing-4` | 16px | Default element spacing, section vertical padding |
| `--spacing-6` | 24px | Section horizontal padding (panel/drawer `px-6` convention) |
| `--spacing-8` | 32px | Layout gaps |
| `--spacing-12` | 48px | Major section breaks |
| `--spacing-16` | 64px | Page-level spacing |

**Values this phase actually uses:** `gap-1` (4px, label→icon), `px-6` (24px, row horizontal — matches every existing panel/drawer section), `py-4` (16px, row vertical — matches the drawer's status/actions rows).

Exceptions: none new. The 14px icon size is the pre-existing declared exception (Phase 04 §Spacing "Status badge icon: 14px") — the `ExternalLink` icon inherits it, matching `Hourglass size={14}` in the drawer's status row.

## Typography

Existing scale (3 sizes + 1 mono variant, 2 weights — `index.css` `--text-*` tokens). Restated; **unchanged this phase**:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 12px | 600 | 1.4 |
| Heading | 18px | 600 | 1.3 |
| Mono | 13px | 400 | 1.4 |

**The Phoenix link uses exactly one role: Mono (13px / 400 / `--font-mono`)** — matching the drawer's metadata values, elapsed-time text, and command text. Rationale: the link's destination is a trace identified by hex IDs; it sits among mono metadata; sans would be the odd one out.

No new sizes, weights, or roles. Underline is permitted on this element only — the Phase 04 contract reserves underlines for "the single anchor element pattern," and this link IS that pattern's first shipped instance.

## Color

Existing 60/30/10 dark-default contract (`index.css`, light theme via `.light-theme` deltas). Restated; **no new colors, no token changes**:

| Role | Value (dark) | Usage |
|------|--------------|-------|
| Dominant (60%) | `#0b0f14` `--color-surface-base` | Page background |
| Secondary (30%) | `#161b22` `--color-surface-raised` | Panel/drawer surfaces |
| Secondary sub-tier | `#1f2937` `--color-surface-overlay` | Hover states |
| Accent (10%) | `#06b6d4` `--color-accent` (cyan — "tide" cue) | Reserved list below — UNCHANGED |
| Destructive | `#f85149` `--color-destructive` | Destructive actions/failed states only — not used this phase |

**Accent reserved-for list (existing 6 items, NOT extended):** primary CTA background · selected-node ring · active wave-band edge · SSE-connected dot · **focus ring outline on keyboard focus** · connected-pill background.

The Phoenix link does **not** use accent for its resting, hover, or icon color. It receives accent only via the existing focus-visible ring (reserved-list item 5, which applies to all focusable elements). Link colors:

| State | Foreground | Background |
|-------|-----------|------------|
| Resting | `--color-text-primary`, underlined | transparent |
| Hover | `--color-text-primary`, underlined | `--color-surface-overlay` (the established "text-primary on surface-overlay" hover contract, rounded) |
| Focus-visible | unchanged | accent focus ring (existing global convention) |

Both themes are covered automatically by the token indirection — no light-theme work.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Link label (the phase's only CTA) | `View trace in Phoenix` |
| Accessible name | `View trace in Phoenix — opens in new tab` (via `aria-label`; begins with the visible text per accessible-name-matching rules) |
| Empty state | **Nothing.** When `phoenixBaseURL` is empty/absent OR the node has no trace identity, no element renders — no heading, no explainer, no disabled affordance. The absence IS the contract (OBS-04: "absent config renders no link — no dead buttons"). |
| Error state | **None in the SPA.** The link cannot fail client-side. A Phoenix-side 404 (e.g. pre-14.2.0 Phoenix without the redirect routes) is Phoenix's surface; the operator-facing note about the ≥ 14.2.0 route requirement belongs in `docs/observability.md`, not in dashboard chrome. |
| Destructive confirmation | none — this phase has no destructive actions |

Copy is IDENTICAL at both mount points. Do not shorten to "View in Phoenix" in the drawer or lengthen to "View this node's trace…" in the panel — one string, one source.

## Component Contract: `<PhoenixTraceLink>`

The single shared implementation both mount points consume (D-09's actual intent: one link *implementation*, two render surfaces — per RESEARCH.md Pitfall 1).

### Files

| File | Role |
|------|------|
| `dashboard/web/src/lib/phoenixLink.ts` (new) | Pure URL assembly — THE one place (D-11). `phoenixTraceURL(baseURL, traceId)` → `{base}/redirects/traces/{traceId}`; `phoenixSpanURL(baseURL, spanId)` → `{base}/redirects/spans/{spanId}` (D-12 as resolved by research — Phoenix ≥ 14.2.0 shareable-URL routes, NOT the `/projects/{project}/...` prior). Trailing-slash normalization happens HERE (strip one trailing `/` from `baseURL`) — the server sends the raw config value. `encodeURIComponent` every interpolated segment (defensive; RESEARCH §Security). |
| `dashboard/web/src/components/PhoenixTraceLink.tsx` (new) | Eligibility + anchor rendering. The ONLY consumer of `phoenixLink.ts`. |

### Props

```ts
type PhoenixTraceLinkProps = {
  baseURL: string;   // raw phoenixBaseURL from GET /api/v1/config ("" when unset)
  traceId: string;   // deterministic run TraceID hex ("" when payload omits it)
  spanId?: string;   // the node's {Level}TraceSpanID hex, when present
  edge: "top" | "bottom"; // which side gets the section border (see mount points)
};
```

### Eligibility (owned by the component — never duplicated at mount points)

Render `null` unless **both** `baseURL` and `spanId` are non-empty. Rationale: `spanId` (the persisted `{Level}TraceSpanID` status field) is the signal that this node's span was actually emitted — "the node has trace identity." A node mid-flight has no span in Phoenix yet; linking to it would be a dead-ish link. `traceId` alone does not qualify.

### Href

`phoenixSpanURL(baseURL, spanId)` — the span redirect opens Phoenix anchored at this node's span (D-12's enhancement tier, resolved in-scope by research). `phoenixTraceURL` is exported and unit-tested as the documented floor: if implementation-time verification finds the span redirect unusable on the target Phoenix, fall back to the trace-root URL — one-line change inside this component, nowhere else.

### Rendered shape (when eligible)

```tsx
<div
  data-testid="phoenix-trace-link-row"
  className={clsx(
    "px-6 py-4 border-[var(--color-border-subtle)]",
    edge === "top" ? "border-t" : "border-b",
  )}
>
  <a
    href={href}
    target="_blank"
    rel="noopener noreferrer"
    aria-label="View trace in Phoenix — opens in new tab"
    data-testid="phoenix-trace-link"
    className="inline-flex items-center gap-1 rounded underline text-[var(--color-text-primary)] hover:bg-[var(--color-surface-overlay)]"
    style={{ fontFamily: "var(--font-mono)", fontSize: "13px" }}
  >
    View trace in Phoenix
    <ExternalLink size={14} aria-hidden="true" />
  </a>
</div>
```

Notes: real `<a href>` — never a `<button>` with `window.open` (preserves middle-click, copy-link-address, SR link semantics). Trailing icon (after the label) is the outbound-navigation convention; 14px matches the drawer's existing icon sizing. `rel="noopener noreferrer"` — one notch stricter than `ArtifactViewer.tsx:108`'s existing `noreferrer`; explicit `noopener` is the contract here.

### Mount points

| Mount | Location | `edge` | Placement |
|-------|----------|--------|-----------|
| Planner nodes (project/milestone/phase/plan) | `App.tsx` `nodePanelContent` construction (~line 615) — ONE insertion covering all four kinds, rendered as the first child inside `NodeDetailPanel`, above `ProjectSettingsPanel` / `ArtifactViewer` | `bottom` | Top of panel content, directly below the panel header. Always visible regardless of artifact scroll (the ArtifactViewer below is the `flex-1` scroll region). |
| Task nodes | `TaskDetailDrawer.tsx`, immediately after the metadata grid `<dl>`, before the Actions row | `top` | Sits with the metadata it relates to; `border-t` matches the drawer's existing section-separator convention. |

When ineligible, the component returns `null` — neither mount renders an empty bordered strip.

### Config consumption

`phoenixBaseURL` arrives via the existing `GET /api/v1/config` one-shot-fetch pattern (`TelemetryView.tsx:1107-1120` precedent). Contract: **fetch config once per page load** (lift to `App.tsx` state or a module-cached hook — planner's structural discretion), pass `baseURL` down as a prop to both mounts. Do NOT fetch per component mount, and do NOT render any loading affordance — pre-resolution absence is indistinguishable from the hidden state, by design. Fetch failure degrades to `""` → hidden (observability never gates; no error surface).

### Data contract (SPA-facing, D-11)

The dashboard API payloads backing the node panel and task drawer gain optional trace-identity fields (camelCase, additive-only per the locked `/api/v1` wire-contract posture — exact field names are planner discretion, e.g. `traceId` / `traceSpanId`). Absent/empty fields → hidden link. The SPA never computes trace IDs — the Go server derives `traceId` via `otelai.TraceIDFromUID` and reads `{Level}TraceSpanID` from CRD status; the browser only assembles the URL from parts it is handed.

## States

| State | Trigger | Render |
|-------|---------|--------|
| Hidden | `baseURL === ""` OR `spanId` absent/empty | nothing (component returns `null`) |
| Visible | both present | the row + anchor above |
| Hover | pointer over anchor | `--color-surface-overlay` background, rounded; text/underline unchanged |
| Focus-visible | keyboard focus | accent focus ring (existing global convention — the only accent involvement) |
| Active/loading/disabled/error | — | **do not exist.** No intermediate states; navigation is the browser's. |

## Accessibility

- Link semantics: `<a href>` with a resolvable URL — focusable, listed in SR link rosters, supports open-in-new-tab/copy-address natively.
- `target="_blank"` + `rel="noopener noreferrer"` on every instance.
- `aria-label="View trace in Phoenix — opens in new tab"` announces the context switch; icon is `aria-hidden`.
- Both mount surfaces are `role="dialog"` focus traps whose `FOCUSABLE_SELECTOR` already includes `[href]` — the anchor participates in the existing Tab cycle with zero trap changes.
- Contrast: `--color-text-primary` on `--color-surface-raised` ≈ 13.6:1 dark / ≥ 15:1 light — passes AA/AAA with the existing tokens; nothing new to verify.

---

## Consistency Contract (the two-mount-point rule)

1. ONE URL-assembly module (`lib/phoenixLink.ts`). Zero inline `${baseURL}/redirects/...` template strings anywhere else — including tests, which import the helpers.
2. ONE component (`PhoenixTraceLink.tsx`) owning eligibility, copy, icon, typography, and states. Mount points pass data; they never re-decide visibility or re-style the anchor.
3. ONE copy string, ONE accessible-name string — identical at both mounts.
4. Only sanctioned divergence between mounts: `edge` (border side) and placement position, both fixed by the Mount points table above.
