**Classification per T-02.2-16 taxonomy: `harness-bug`**

Two distinct issues surface as cascade-7:

**Issue A — caps_test.go AC5 (hang-task Job never created):**
- Classification: **harness-bug** (missing Project hierarchy — same class as cascade-5)
- Root cause: `caps_test.go` creates `cap-plan` + `hang-task` via `applyYAML` without a Project, Milestone, Phase, or Secret in `caps-test` namespace. `resolveProject(ctx, task)` at `task_controller.go:217` returns `"no project found in namespace caps-test"`. No Job is dispatched. The 60s `Eventually` at `caps_test.go:100` times out.
- Evidence: Zero manager log entries for `hang-task` or `caps-test` or `cap-plan` (same pattern as cascade-5 for credproxy). Ginkgo failure: `"A Job should be created for the hang-task" Expected <int>: 0 to be >= <int>: 1`
- Fix shape: `caps_test.go` must call `applyHierarchy(ctx, capsNS, "cap-plan", "hang-task")` instead of `applyYAML(capPlanYAML)` to create the full Project+Milestone+Phase+Plan+Task+Secret hierarchy. The Plan can set `spec.caps.wallClockSeconds: 10` on the Task via a modified `applyHierarchy` call OR the helper needs to accept task-spec overrides.
- `output_test.go` is in the same class: `applyYAML` without `applyHierarchy`; will fail for the same reason when reached.

**Issue B — credproxy_test.go HARN-03 pod-wait still timing out at 120s:**
- Classification: **spec-flake** (timeout still too tight) OR **harness-bug** (ResourceVersion conflict causing very slow dispatch)
- The 120s budget (bumped from 60s in Task 1) was consumed entirely. `credproxy-task` dispatch end-to-end takes > 120s. The ResourceVersion conflict between `PlanReconciler.stampTaskLabels` and `TaskReconciler` first-status-update is more severe than estimated in `02.2-07-VERIFICATION.md` §Section 5 (which estimated the retry adds ~10-30s). The actual overhead is > 60s additional.
- Note: fixing Issue A (caps hierarchy) will reduce controller work-queue pressure from repeated `resolveProject` errors for `hang-task` (which runs BEFORE credproxy in spec order). With caps getting a proper Project hierarchy, the controller won't be spending reconcile cycles on repeated `resolveProject` failures — credproxy dispatch may complete faster. This is worth testing before concluding the credproxy timeout needs to be bumped further.
- T-02.2-21 flag fires: `dispatch latency marginal` — credproxy pod-wait consumed full 120s budget.

### Fix Landscape (for Plan 02.2-09)

**Root cause structural analysis:**

Three test files use `applyYAML` without `applyHierarchy`:
1. `caps_test.go` — creates Plan+Task in `caps-test` namespace
2. `output_test.go` — creates Plan+Task in `output-test` namespace
3. `failure_test.go` — creates Plan+Tasks in `failure-test` namespace (only tests in `failure_test.go` use `createNamespace` + `applyYAML`, NOT `applyHierarchy`)

`wave_test.go` already uses `testdata/three-task-wave.yaml` which HAS the full hierarchy. That's why wave specs run without `resolveProject` failures.

**Plan 02.2-09 fix shape:**

Option A — Extend `applyHierarchy` to accept task-spec overrides (caps, testMode) and update `caps_test.go`, `output_test.go`, and `failure_test.go` to call it:
- `applyHierarchy` current signature: `func applyHierarchy(ctx context.Context, ns, planName, taskName string) error`
- Extended signature: `func applyHierarchy(ctx context.Context, ns, planName, taskName string, overrides ...TaskSpecOverride) error` where `TaskSpecOverride` is a functional option
- Caps spec needs `wallClockSeconds: 10` on the Task; the helper would pass this through to the Task YAML
- Output spec needs `caps.declaredOutputPaths` or similar
- Failure spec needs `testMode: fail-exit-1` for beta-task

Option B — Create a simpler helper `createProjectHierarchy(ctx, ns)` that creates only the Project+Milestone+Phase+Secret (the namespace-level objects), then let individual tests create Plan+Task via `applyYAML`. This is smaller surface than extending `applyHierarchy`.

Option C — Add inline Project+Milestone+Phase+Secret YAML to each test's existing `applyYAML` call. Simplest, no helper changes required.

**Recommendation:** Option B (createProjectHierarchy helper) is cleanest: it addresses the root cause (`resolveProject` gate) without duplicating the hierarchy YAML inline in each test, and doesn't require modifying `applyHierarchy`'s signature for the caps/failure/output variations.

**After cascade-7 is fixed:** credproxy pod-wait (Issue B) will likely improve as controller pressure drops. If credproxy still times out at 120s after Issue A is fixed, bump to 180s or 240s. But fixing Issue A first is the correct sequencing.
