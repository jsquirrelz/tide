### Root-Cause Summary (for the next debug plan)

1. **`tide-projects` PersistentVolumeClaim requires `ReadWriteMany` access mode** (chart-side, `charts/tide/templates/projects-pvc.yaml` line `accessModes: [ReadWriteMany]`). The chart's design intent is to share workspace state across concurrent task pods (the spec's "artifact PVC during a run; orchestrator pushes to the target repo's git remote at each level boundary" pattern). RWX is correct for the production deployment target (any CSI driver that supports RWX — e.g., NFS, CephFS, EFS).

2. **kind's default `rancher.io/local-path` provisioner only supports `ReadWriteOnce` / `ReadWriteOncePod`** (per the captured Warning event: *"NodePath only supports ReadWriteOnce and ReadWriteOncePod (1.22+) access modes"*). The PVC binding fails permanently on the kind cluster used by Layer B integration tests.

3. **`tide-controller-manager` Pod stays `Pending`** because Kubernetes' default scheduler refuses to bind a Pod whose PVC volume reference cannot resolve. Liveness/readiness probes never fire. helm `--wait` blocks forever (chart `--timeout 5m` would fire, but the outer `timeout 300s go test` fires first).

4. **The Phase 02.2 source-shape fixes (kindTestTimeout = 7m, --replace, cleanupKindCluster, cert-manager v1.20.2, CI YAML DUR drop) are all working correctly** but they cannot rescue this blocker because it lives in the chart's PVC contract, not in the test infrastructure timing.

### Fix landscape (architectural — Rule 4 territory; requires planner re-evaluation)

- **Option A — Override accessModes via helm values:** Currently the chart's `accessModes` is hardcoded in `charts/tide/templates/projects-pvc.yaml` to `ReadWriteMany`. Expose accessModes as a values key (e.g., `.Values.workspaces.pvc.accessModes`) defaulting to `[ReadWriteMany]` and add `--set workspaces.pvc.accessModes={ReadWriteOnce}` to the test's helm install args. Tradeoff: production-deployment intent is RWX; the test override means "this works because the test runs a single-node kind cluster where RWO is sufficient." Surface a comment in `applyController()` explaining the override.
- **Option B — Deploy an RWX-supporting CSI driver into kind:** e.g., `kubernetes-sigs/nfs-server-provisioner` or `csi-driver-host-path` configured for RWX. Adds a BeforeSuite step. Tradeoff: more infrastructure surface in the test runtime; slower BeforeSuite; new dep to maintain.
- **Option C — Use an `emptyDir` volume instead of the PVC in the test:** Override the chart Deployment to mount an `emptyDir` on `/workspaces` for the test cluster. Tradeoff: workspace state doesn't persist across Pod restarts; the test may not exercise the RWX-shared-state code path the production chart relies on.
- **Option D — Use the chart's existing `workspaces.pvc.create=false` switch:** Skip chart-provisioned PVC creation; have the test create a PVC + manually-provisioned PV via cluster.yaml's `extraMounts` host-path. Tradeoff: spreads chart contract across test surfaces; harder to keep in sync.
- **Option E — Pin storage class to a hostpath CSI provisioner that supports RWX in single-node:** e.g., set `storageClassName: standard-rwx` after installing such a CSI driver. Tradeoff: extra setup; not all RWX-claiming drivers actually deliver RWX semantics on a single node.

These are architectural — the chart-side test override (Option A or D) is cheapest and matches the existing `workspaces.pvc.storageClassName` precedent. **Decision is for a planner, not Phase 02.2's executor.**
