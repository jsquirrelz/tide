### Cascade-10 Classification — `harness-bug` (PVC namespace-scoping gap)

**Specific failing spec(s):**
- Clean run: Layer B AC1 (`wave_test.go:86`) — Task `alpha` reached `Failed` phase
- Rerun: Layer B AC3 (`failure_test.go:140`) — Task `alpha-fail` reached `Failed` phase
- In BOTH cases, Tasks transition Pending→...→Failed without ever reaching Succeeded

**Decisive runtime evidence (Pod scheduling event):**
```
LAST SEEN   TYPE      REASON             OBJECT
102s        Warning   FailedScheduling   pod/tide-task-bf84abf4-...-njbm8
            0/1 nodes are available: persistentvolumeclaim "tide-projects" not found.
            preemption: 0/1 nodes are available: 1 Preemption is not helpful for scheduling.
```

**Root cause:** TaskReconciler.reconcileDispatch creates Jobs in the TEST NAMESPACE (e.g. `tide-int-test`, `failure-test`) via `BuildJobSpec` with `PVCName: "tide-projects"`. The `tide-projects` PVC is provisioned by the Helm chart ONLY in the `tide-system` namespace. The Pod attempts to mount `persistentvolumeclaim "tide-projects"` from its OWN namespace — and finds none → FailedScheduling → Pod never starts → Job never completes → TaskReconciler reattempts → exhausts max 3 attempts → Task Status.Phase = Failed (via `task_controller.go:318`).

**Why this surfaced now and not before Plan 02.2-10:** Before cascade-8 closure (Plan 02.2-10's Dispatcher wiring), the manager NEVER reached `r.Create(ctx, job)` because the `Dispatcher` field was nil — reconcileDispatch returned early. Plan 02.2-10 made dispatch live for the first time; the wave-test fixture Tasks got Jobs created in `tide-int-test` for the first time; the PVC namespace gap surfaced for the first time at the Pod scheduling layer.

**Why cascade-9A's fix did NOT close this:** Cascade-9A's mechanism was `activeDeadlineSeconds=60` killing Pods that DID start. Cascade-10's mechanism is `FailedScheduling` — the Pods never get scheduled at all because of a missing PVC. Cascade-9A's fix is necessary AND CLOSED, but Pods now block earlier, at Pending state with `FailedScheduling`, never reaching the activeDeadline mechanism.

**Cascade-10 fix-shape proposal for Plan 02.2-12 (per T-02.2-35 acceptance):**

**Option A — Fixture-side: provision `tide-projects` PVC in each test namespace (CHOSEN — smallest surface; Option-B test-only contract preserved):**
- Edit each test fixture YAML in `test/integration/kind/testdata/*.yaml` to declare a `PersistentVolumeClaim` named `tide-projects` in the same namespace as the test's Tasks, with `accessModes: [ReadWriteOnce]` (matches the chart's RWO contract from cascade-1)
- Files affected: `three-task-wave.yaml` (tide-int-test), and any other test-namespace fixtures (failure-test, output-test, etc.)
- Zero production-code changes; preserves Option-B test-only contract; mirrors the cascade-7 fixture-fix shape from Plan 02.2-09

**Option B — Test-suite-side: have `suite_test.go` create the PVC programmatically per namespace before each spec (REJECTED for surface):**
- Larger refactor; would touch BeforeEach blocks in multiple test files

**Option C — Production-side: switch the chart to use a cluster-scoped PV or a per-namespace PVC template (REJECTED for scope):**
- This is the "correct" long-term fix per the spec's namespace-per-project tenancy design, but it requires a chart redesign that's out of cascade-10 closure scope
- Plan 03+ should evaluate whether `tide-projects` PVC namespace-scoping needs a chart redesign (per-Project PVC vs shared); this is a Phase 03+ production-code follow-up
