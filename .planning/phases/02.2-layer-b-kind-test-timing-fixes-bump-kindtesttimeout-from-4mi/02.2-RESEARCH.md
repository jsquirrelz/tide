## Summary

Phase 02.1 Plan 04 Task 2 captured a BLOCKED gate (`02.1-04-VERIFICATION.md`) with four distinct root causes. Each has a tactical fix. Phase 02.2's job is to land all four and re-run the same reproducer to verify the same suite reaches 7/7 PASS on both clean-start and idempotent rerun.

The four fixes are **structurally independent** — they touch different files, share no compilation unit, and verify independently — but they have a hard temporal dependency in the success criteria: fix #1 (kindTestTimeout bump) is necessary to make helm finish; fix #2 (AfterSuite zombie cleanup) is only exercised on a failed BeforeSuite and is what makes idempotency rerun work; fix #3 (Makefile wall-time re-scope) is the only thing that lets the verification gate's "≤ 300s" assertion pass on a cold-cache run; fix #4 (cert-manager bump) is optional and may shave 10-20s.

**Primary recommendation:** Land all four in a single plan (one wave, parallel tasks if desired) because they share the same verification surface (`make test-int` clean + `KEEP_KIND_CLUSTER=true make test-int` rerun). Splitting them across plans multiplies the verification cost without buying isolation — the runtime gate has to fire either way, and individual fixes are too small to justify their own plan.

The single non-obvious mechanic that the planner must get right is the **AfterSuite zombie-container detection pattern**. `kind delete cluster` already shells out to `docker rm -f -v <cluster>-control-plane` internally; when Docker's `rm -f` fails with "could not kill container, but did not receive an exit event" [VERIFIED: from `02.1-04-VERIFICATION.md` §"Failure Detail"], a second `kind delete` call won't help — you need a direct `docker rm -f` (or `docker ps -aq --filter name=<cluster>-control-plane | xargs docker rm -f`) as a fallback. This is also where Phase 02.1's idempotent rerun broke.

### Pattern 1: Outer-context-must-exceed-subprocess-budget

**What:** When a Go test process shells out to a long-running subprocess (here: `helm install --wait --timeout 5m`), the outer `context.WithTimeout` MUST exceed the subprocess's own budget plus the pre-subprocess setup cost. Otherwise the parent kills the subprocess before its own timeout can fire, masking the real failure mode.

**When to use:** Always, when calling `exec.CommandContext(ctx, ...)` with a subprocess that has its own `--timeout` flag.

**Example math:**

```
BeforeSuite setup (kind connect + CRDs + cert-manager apply + cert-manager wait): ~50s
  (verified: 07:02:23 → 07:03:11 = 48s from 02.1-04-VERIFICATION.md §"BeforeSuite Detail")
+ helm install --timeout 5m (worst-case actual cost): 5m = 300s
+ waitForControllerReady fallback poll: ~5s (usually no-op when helm --wait succeeded)
+ safety margin for stragglers (Pod scheduling, image pull, readiness probe): 30-45s

Total minimum kindTestTimeout: ~50 + 300 + 5 + 30 = 385s = 6m25s
```

**Recommended:** `kindTestTimeout = 7 * time.Minute` — gives a clean 2m margin above helm's 5m budget, well above the 385s floor, and survives a slow CI runner adding 60-90s of variance. Going to 7m costs nothing (the test wall-time is bounded by helm's 5m anyway; the extra margin is only consumed on actual slow paths).

**Alternative (less recommended):** `kindTestTimeout = 6 * time.Minute` — gives only 60s margin above helm's 5m. Works on a cold-cache laptop today, but a slower CI runner or a future addition (e.g., 30s of extra setup) would re-introduce the same shadow-kill bug. The 60s of saved budget buys nothing useful.

**Decision driver:** the test process is bounded by helm's own `--timeout 5m`, not by `kindTestTimeout`. The outer context only fires if helm overshoots its own budget — in which case the test has failed regardless. **Use 7m. The extra minute is free.**

**Source:** Verified parent-context-kills-child behavior in `02.1-04-VERIFICATION.md` §"Root-cause analysis": *"helm --timeout 5m in suite_test.go:336 would have given helm 5 minutes to complete. But the test process's outer context cancels the helm subprocess at 4 minutes from BeforeSuite start."*

### Pattern 2: Robust Docker container cleanup as fallback to `kind delete`

**What:** When `kind delete cluster` fails (zombie container, Docker daemon hung kill event, partial half-installed state), use a direct `docker rm -f -v` fallback against containers labeled `io.x-k8s.kind.cluster=<cluster-name>`.

**When to use:** Always in AfterSuite of any kind-based test that calls `kind delete cluster`, because kind's own deletion is not idempotent against half-broken container states. [VERIFIED: kind issue #1116 documents this regression class — "failed to delete nodes" when a node is wedged.]

**Example:**

```go
// AfterSuite cleanup with zombie-container fallback.
// Source: 02.1-04-VERIFICATION.md §"Failure Detail" — the exact error string
// is "container ... is not running" / "cannot remove container ... could not
// kill container, but did not receive an exit event". See also kind issue
// #1116 and moby/moby#51845 (Docker 28→29 regression in kill-then-rm).
func cleanupKindCluster() {
    // Attempt 1: standard kind delete (works in the happy path).
    cmd := exec.Command("kind", "delete", "cluster", "--name", kindClusterName)
    if out, err := cmd.CombinedOutput(); err != nil {
        GinkgoWriter.Printf("kind delete cluster failed: %v\n%s\n", err, out)
    } else {
        // Verify no residual containers; kind may have exited 0 but left
        // containers in a stuck state.
        if !hasKindContainers(kindClusterName) {
            return // happy path: clean exit
        }
        GinkgoWriter.Printf("kind delete exited 0 but residual containers detected\n")
    }

    // Attempt 2: direct docker rm -f against kind's labeled containers.
    // kind labels every node container with io.x-k8s.kind.cluster=<name>.
    listCmd := exec.Command("docker", "ps", "-aq",
        "--filter", fmt.Sprintf("label=io.x-k8s.kind.cluster=%s", kindClusterName))
    ids, err := listCmd.Output()
    if err != nil {
        GinkgoWriter.Printf("docker ps fallback failed: %v\n", err)
        return // best-effort; don't fail the test
    }
    containerIDs := strings.Fields(strings.TrimSpace(string(ids)))
    if len(containerIDs) == 0 {
        return // nothing to clean
    }
    rmArgs := append([]string{"rm", "-f", "-v"}, containerIDs...)
    rmCmd := exec.Command("docker", rmArgs...)
    if out, err := rmCmd.CombinedOutput(); err != nil {
        GinkgoWriter.Printf("docker rm -f fallback failed: %v\n%s\n", err, out)
        // Last resort: don't crash the suite; idempotency of NEXT run
        // depends on test-int-kind-prep's `kind get clusters | grep`
        // also detecting the stale state and re-creating.
    }
}

func hasKindContainers(clusterName string) bool {
    cmd := exec.Command("docker", "ps", "-aq",
        "--filter", fmt.Sprintf("label=io.x-k8s.kind.cluster=%s", clusterName))
    out, err := cmd.Output()
    if err != nil {
        return false
    }
    return len(strings.TrimSpace(string(out))) > 0
}
```

[CITED: container label `io.x-k8s.kind.cluster=<name>` is kind's documented internal convention — kind source `pkg/cluster/internal/providers/docker/util.go`]

**Recommendation:** Implement this as a separate `cleanupKindCluster()` helper function (not inline in AfterSuite) so it's testable and the failure paths are explicit. Wire it as the only body of AfterSuite (after the `KEEP_KIND_CLUSTER=true` short-circuit).

### Pattern 3: Wall-time budget scoped to test invocation only

**What:** The "≤ 300s" budget from REQ-TEST-02 should bound only the `go test` invocation, not the full `make test-int` dependency chain. The current Makefile already does this correctly (`timeout 300s go test ./test/integration/... -v -timeout=5m`). The fix here is **documentation and CI assertion alignment**, not source-code change.

**The current shape:**
- `Makefile:122-124`: `test-int: manifests generate fmt vet setup-envtest test-int-kind-prep` then `timeout 300s go test ...`
- `ci.yaml:200-211`: `DUR=$((END-START))` measures the full `make test-int` (including kind-prep) and asserts `DUR ≤ 300s`

**The problem:** CI's wall-time assertion measures the wrong span. On a clean GitHub Actions runner, `test-int-kind-prep` alone (docker builds + kind create + image loads) takes ~880s on the captured laptop run. CI will always bust the 300s budget on cold cache.

**The fix:** Re-scope CI's assertion to measure ONLY `go test`, matching the Makefile's existing `timeout 300s` scope. Options:

| Option | Approach | Pros | Cons |
|--------|----------|------|------|
| (a) **Drop CI's outer DUR check entirely** | Remove the `if [ "$DUR" -gt 300 ]; then exit 1; fi` block at `ci.yaml:208-211`. The Makefile's `timeout 300s` already enforces the contract on the right span (go test). | Simplest. One place to enforce. | CI loses an explicit assertion; relies on `timeout 300s` exiting 124 to fail. |
| (b) **Move CI's DUR check inside the Makefile recipe** | Have the Makefile recipe capture `START`/`END` around just the `go test` line, error if `DUR > 300`. CI just runs `make test-int`. | One place to enforce; explicit assertion. | Adds Bash-in-Makefile complexity. |
| (c) **Split into two CI steps with separate DUR checks** | Step 1: `make test-int-kind-prep` (no budget). Step 2: `time make test-int-only-no-prep` (budget = 300s). | Cleanest split of concerns. | Requires a new Makefile target (`test-int-only-no-prep` or similar) that runs `go test ./test/integration/...` without re-running the prep deps. |
| (d) **Add a `TEST_INT_GO_TIMEOUT` env var** | Default = 300s; Makefile uses `timeout $TEST_INT_GO_TIMEOUT go test ...`; CI sets it explicitly; the assertion stays in CI. | Configurable per environment. | Adds env-var surface; the value never actually changes. |

**Recommended:** **Option (a)** — drop the CI outer wall-time check. The Makefile's `timeout 300s` already enforces the contract correctly on the right span. CI's redundant check just creates confusion when one fires and the other doesn't.

**Rationale:** The Makefile is the source of truth for what "≤ 300s" means; CI shouldn't re-implement it. If the Makefile's `timeout 300s` fires (exit 124), CI fails. If it doesn't fire, the test passed within budget. CI's separate `DUR > 300` check on the WHOLE `make test-int` is measuring the wrong thing — it's bounded by `timeout-minutes: 6` (the GitHub Actions job-level fence), which is the genuine outer ceiling on the kind-prep stage.

**Alternative recommendation if you want explicit CI assertion:** **Option (c)** — but this is more code for marginal benefit; the Makefile + GitHub Actions `timeout-minutes` together already give you the right bounds.

### Anti-Patterns to Avoid

- **Don't set `kindTestTimeout = 5 * time.Minute`** equal to helm's `--timeout 5m`. The pre-helm setup costs ~50s; the suite's outer context would still kill helm ~50s into helm's 5m budget. **You must leave ≥ 1m margin above helm's budget plus the pre-helm setup.**
- **Don't replace `kind delete cluster` with `docker rm -f` directly.** kind's own delete handles config-file cleanup, kubeconfig context removal, and other state outside the container. Use `docker rm -f` only as a fallback after `kind delete` exits.
- **Don't add `--force` flags to `kind delete cluster`** — kind doesn't have a `--force` flag; the only escalation path is direct `docker rm -f`.
- **Don't add `set -e` around the AfterSuite cleanup** — a failed cleanup must not fail the entire test suite. AfterSuite's job is best-effort; the next BeforeSuite's `test-int-kind-prep` will recreate the cluster if needed.
- **Don't put the kindTestTimeout in an env var** like `TIDE_KIND_TEST_TIMEOUT`. The value is a code-level invariant tied to helm's `--timeout 5m`; an env var would invite the same kind of shadow-kill bug at a different layer.
- **Don't widen the Makefile's `timeout 300s`.** The 300s is REQ-TEST-02's contract. If go-test itself genuinely needs more, the contract should be amended in a separate planning cycle — not silently bumped here.
- **Don't drop `helm --wait`.** Without it, `helm install` returns before resources are Ready, and the test races against pod scheduling. Phase 02.1 Plan 02's design depends on `--wait` blocking until the controller Deployment is Ready.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Detect zombie kind container after failed `kind delete` | A custom polling loop that pings the container's healthcheck endpoint | `docker ps -aq --filter label=io.x-k8s.kind.cluster=<name>` — kind's documented labeling convention | Label-based detection is robust to name changes, multi-node clusters, and Docker daemon weirdness; healthchecks may not exist on all kind node images |
| Bound subprocess wall-time | A goroutine that watches `time.After()` and calls `cmd.Process.Kill()` | `exec.CommandContext(ctx, ...)` with a `context.WithTimeout` that exceeds the subprocess's own budget | controller-runtime style; the parent context's cancellation propagates naturally; no goroutine leaks |
| Tee `go test` wall-time | Custom wrapper script that times Go test invocations | The `timeout 300s` shell builtin (BSD/GNU) wrapping the `go test` invocation | One-line shell idiom; matches existing Makefile pattern |
| Coordinate AfterSuite cleanup with `KEEP_KIND_CLUSTER=true` | A separate teardown phase or environment-aware shutdown | A simple `if os.Getenv("KEEP_KIND_CLUSTER") == "true" { return }` at the top of AfterSuite (current pattern, preserved) | The existing pattern in `suite_test.go:133-136` is the right idiom; don't replace |

**Key insight:** every fix in this phase is **tactical** — change a constant, add a fallback, adjust a measurement scope. There is no architecture to design and no library to add. The temptation to over-engineer (env-var configurability, retry policies, "robust" abstractions) should be resisted. The Phase 02.1 BLOCKED gate documented exactly what needs to change; Phase 02.2 changes exactly that.

## Common Pitfalls

### Pitfall 1: 6m kindTestTimeout is too tight for slow CI runners
**What goes wrong:** A 6-minute outer context gives only 60s of margin above helm's 5m budget. A slow GitHub Actions runner with `runs-on: ubuntu-latest` can vary 30-90s on cold-cache CRD apply + cert-manager wait, eating that margin.
**Why it happens:** GitHub-hosted runners share hardware; cold-cache cert-manager pull can stall on registry rate limits or image-layer cache misses.
**How to avoid:** Use `kindTestTimeout = 7 * time.Minute`. The extra 60s costs nothing (it's only consumed on actual slow paths) and survives a flaky runner.
**Warning signs:** Helm install reported as "killed: signal: killed" (parent-context cancellation) in CI logs while the Makefile reports the test as exit-0 (Ginkgo soft-skipped specs); look for `helm install failed after Xm: signal: killed` in `applyController()` output where X is just under 6m.

### Pitfall 2: AfterSuite cleanup fails silently when KEEP_KIND_CLUSTER=true
**What goes wrong:** If a developer runs `KEEP_KIND_CLUSTER=true make test-int` to preserve the cluster, the AfterSuite cleanup must short-circuit BEFORE the zombie detection logic — otherwise the "zombie cleanup" path could detect a perfectly-healthy preserved cluster and delete it.
**Why it happens:** Order of conditionals matters; the env-var check must come first.
**How to avoid:** Preserve the current `if os.Getenv("KEEP_KIND_CLUSTER") == "true" { return }` short-circuit at the very top of AfterSuite (after only `cancel()`). The zombie-cleanup helper runs only when the developer wants cleanup to happen.
**Warning signs:** Developer reports "my preserved cluster got deleted by the test suite."

### Pitfall 3: Re-running on a stale cluster after helm install partial success
**What goes wrong:** `helm install tide ...` partially renders resources (creates namespace, applies some templates) but then fails (out of memory, hook failure, etc.). The `tide` release is left in a `failed` state in helm's history. The next `make test-int` rerun with `KEEP_KIND_CLUSTER=true` calls `helm install tide ...` AGAIN — but the release name already exists, so helm errors with "cannot re-use a name that is still in use."
**Why it happens:** Helm 3 doesn't auto-replace failed releases; you need `--replace` or `helm uninstall` first.
**How to avoid:** Two options:
- **(a)** Add `--replace` to the helm install args in `applyController()`. Idempotent; harmless on first-install runs.
- **(b)** Detect existing `tide` release before install (`helm status tide -n tide-system`); if found, uninstall first.

**Recommendation:** **(a)** — single-line change, matches helm idiom for "ensure this release exists." Note that this is separate from fixes #1-4 listed in the phase scope but is a directly-related pitfall the planner should consider.

**Warning signs:** Rerun log shows `Error: cannot re-use a name that is still in use` immediately after `applyController()` starts.

### Pitfall 4: cert-manager v1.20 issuerRef defaults broke a chart-internal Certificate
**What goes wrong:** cert-manager v1.20 reverted the v0.19.0 default for `issuerRef.kind`/`group`. If any chart template omits these fields, the Certificate fails to reconcile.
**Why it happens:** The revert was intentional but removed implicit defaults the chart may have relied on.
**How to avoid:** Verify `grep -A3 "issuerRef" charts/tide/templates/*.yaml` shows explicit `kind:` and `group:` (or `name:` referring to a fully-qualified Issuer) on every issuerRef. The current chart's `serving-cert.yaml`, `metrics-certs.yaml`, and `selfsigned-issuer.yaml` should be audited.
**Warning signs:** Post-bump, cert-manager logs show `error decoding "issuerRef": missing required field "kind"`; chart-installed Certificates stuck Pending.
**Mitigation:** This is **low-risk** because the chart's Certificate templates were authored by helmify from kubebuilder's `config/certmanager/` overlay, which has historically been explicit. Verify, don't assume.

### Pitfall 5: Makefile recipe ordering hides the prep cost from CI's wall-time assertion
**What goes wrong:** `make test-int` runs `manifests generate fmt vet setup-envtest test-int-kind-prep` BEFORE the timed `go test`. CI's outer `DUR=$((END-START))` measures the whole chain, including prep. On a cold cache, prep alone exceeds 300s; the test wall-time gate fires on the wrong span.
**Why it happens:** The Makefile recipe is correct in its own scope (it timeouts only `go test`); CI's wrapping assertion just doesn't share that scope.
**How to avoid:** Drop CI's `DUR > 300` check (Option (a) from Pattern 3) OR split CI into prep + test steps with separate measurements (Option (c)).
**Warning signs:** CI fails with `Integration test duration: 1187s` immediately after a successful Layer A/B run that fit comfortably within the 300s `timeout` inside the Makefile.

### Pitfall 6: Bumping cert-manager const without updating env-var passthrough
**What goes wrong:** `installCertManager()` accepts `TIDE_CERT_MANAGER_VERSION` env var for local override (line 224). If a developer has this var set to v1.16.2 in their shell, the bump to v1.20.2 has no effect for them.
**Why it happens:** Env var override defeats the constant default; it's intentional for backward compat.
**How to avoid:** Document in the SUMMARY.md that local developers may have `TIDE_CERT_MANAGER_VERSION` set; the bump in the const is for the default path only.
**Warning signs:** "I pulled the latest, my tests still show v1.16.2 in the log."

### Pitfall 7: AfterSuite `cancel()` called before zombie cleanup uses ctx
**What goes wrong:** Current AfterSuite calls `cancel()` first, then `kind delete`. The current `kind delete` uses `exec.Command(...)`, not `exec.CommandContext(...)`, so the cancellation is harmless. But if you naively refactor the zombie cleanup to use `exec.CommandContext(ctx, "docker", "rm", "-f", ...)`, the cancelled context aborts your cleanup immediately.
**Why it happens:** Subtle Go ctx semantics — the outer `ctx` is the BeforeSuite/test budget context, not a cleanup context.
**How to avoid:** Use plain `exec.Command()` (no ctx) in AfterSuite cleanup helpers, OR create a fresh cleanup context with a short bounded timeout (e.g., `cleanupCtx, cancel := context.WithTimeout(context.Background(), 60*time.Second); defer cancel()`).
**Warning signs:** Cleanup commands exit immediately with `signal: killed` or `context canceled`.

### Fix #2: AfterSuite zombie cleanup

```go
// test/integration/kind/suite_test.go — replace AfterSuite body (lines 130-144)
// and add a cleanupKindCluster() helper function below.

var _ = AfterSuite(func() {
    cancel()

    if os.Getenv("KEEP_KIND_CLUSTER") == "true" {
        GinkgoWriter.Println("KEEP_KIND_CLUSTER=true; keeping kind cluster for debug inspection")
        return
    }

    By("Deleting kind cluster " + kindClusterName)
    cleanupKindCluster()
})

// cleanupKindCluster performs robust kind cluster cleanup with a docker-rm
// fallback for the zombie-container case (kind delete cluster fails when
// docker rm -f cannot kill a control-plane container that's stuck in a
// non-responsive state — see kind issue #1116 and moby/moby#51845 for the
// Docker 28→29 kill-event regression).
//
// Cleanup is best-effort; failure here does not fail the test suite — the
// next make test-int-kind-prep run will detect any residual state and
// attempt a fresh cluster create.
//
// Source of the failure mode: 02.1-04-VERIFICATION.md §"Failure Detail".
func cleanupKindCluster() {
    // Attempt 1: kind delete (happy path).
    cmd := exec.Command("kind", "delete", "cluster", "--name", kindClusterName)
    out, err := cmd.CombinedOutput()
    if err != nil {
        GinkgoWriter.Printf("Warning: kind delete cluster failed: %v\n%s\n", err, out)
    }

    // Verify no residual containers regardless of kind's exit code; kind has
    // been observed to exit 0 but leave containers in a stuck state
    // (Docker daemon kill-event race — moby/moby#51845).
    listCmd := exec.Command("docker", "ps", "-aq",
        "--filter", fmt.Sprintf("label=io.x-k8s.kind.cluster=%s", kindClusterName))
    idsOut, listErr := listCmd.Output()
    if listErr != nil {
        GinkgoWriter.Printf("Warning: docker ps fallback failed: %v\n", listErr)
        return
    }
    ids := strings.Fields(strings.TrimSpace(string(idsOut)))
    if len(ids) == 0 {
        return // success: kind delete cleaned up properly
    }

    GinkgoWriter.Printf("Residual kind containers detected after delete: %v; force-removing\n", ids)
    rmArgs := append([]string{"rm", "-f", "-v"}, ids...)
    rmCmd := exec.Command("docker", rmArgs...)
    if rmOut, rmErr := rmCmd.CombinedOutput(); rmErr != nil {
        GinkgoWriter.Printf("Warning: docker rm -f fallback failed: %v\n%s\n", rmErr, rmOut)
    }
}
```

**Wave structure:**
- **Wave 1:** T1, T2, T3, T4 (parallelizable — same file but no overlapping line ranges; could be a single atomic commit for cleanliness)
- **Wave 2:** T5 (runtime gate; depends on Wave 1 source-shape changes being merged)
- **Wave 3:** T6 (closeout; depends on T5 APPROVED gate decision)

Alternative plan shapes considered and rejected:
- **One plan per fix (4 plans):** rejected because all four fixes share the same verification surface (`make test-int` clean + rerun); splitting them quadruples the verification cost without isolating any risk.
- **Two plans (source-shape + verification):** equivalent shape to the recommendation above but with the closeout (T6) folded into the verification plan. Acceptable alternative.
- **Per-failure-mode grouping (timing-fixes + idempotency-fixes):** rejected because fix #3 (Makefile re-scope) isn't a "fix" per se — it's a measurement-scope correction — and doesn't fit either group cleanly.
