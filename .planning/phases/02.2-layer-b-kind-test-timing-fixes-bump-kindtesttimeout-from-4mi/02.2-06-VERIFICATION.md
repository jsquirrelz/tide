### Root Cause of credproxy FAIL (cascade-5 harness-bug analysis)

The credproxy test fixture creates:
1. A `Plan` CRD (`credproxy-plan` in `credproxy-test` namespace) — no `phaseRef` pointing to an actual Phase CRD
2. A `Task` CRD (`credproxy-task` in `credproxy-test` namespace) with `spec.dev.testMode: success`

The task controller reconcile path for dispatching a Job requires:
- The parent Wave to be in the dispatch-ready state (which requires Wave → Plan → Phase → Milestone hierarchy)
- OR the `testMode` dispatch shortcut path to be implemented

Because the credproxy test creates only Plan+Task without the full Milestone/Phase/Wave hierarchy, the task controller's dispatch guard blocks Job creation. The test expects a Pod to appear within 60s, but the Job is never created. The manager log confirms: `task cleanup` runs (finalizer cleanup) without any intermediate `dispatch` or `job create` log lines.

This is a **harness-bug** in the test fixture — not a controller bug. The credproxy test was authored to verify the sidecar topology, but the fixture doesn't create the parent hierarchy needed for actual dispatch. The fix is one of:
- Option A: Add the full Milestone/Phase/Wave hierarchy to the credproxy test fixture
- Option B: Implement a `testMode` dispatch shortcut in the task controller that bypasses parent-hierarchy checking for test tasks (i.e., if `spec.dev.testMode` is set, dispatch directly without checking Wave state)
- Option C: The test fixture should use the same `tide-int-test` namespace + fixture hierarchy that `wave_test.go` uses (which does dispatch correctly per the prior plan's evidence)

**Note:** This cascade-5 is NOT about the Makefile budget (600s is now sufficient). The outer timeout fix IS effective.
