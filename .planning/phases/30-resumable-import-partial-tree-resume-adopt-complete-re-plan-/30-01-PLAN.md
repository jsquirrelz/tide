<objective>
Fix the core defect: incomplete/missing-envelope seed nodes must materialize re-plannable (empty status) instead of as `Running`-with-no-envelope zombies, while complete-envelope nodes keep adopting their salvaged status.

The completeness signal currently lives only in the `tide-import` Job (`isEnvelopeComplete`), which runs AFTER `reconcileCreatingCRs` has already stamped status from the seed. The seed itself carries no completeness signal â€” `buildSeedManifest` copies `Status: ms.Status.Phase` verbatim from the live (mid-authoring `Running`) CR. This plan moves the completenessâ†’empty-status bridge to EXPORT time and promotes the completeness check to a single shared definition.

Purpose: Make partial-tree import re-plannable at the source-of-truth (the seed manifest), with zero new schema and zero change to `reconcileCreatingCRs` logic (its existing `if Status != ""` guard already does the right thing).
Output: Shared `IsEnvelopeComplete`, export-time empty-Status for incomplete/missing nodes, and an envtest pinning the per-node materialization branch.
</objective>

<chosen_design>
## Where the completenessâ†’empty-status bridge lands (OPEN DESIGN DECISION â€” RESOLVED)

**Chosen: Option (b), at export time in `buildSeedManifest`, backed by a shared completeness helper.**

Three options were evaluated against the verified control flow:

- **(a) tide-import Job records skipped nodes; ImportController consults it.** REJECTED. The import Job runs in `reconcileCopyingEnvelopes`, which is the state AFTER `reconcileCreatingCRs` has already created the CRs and patched their status from the seed (state machine: `CreatingCRs â†’ CopyingEnvelopes`, import_controller.go:226-233). Consuming the Job's skip-list would require a second pass that un-stamps already-`Running` status â€” fighting the state machine.
- **(c) ImportController checks per-node envelope completeness during materialization.** REJECTED. The ImportController only reads the seed ConfigMap; it never reads PVC envelope bytes. Making it read the PVC would duplicate the import Job's PVC access in a controller that today has the seed as its single source of truth.
- **(b) export tooling sets `Status=""` for incomplete/missing-envelope nodes.** CHOSEN. `buildSeedManifest` already holds `envelopes map[string][]byte` (uid â†’ repaired out.json bytes, export_envelopes_run.go:108-115). It can run the completeness check per node exactly where it sets `Status: ms.Status.Phase`. The seed stays the single source of truth, `reconcileCreatingCRs`'s existing `if Status != ""` guard (import_controller.go:424/471/518) already skips the status patch (and, for Plans, the `ValidationState = "Validated"` stamp at line 533) for empty status â€” so the controller needs NO logic change.

**Single source of truth (WR-02 mitigation):** `isEnvelopeComplete` is duplicated today (cmd/tide-import/main.go:323 + the implicit assumption in export). Promote it to `pkg/dispatch.IsEnvelopeComplete(EnvelopeOut) bool` and have BOTH the import Job and `buildSeedManifest` call it. This guarantees export-time "incomplete" and import-time "skip" agree by construction â€” the exact mismatch that produced run #2's zombies.

**Missing-envelope nodes:** a node whose UID has no entry in the `envelopes` map (no out.json) is treated identically to incomplete â†’ `Status=""`. This matches the import Job, which treats a missing/unreadable out.json as incomplete (cmd/tide-import/main.go:205-211).
</chosen_design>
