<chosen_design>
## Guard shape and position (Fork 3 — LOCKED to ImportComplete-based, not count-based)

The guard gates on `ImportComplete=True` AND presence of at least one owned Milestone — NOT on a milestone count. This is safe for N>1 milestones because `ImportComplete=True` fires only after `reconcileCreatingCRs` has materialized ALL seed nodes (import_controller.go transitions `CreatingCRs → CopyingEnvelopes` only after the full materialization loop, then `→ Complete`/ImportComplete=True after the import Job). So at the moment this guard can fire, the milestone list is already complete — the mid-stream abort the existing comment (1037-1043) warns about cannot occur, because that concern was specifically about a count guard firing DURING incremental materialization, which is the pre-ImportComplete window.

**Position:** place the new arm immediately after the existing import HOLD at project_controller.go:1078-1084 (which parks UNTIL ImportComplete) and BEFORE the PlannerPool acquire at 1086 (Pitfall 2: parking/returning after acquire would leak a pool slot). The hold handles "import not done yet → park"; this new arm handles "import done AND milestones materialized → permanently skip." Gate the whole arm on `project.Spec.ImportSource != nil` so non-import Projects are untouched.

**Owned-Milestone detection:** list Milestones in the Project namespace and match `Spec.ProjectRef == project.Name` (the same ownership predicate buildSeedManifest uses at export_envelopes_run.go:350). Return `ctrl.Result{}, nil` on the first match.
</chosen_design>
