<objective>
Seed item 9: normalize `ConditionParentUnresolved` polarity. Today the same condition type means opposite things per controller — Task sets Status=True when the parent is unresolved (correct: matches the name); Milestone/Phase's `surfaceParentRefUnresolved` sets Status=False when the parent is missing (wrong). Conditions are the operator API; one type gets one truth semantics. D-04 locks the direction: True == parent unresolved. Fix the two wrong sites, add the missing clear-on-resolve half, sweep every consumer and test in the same commit, and document the observable change.

Purpose: `ParentUnresolved=False` currently means "fine" on a Task but "parent missing" on a Milestone — any operator tooling reading the condition gets inverted answers per kind.
Output: two commits; Parent suites green; the change documented as observable status semantics (this is the one intentional behavior change in the phase, sanctioned by D-04).
</objective>
