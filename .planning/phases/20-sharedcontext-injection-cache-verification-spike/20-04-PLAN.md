<objective>
Build the CACHE-01 verification spike: a standalone `//go:build spike` harness
(`cmd/tide-spike/main.go`, mirroring `cmd/tide-eval`) that dispatches a throwaway
≥1,024-token identical-prefix prompt twice through the real credproxy on the durable
`kind-tide-dogfood` cluster, using two DIFFERENT `--add-dir eventsDir` paths to simulate
two pods (D-01). It reads `cache_read_input_tokens` from `events.jsonl` to render the
PASS/FAIL verdict (D-02). For the FAIL path it adds a minimal credproxy body-tee so the
exact per-pod prefix divergence (CWD? `--add-dir` path? per-pod workspace id?) can be
diffed and named — that root cause, not a bare "it didn't fire," is what Plan 05 records
in PROJECT.md.

This plan is empirically independent of the field work (Plans 01-03): the probe prompt
is synthetic, not the production templates. It can build and pass `go build -tags spike`
in Wave 1 alongside Plan 01; the live run happens in Plan 05.

Purpose: Settle the genuine unknown — does `claude -p --bare` emit byte-identical prefix
bytes across pods, or does the per-pod `--add-dir` path break prefix identity?
Output: A runnable `make spike` harness + a credproxy tee mechanism for the FAIL diff.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| spike harness → credproxy | The harness passes only the HMAC signed token (never the raw Anthropic key) as ANTHROPIC_API_KEY. |
| credproxy → upstream Anthropic | The credproxy injects the real key from ~/.tide/anthropic.key; the request body (teed on FAIL) carries the prompt but not the raw key (key is a header injected at the proxy). |
| tee files → local disk | Captured request bodies persist to a tmpdir during the FAIL diff. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-04-01 | Information Disclosure | credproxy body-tee writing the real key / signed token to disk | mitigate | Tee captures the outbound BODY only; the real ANTHROPIC_API_KEY is injected by the credproxy as a header to the upstream and is NOT echoed into the teed body. Any header capture redacts `x-api-key`/`authorization`. Asserted-absent by test/manual check (Task 1 acceptance). |
| T-20-04-02 | Tampering | predictable tee-file path exploited by a concurrent spike run | mitigate | Tee directory created by the spike with a random suffix + 0700 perms; cleaned up after the spike exits (RESEARCH Security Domain). |
| T-20-04-03 | Information Disclosure | spike harness logging the signed token | mitigate | `requireFlag` copied verbatim never logs the value; no Printf/Println includes the token var (asserted by grep in acceptance). |
| T-20-04-04 | Information Disclosure | throwaway spike signing key committed to the repo | mitigate | The signing key is the ephemeral `make eval` throwaway minted at runtime; never committed (existing recipe; .gitignore covers .secrets/). The spike consumes a pre-minted TIDE_SIGNED_TOKEN, it does not persist a key. |
| T-20-04-SC | Tampering | npm/pip/cargo installs | mitigate | Zero new dependencies (RESEARCH Package Legitimacy Audit confirms). No installs. |
