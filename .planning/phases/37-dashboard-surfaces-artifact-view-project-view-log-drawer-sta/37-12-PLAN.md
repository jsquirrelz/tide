<objective>
Gap 37-G2 closure. On a mid-stream connection drop the log drawer correctly shows "Reconnecting to log stream…" and auto-retries (never blank, never a false pod-gone) — but the reconnecting state has no explicit manual Reconnect control. The 37-10 UAT expected an operator-triggerable Reconnect per the D-14 UI-SPEC intent (a recoverable stream state must carry a reconnect affordance).

Today the manual Reconnect button lives ONLY in the `stream-error` branch (backend-emitted `error` frame). The `reconnecting` branch (transport `onerror`, auto-backoff in flight) renders spinner + copy only. Fix: add a manual Reconnect button to the reconnecting-state branch, wired to the `reconnect()` already returned by useTaskLog, so the operator can force a re-subscribe without waiting out the exponential backoff — while the auto-retry continues.

No `sse.ts` change is required: `reconnect()` → `doReconnect` (dashboard/web/src/lib/sse.ts:364-388) already clears the pending reconnect timer, resets backoff (attemptRef=0), and constructs a fresh EventSource — so force-resubscribing from the reconnecting state is already safe.

Purpose: DASH-04 — every recoverable log-drawer state offers a manual Reconnect; the operator is never stuck waiting on backoff.
Output: reconnecting-state Reconnect button + test, embedded dist rebuilt.
</objective>
