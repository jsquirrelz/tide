<objective>
Wire the D4 guard into both exposed succession sites: add `patchPhaseFailed`/`patchMilestoneFailed` helpers mirroring `patchPlanFailed`, insert the `isPlannerFailure` guard inside `if envReadOK {` ORDERED BEFORE the `expected == 0 → patchXSucceeded` branch at both the phase and milestone controllers, and lock the contract with envtests (PLANFAIL-01/02/03) plus a resume recovery test (PLANFAIL-04).

Purpose: phase and milestone take a direct `expected == 0 → patchXSucceeded` shortcut that ignores `out.ExitCode`, so a planner that fails with zero children is misclassified as a genuine leaf and falsely advances its parent — corrupting the planning DAG. Plan and project are NOT exposed (they succeed only via `gates.BoundaryDetected`, which returns `matched > 0`, false on zero children) and are deliberately excluded (D-01/D-02). The guard patches a permanent Failed condition (status patch, NOT a Go error) so there is no controller retry storm; recovery is the existing `tide resume --retry-failed` walker, which already resets Failed phases and milestones — PLANFAIL-04 needs no new resume code, only the guard to set `Status.Phase=Failed` and a test asserting the reset.

Output: the two guard insertions + two new helpers + four envtests/unit tests covering PLANFAIL-01..04.
</objective>
