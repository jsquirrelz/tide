> Visual and interaction contract for the GlobalExecutionDAGView component and
> its supporting API endpoint. This is a CONSTRAINED extension of the existing
> TIDE dashboard — all palette, typography, spacing, and design-system choices

## Scope Statement

Phase 26 adds exactly ONE new visual surface to the dashboard:

**`GlobalExecutionDAGView`** — a project-scoped execution DAG rendering the
whole-Project α…θ wave schedule derived by the global Kahn engine (Phases 24–25).
It is driven by a new API endpoint `GET /api/v1/projects/{name}/execution-dag`.

The existing `ExecutionDAGView` (per-plan) is NOT replaced or modified.
The existing `PlanningDAGView` (planning containment graph) is NOT modified.
The existing `RunningWavesView` is NOT modified.

The new component's screenshots replace BOTH README mermaid diagrams (D-07).

## Node Visual Contract — GlobalExecutionDAGView

The global view renders the SAME `TaskNode` component as `ExecutionDAGView`.
No new node types. No milestone provenance shown on individual nodes.

**Node dimensions:** 260px wide, 64px min-height (TASK_WIDTH=260, TASK_HEIGHT=64 — reuse the constants from ExecutionDAGView).

**Node anatomy** (locked by TideNodeShell):
```
┌────────────────────────────────────────────┐
│ <SquareTerminal icon> <task-name>          │
│ ──────────────────────────────────────────  │
│ <StatusBadge>  wave <N> · attempt <K>      │
└────────────────────────────────────────────┘
```

Node borders:
- Base: `1px solid var(--color-border-subtle)`, background `var(--color-surface-raised)`
- Hover: background `var(--color-surface-overlay)`
- Selected: `2px ring var(--color-accent)`
- Failed (status in {Failed, PushLeaseFailed, PushLeakBlocked, Rejected}): `4px left border var(--color-destructive)`

**Milestone provenance:** NOT shown on individual nodes. The global wave schedule treats tasks as first-class; milestone origin is implicit in task naming via the test fixture (sc-alpha, sc-beta, etc.) but is not rendered as a badge, label, or color distinction on the node. Milestone membership is available in the API response but must not be displayed in the initial implementation — it would require a new visual contract and is not required by SPEC-01.

## Edge Contract

All edges in GlobalExecutionDAGView use the same rules as ExecutionDAGView:

| Property | Value | Source |
|----------|-------|--------|
| Stroke color | `var(--color-border-strong)` (#484f58) | EDGE_STROKE constant |
| Stroke width | 1.5px | EDGE_STYLE constant |
| Arrow marker | ArrowClosed, 16×16, color matches stroke | EDGE_MARKER constant |
| Intra-wave edges | type "default" (straight) | annotateEdges() logic |
| Cross-wave edges | type "smoothstep" | annotateEdges() logic |

Cross-milestone edges (e.g. γ→η in the SPEC-01 fixture) are cross-wave edges
(γ in wave 0, η in wave 1) and therefore render as smoothstep, indistinguishable
from any other cross-wave edge. No special styling for cross-milestone edges.

have no incoming edges from higher-index waves; this matches the README execution
diagram's left-to-right reading.

## New API Endpoint: GET /api/v1/projects/{name}/execution-dag

**File:** `cmd/dashboard/api/execution_dag.go` (new handler)

**Response shape:**
```json
{
  "projectName": "spec-conformance-project",
  "tasks": [
    {
      "name": "sc-alpha",
      "phase": "Pending",
      "waveIndex": 0,
      "attempt": 0,
      "dependsOn": []
    },
    {
      "name": "sc-eta",
      "phase": "Pending",
      "waveIndex": 1,
      "attempt": 0,
      "dependsOn": ["sc-gamma", "sc-zeta"]
    }
  ]
}
```

`planTaskCard` is reused from `cmd/dashboard/api/plans.go` — the JSON field
names `name`, `phase`, `waveIndex`, `attempt`, `dependsOn` match the existing
per-plan shape. The frontend maps `phase` to `status` (as `ExecutionTaskData.status`).

**Implementation:** `List(Tasks, MatchingLabels{owner.LabelProject: projectName})`
then build `waveByTask` map from Wave CRs (`Wave.Status.TaskRefs`) following the
existing `plans.go:121–127` pattern. No new list strategy needed.

## App.tsx Wiring

The global execution-DAG view needs an entry point in App.tsx. For the Phase 26
screenshot deliverable, add a third right-pane state: when no plan is selected
AND the user navigates to the GLOBAL EXECUTION DAG tab, render
`GlobalExecutionDAGView` for the selected project.

**Minimal wiring approach (preferred for D-07 screenshot capture):**

Add a "Global DAG" button to the EXECUTION pane header (right-pane PaneHeader
action slot, matching the existing "All waves" button pattern). Clicking it
clears `selectedPlan` and sets a new `showGlobalDAG: boolean` state flag. The
right pane renders:

```
selectedPlan !== null      → ExecutionDAGView (unchanged)
showGlobalDAG === true     → GlobalExecutionDAGView
otherwise                  → RunningWavesView (unchanged)
```

The "Global DAG" button label is `Global DAG`. The existing "All waves" button
returns from ExecutionDAGView to RunningWavesView; a second click or selecting
a plan clears `showGlobalDAG`. Pane header label when global DAG is active:
`GLOBAL EXECUTION DAG` (mono 12px semibold muted, matching other pane headers).

**URL hash:** No hash route needed for the global DAG view. The screenshot is
taken by manual navigation; deep-linking is out of scope.

## Screenshot Acceptance Criteria (D-07)

`docs/screenshots/execution-dag.png` must satisfy:

**execution-dag.png (GlobalExecutionDAGView):**

2. Wave 0 band contains exactly 4 nodes: α (sc-alpha), β (sc-beta), γ (sc-gamma), ζ (sc-zeta) — names legible.
3. Wave 1 band contains exactly 2 nodes: δ (sc-delta), η (sc-eta) — names legible.
4. Wave 2 band contains exactly 2 nodes: ε (sc-epsilon), θ (sc-theta) — names legible.
5. Cross-wave edges visible as smoothstep curves connecting the correct source/target nodes (α→δ, β→δ, γ→η, ζ→η, δ→ε, η→θ).
6. The cross-milestone edge γ→η is present and visually connects Wave 0 to Wave 1 — no special cross-milestone styling required (same smoothstep as other cross-wave edges).
7. Wave band labels "WAVE 0", "WAVE 1", "WAVE 2" are visible in the band corners.
8. All 8 task nodes are fully visible (fitView maxZoom 1, padding 0.2 applied).
9. The screenshot is 1280×720px or larger (browser capture at 1x DPR).

**planning-dag.png (PlanningDAGView of same fixture):**
1. Project root node visible with correct name.
2. Two Milestone child nodes visible (ms-spec-a, ms-spec-b) with the planning edge MA→MB.
3. Phase and Plan nodes nested under each milestone — full containment hierarchy legible.
4. All nodes have status badges. Layout flows LR.
5. Same resolution requirement as execution-dag.png.
