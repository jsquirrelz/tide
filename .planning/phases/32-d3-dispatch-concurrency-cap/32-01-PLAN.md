<objective>
Bound in-flight planner Jobs at steady state with a live `client.List` count-check before pool acquire at all four planner dispatch sites, and lower the default `plannerConcurrency` from 16 to a single-node-safe 4 in both the binary and the chart.

Purpose: The planning cascade can OOM a single-node cluster because the `PlannerPool` semaphore releases on reconcile return (milliseconds after `r.Create`), not on Job terminal state — so it caps concurrent Job-creation reconciles, not in-flight running pods. A live count of non-terminal planner Jobs is the only mechanism that correctly bounds running pods (CONTEXT D-01 / Option B, locked).
Output: `plannerInFlightCount` helper, `Pool.Capacity()`, the gate wired identically at four sites, the reduced default in `config.go` + `values.yaml`, and Wave 0 cap-gate tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| in-cluster Job objects → controller List | Controller counts Jobs carrying `tideproject.k8s/role=planner`; an actor able to create Jobs in a watched namespace could influence the count |
| chart values → controller config | `plannerConcurrency` flows from chart (FIXED contract) into the in-process cap; no external/runtime input |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-01 | Spoofing | plannerInFlightCount List selector | accept | An actor that can create Jobs labelled `tideproject.k8s/role=planner` in a watched namespace already has Job-create RBAC in the cluster — a far stronger capability than gaming a count. Inflating the count only DEFERS planner dispatch (a fail-safe direction), it cannot bypass the cap. Forging a deflated count is impossible (real running Jobs are what's counted). No mitigation beyond existing namespace RBAC. |
| T-32-02 | Denial of Service | stuck non-terminal Job permanently blocking dispatch | mitigate | A planner Job stuck non-terminal (no Complete/Failed condition) would hold a cap slot indefinitely. Mitigated by the existing Job `activeDeadlineSeconds`/`backoffLimit` on planner Jobs (jobspec) which drives stuck Jobs to a Failed terminal condition, freeing the slot. The count uses `isJobTerminal` (Complete OR Failed), so a backoff-exhausted Job is correctly excluded. No new code; relies on existing Job spec. |
| T-32-03 | Denial of Service | slot leak (park after Acquire) | mitigate | D-03 ordering invariant: the gate returns RequeueAfter BEFORE PlannerPool.Acquire, so a parked reconcile never takes a semaphore slot. Enforced by acceptance-criteria line-ordering check in Task 2. |
| T-32-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this phase — all code uses existing go.mod imports (RESEARCH Package Legitimacy Audit: N/A). No supply-chain surface introduced. |
</threat_model>
