Key verified mechanics (RESEARCH Pattern 2, checked against OTel SDK source — do not re-derive):
- The SDK's `newSpan()` inherits the incoming context's TraceID whenever `psc.TraceID().IsValid()` is true, INDEPENDENT of SpanID validity. So `trace.SpanContext{TraceID: <deterministic>, SpanID: trace.SpanID{}}` yields a deterministic-TraceID ROOT span (Project's case), and the same code path with a real SpanID yields real parenting — one uniform mechanism, zero custom IDGenerator.
- D-08: `otelai.TraceIDFromUID(uid string) (trace.TraceID, error)` — string in, error out.
- Immediate-parent asymmetry: Milestone's parent (Project) is already fully resolved at its completion site; Phase and Plan each need a genuinely new `client.Get` on their immediate parent (Milestone via `ph.Spec.MilestoneRef`, Phase via `plan.Spec.PhaseRef`). Per RESEARCH A3, this plan uses dedicated single-object Gets rather than restructuring `resolveProject`'s return shape — `resolveProject` has 4 call sites in phase_controller.go and `resolveProjectForPlan` has 7 in plan_controller.go; a dedicated fetch at the 1 site needing it is lower blast radius (one extra API read per completion is negligible at completion-event frequency).
</context>

// synthesizePlannerSpan — parentSpanID is trace.SpanID{} (zero) for Project/root,
