## Design System

The dashboard's design system already exists and is **locked** by the v1.0.0 UI-SPEC (`.planning/milestones/v1.0.0-phases/04-gates-observability-dashboard-cli/04-UI-SPEC.md`), realized as Tailwind v4 `@theme` tokens in `dashboard/web/src/index.css`. This phase INHERITS it wholesale — no new tokens, no new fonts, no new colors. Do NOT introduce an editorial cream/serif/terracotta palette (project CLAUDE.md explicitly forbids it for this dev-tool dashboard).

| Property | Value |
|----------|-------|
| Tool | none (hand-rolled Tailwind v4 `@theme` tokens — `dashboard/web/src/index.css`) |
| Preset | not applicable |
| Component library | none (bespoke React components in `dashboard/web/src/components/`) |
| Icon library | none (no icon font/library in the dashboard; text + color carry state) |
| Font | System stacks only: `--font-sans` (ui-sans-serif …), `--font-mono` (ui-monospace …) — zero web-font bytes |
| Theme | Dark default, light opt-in via `<html class="light-theme">` — all new UI MUST use `var(--color-*)` tokens so both themes work for free |

## Spacing Scale

Inherited from `index.css` (`--spacing: 4px` base, 8-point named tokens). All values multiples of 4.

| Token | Value | Usage |
|-------|-------|-------|
| xs (`--spacing-1`) | 4px | Icon gaps, inline padding, stat-label gaps |
| sm (`--spacing-2`) | 8px | Compact element spacing, segmented-control padding |
| md (`--spacing-4`) | 16px | Default element spacing, card/panel padding (`p-4`) |
| lg (`--spacing-6`) | 24px | Section padding |
| xl (`--spacing-8`) | 32px | Layout gaps |
| 2xl (`--spacing-12`) | 48px | Major section breaks |
| 3xl (`--spacing-16`) | 64px | Page-level spacing |

Exceptions: `--spacing-3` (12px) exists in the token file and is legal (multiple of 4). The banner uses **16px padding (`p-4`), 8px internal gap (`gap-2`)** — matching `ChartPanel`.

## Banner Contract (TELEM-03)

### Signals and states

Two independent signals (per locked D-14 + the RESEARCH.md Q2 ruling):

| Signal | Source | Meaning |
|--------|--------|---------|
| `telemetryEnabled` | Chart `prometheus.enabled` → `PROMETHEUS_ENABLED` env on dashboard deployment → dashboard server's config surface → UI (transport mechanism is planner's choice; contract is only that the UI receives a boolean) | Authoritative disabled-by-config signal |
| Panel results | Existing `/api/v1/query_range` proxy responses | `unavailable` sentinel (defensive fallback), `success` with empty `result` arrays = no-data |

Banner state machine (exactly one of three renderings, evaluated in this precedence order):

| State | Condition | Rendering |
|-------|-----------|-----------|
| **disabled-by-config** | `telemetryEnabled === false` OR (defensively) all panels resolve `unavailable` | Warning-toned banner (below) |
| **no-data** | `telemetryEnabled === true` AND every chart panel resolved `kind:"data"` with zero points AND no panel is `loading`/`unreachable` | Neutral-toned banner (below) |
| **hidden** | Any panel has data, any panel still loading, or panels are `unreachable` (unreachable keeps the existing per-panel notice — it is a connectivity error, not a config nudge) | No banner rendered |

The banner never replaces the per-panel `TelemetryUnavailableNotice` — those remain as the panel-slot degradation (EC-6). The banner is a single view-level summary above them.

### Placement & anatomy

- Rendered **once**, as the **first child** of the `telemetry-view` container (`TelemetryView.tsx:1204`), full-width, above the toolbar. Telemetry-view only — not a global app-shell banner (the nudge is about this view's data; see Assumptions A-2).
- Anatomy (matches `ChartPanel` card idiom): `rounded border p-4`, `background: var(--color-surface-raised)`, flex column `gap-2`. Two lines:
  1. Label line — 12px/600/mono/uppercase (see §Typography)
  2. Message line — 13px mono, `--color-text-muted`
- Border color is the state discriminator: `--color-status-warning` (disabled-by-config) vs `--color-border-subtle` (no-data). Label line color follows the border token.
- **No buttons, no dismiss control, no links required** (read-only dashboard; matches `TelemetryUnavailableNotice`'s link-free style). State-driven: it disappears when config/data changes.
- `data-testid="telemetry-disabled-banner"` with a `data-state` attribute (`disabled-by-config` | `no-data`) for tests.

### Accessibility

- `role="status"` (polite live region — informational, matches `Toast.tsx` `ROLE` mapping for info/warning variants; never `role="alert"` — nothing here is urgent).
- Color is never the only discriminator: the label copy differs per state (see §Copywriting).
- No animation; nothing to add under `prefers-reduced-motion`.
- Text-on-surface contrast: `#d29922` on `#161b22` for the 12px/600 label is ≥ 4.5:1; muted `#8b949e` message text on `#161b22` matches the already-shipped notice styling.

## Copywriting Contract

Voice: tight, declarative, em-dash-heavy (project docs voice). All copy names the fix path — never a dead end.

| Element | Copy |
|---------|------|
| Primary CTA | none this phase (read-only dashboard — no CTA surfaces) |
| Banner label — disabled-by-config | `TELEMETRY DISABLED` |
| Banner message — disabled-by-config | `prometheus.enabled is false — run telemetry beyond the budget tally is dark. Enable it via the "Enable telemetry" step in docs/INSTALL.md.` |
| Banner label — no-data | `NO TELEMETRY DATA` |
| Banner message — no-data | `Prometheus is enabled but returned no samples in this range — metrics appear once the first dispatch is scraped. Check the Targets page if this persists.` |
| Empty state (existing, unchanged) | `No data in range` (per-chart, `TelemetryView.tsx:474`) / `No projects in this cluster` |
| Error state (existing, unchanged) | `Prometheus endpoint is unreachable (HTTP {status})` via `TelemetryUnavailableNotice` |
| Destructive confirmation | none — no destructive actions exist in the dashboard |

Copy consistency rules: sentence-case messages, uppercase mono labels, backtick-free (rendered as plain mono), the words "disabled" vs "no … data" must appear in their respective states so the two are distinguishable by text alone (TELEM-03's core requirement).

## Assumptions (headless run — could not ask the operator)

| # | Assumption | Basis | Risk if wrong |
|---|------------|-------|---------------|
| A-1 | The UI receives `telemetryEnabled` as a boolean via the dashboard server (transport — e.g. a small config endpoint or an existing response field — is planner's choice) | D-14 (locked): "the dashboard server exposes it to the UI via its existing config surface"; RESEARCH.md Q2 ruling / A6 | If the operator overrides the Q2 ruling to existing-keys-only, the banner keys off the `unavailable` sentinel alone and the no-data branch is unchanged — copy for disabled-by-config drops the `prometheus.enabled` key name in favor of "Prometheus is not configured" |
| A-2 | Banner lives in the Telemetry view only, not the global app shell | TELEM-03 is a telemetry nudge; a global banner would nag on every DAG view; matches EC-6's no-takeover degradation contract | Trivial relocation — the component is placement-agnostic |
| A-3 | Warning tone (not red, not neutral) for disabled-by-config | It's a nudge (actionable config gap), stronger than the neutral per-panel notice but not an error; NOTES.txt uses "WARNING" for the same condition | One-token change (`--color-status-warning` → `--color-border-subtle`) |
| A-4 | `unreachable` panels suppress the banner (existing per-panel messaging owns that case) | Unreachable = connectivity fault, not the "telemetry disabled" condition TELEM-03 names | Planner may add a third banner state later; contract leaves room |
| A-5 | COST-02's condition gets no new dashboard surface in Phase 38 | CONTEXT.md scopes D-02 to condition + metric; dashboard project view is Phase 37; orchestrator constraint says mark non-UI items N/A | If Phase 37 lands first with a conditions list, `PricingFallbackActive` renders there via `ConditionBadge` with zero Phase-38 work |
