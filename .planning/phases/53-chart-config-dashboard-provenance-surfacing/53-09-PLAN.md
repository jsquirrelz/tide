<objective>
The live half of ROADMAP SC2's "proven by an upgrade-path test": a kind-cluster proof that
D-05's marker makes fresh-install posture STICKY across upgrades while a pre-existing
install stays off — the one behavior `helm template` cannot exercise (lookup is always
empty without a live cluster, RESEARCH Finding 1). Then the D-10 phase-gate closeout: run
every ci.yaml-only gate in-phase (the v1.0.8 release-cascade lesson) so nothing waits for
release pre-flight.

Isolation design (RESEARCH Finding 7/9, Pitfall 5 — decided here, not at execution time):
the test drives its OWN release (`tide-posture`) in its OWN namespace with its own
install→upgrade→cleanup lifecycle, marked Serial. The shared suite's single
`helm upgrade --install tide` stays untouched — a second upgrade on the shared release
would roll the manager Deployment under 14+ concurrent specs AND is unreliable as a
"fresh install" fixture across KEEP_KIND_CLUSTER reruns (Finding 9). Assertions are on
RENDERED/APPLIED resources (Deployment args + marker ConfigMap presence), never on pod
readiness — the throwaway release's pods may crash-loop on missing images and that is
irrelevant to the posture contract. Cluster-scoped collisions (ClusterRole names etc.) are
the one risk: if the chart's cluster-scoped names don't template on release name, fall
back to the marker-deletion trick against the shared release ordered as the suite's LAST
Serial spec — the fallback is pre-authorized here so the executor never invents a third
approach.
</objective>
