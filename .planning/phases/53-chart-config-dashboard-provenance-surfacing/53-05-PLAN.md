<objective>
The D-01/D-03/D-05 chart surface: the `subagent.verify` values block (model scalar, posture
tri-state, per-level LoopPolicy defaults), the sticky install-time posture marker (the
signing-secret `lookup`+`resource-policy: keep` idiom applied to a ConfigMap), and the
conditional `--verify-levels-json` manager arg — plus the D-06 render-pair test proving
fresh-install-ON / in-place-upgrade-OFF with no cluster.

Posture semantics (concrete, D-05): effective posture =
- `subagent.verify.posture` == "enabled" → tier ON;
- == "disabled" → tier OFF;
- == "auto" (default): marker ConfigMap `tide-verify-posture` found by `lookup` with
  data.posture == "enabled" → ON; no marker AND .Release.IsInstall → ON (fresh install);
  no marker AND upgrade → OFF (pre-existing install upgrading in place).
The marker is created only on `(not lookup-result) and .Release.IsInstall`, records
`posture: "enabled"`, and carries `helm.sh/resource-policy: keep` so uninstall/upgrade
never deletes it (fresh-install lineage survives every later `helm upgrade` — bare
IsInstall would flip posture off on the first upgrade, the documented D-05 failure mode).

CRITICAL (RESEARCH Finding 1): under `helm template`, `lookup` ALWAYS returns empty and
IsInstall defaults true — so plain template renders ON and `--is-upgrade` renders OFF,
which is exactly what the render-pair test asserts. Sprig `default` treats boolean false
as empty (Pitfall 2) — the posture is a STRING enum, never a boolean; never use
`default true` on any boolean values key.

All authored in hack/helm/, regenerated via `make helm-controller`, committed together
(Pitfall 1). values.yaml additions are additive-only (FIXED contract).
</objective>
