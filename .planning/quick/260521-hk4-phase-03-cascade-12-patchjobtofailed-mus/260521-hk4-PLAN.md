<objective>
Close Phase 03 Cascade 12 (K8s 1.31+ Job status validation) by adding a `FailureTarget=True` condition entry BEFORE the existing `Failed=True` entry inside `patchJobToFailed`'s status patch body in `test/integration/kind/push_lease_test.go`. Surfaced by the cascade-11 runtime gate (Stage 1 isolation log at `/tmp/cascade-11-isolation.log`), which showed the kubectl-driven Job status patch failing with: `The Job "tide-push-<uid>" is invalid: status.conditions: Invalid value: cannot set Failed=True condition without the FailureTarget=true condition`.

Purpose: Kubernetes 1.31+ tightened Job status validation. The API server now rejects status patches that set the `Failed=True` condition unless the antecedent `FailureTarget=True` ("tentative failure") condition is also present in the same conditions array. TIDE's stack is pinned at Kubernetes ≥ 1.33 (per CLAUDE.md Technology Stack section), so this validation is live in every kind-based integration run. Pre-1.31 clusters were permissive — `patchJobToFailed` was authored against that looser contract and worked for previous kind node images. The fix is to author the conditions array with the semantic ordering K8s requires: FailureTarget first, Failed second. Production code is untouched: TIDE's controllers never manually patch Jobs to Failed — that path is exclusive to this test helper, which mocks push-lease rejection without actually running `tide-push` against a real remote.

Output: ~12 net lines added (one new map entry with 6 fields and a 2-line explanatory comment) and 0 lines removed inside one function in one file. One atomic commit: `fix(test): set FailureTarget=True before Failed=True in patchJobToFailed (cascade-12, K8s 1.31+)`. Code-shape correctness via `go vet` + `go build` + grep gates is the planning-time bar; runtime gating via `make test-int GINKGO_FOCUS='Push lease semantics'` (~5–7 min isolation) followed by full `make test-int` (~18 min) is the verification gate documented in the Verification section.

**Cascade-12 root cause → fix shape (one-liner)**:
K8s 1.31+ Job status validation now requires `FailureTarget=True` (an antecedent "tentative" condition) before `Failed=True` can be set in a status patch. Append a new map entry FIRST in the conditions array carrying `type: batchv1.JobFailureTarget`, then leave the existing `Failed=True` entry unchanged in second position. Same Reason, Message, and Timestamps for both — the API server validates the relationship between the two condition types, not their independent metadata.

**Observed failure (verbatim from /tmp/cascade-11-isolation.log; this is the cascade-11 Stage-1 log that surfaced the next cascade)**:
```
[FAILED] patch Job status failed: exit status 1
The Job "tide-push-<uid>" is invalid: status.conditions: Invalid value:
cannot set Failed=True condition without the FailureTarget=true condition
```

This error fires from the `Fail(...)` call at line 273 of push_lease_test.go inside `patchJobToFailed`, which is the helper invoked by Tests 3 and 4 in the push-lease Describe block.

**Why this is test-side, not production-side**:
- TIDE's controllers (`internal/controller/project_controller.go`, `internal/controller/task_controller.go`, et al.) dispatch Jobs via `client.Create` and READ their status — they never manually patch a Job to Failed. The Failed transition happens via Pod-failure-driven BackoffLimit semantics inside the K8s Job controller, which already authors both FailureTarget and Failed conditions in the correct order (K8s controller-internal code went through the same validation update in K8s 1.31).
- This test helper exists exclusively to MOCK lease-rejected push Jobs in `push_lease_test.go` without running real `tide-push` binaries against a real Git remote. It bypasses real Pod-failure plumbing by patching `status.failed=1` + a Failed condition directly. The K8s 1.31+ validator catches this shortcut and demands the antecedent condition.
- Per CLAUDE.md Working Rule #1 (Observe First) and the spec's "tests catch up to production, never reverse" principle, the test is what's out of sync. Bring it up to the K8s 1.31+ contract.
