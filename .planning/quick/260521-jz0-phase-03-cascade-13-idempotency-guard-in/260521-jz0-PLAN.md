<objective>
Close Phase 03 Cascade 13 (push-lease Project.Status.Phase reverts to Initialized after push-Job failure). The locked diagnosis in `.planning/debug/push-lease-phase-revert.md` Resolution section identifies the root cause: `handleInitJobCompletion` at `internal/controller/project_controller.go:297-330` is called on every reconcile pass (the init Job remains Succeeded permanently after first completion). The `isJobSucceeded` branch unconditionally patches `Project.Status.Phase = PhaseInitialized`, clobbering any forward Phase transitions (PhaseComplete, PhasePushLeaseFailed, etc.) that the test's `forcePushReady` patch or the push-Job-failed branch at line 480 had set. The push-Job-failed branch at line 480 is gated on `Phase == Complete` at line 440 — when `handleInitJobCompletion` reverts Phase=Complete → Phase=Initialized, the push-failed transition never fires, and push_lease Tests 3+4 time out at 90s observing `Phase=Initialized` instead of `Phase=PushLeaseFailed`.

The fix is surgical: a Phase-state idempotency guard inside the `isJobSucceeded` branch that short-circuits the re-patch when `Status.Phase` is already past `Initialized`. The four forward-progressed states are `PhaseRunning`, `PhaseComplete`, `PhasePushLeaseFailed`, and `PhasePushLeakBlocked` (per `api/v1alpha1/project_types.go:327-351`). The guard explicitly does NOT skip `PhasePending`, `PhaseInitialized`, `PhaseInitFailed`, or `PhaseBudgetExceeded` — those are either pre-initialization states (still want to advance to Initialized on first observation) or terminal-error states that aren't reachable via this path.

Purpose: unblock push_lease Tests 3+4 to reach `Phase=PushLeaseFailed` after `patchJobToFailed` mocks the push Job's status. Expected post-fix `make test-int` shape: 13/13 specs PASS (was 11/13 with Tests 3+4 timing out at 90s on the Eventually wait).

Output: a single-block guard insertion in `project_controller.go` (~10 lines including the switch + comment block) and a single envtest in `project_controller_test.go` (~50-70 lines mirroring the existing `TestProjectReconciler_OnInitJobSuccess_SetsPhaseInitialized` scaffolding at line 209-280). **Why the guard lives in `handleInitJobCompletion`, not at the call site:**

The call site at `reconcileProjectPhase2:267` (`result, hErr := r.handleInitJobCompletion(ctx, project, &existingJob)`) is unaware of the bug — it correctly delegates Job-completion handling to the helper. Moving the guard to the call site would force every future caller of `handleInitJobCompletion` to know about the Phase-state semantics, which is a leak of internal knowledge. Function-level guards are the canonical pattern when a helper has internal idempotency requirements that depend on observed external state. (See `ensureInitJob` at line 281 — it's idempotent against the `AlreadyExists` API response, not against the calling site's loop semantics.)

**Why the four forward-progressed states are enumerated explicitly, not derived:**

There are exactly 8 Phase constants and 4 are forward-progressed-past-Initialized. Enumerating them in a `switch` makes the contract grep-visible — the test's grep assertion (`grep -cE 'PhaseRunning,\s*$|PhaseComplete,\s*$|PhasePushLeaseFailed,\s*$|PhasePushLeakBlocked' ≥ 4`) directly verifies the four required cases. A helper like `Phase.IsForwardOf("Initialized")` would obscure that contract and require a separate test for the helper. The Phase state machine is small enough that explicit enumeration is more honest.

**Why the isJobFailed branch is NOT touched:**

The `isJobFailed` branch at line 316-330 patches `Phase = PhaseInitFailed`. This is a terminal-error state — once set, the reconciler shouldn't be reaching this code path again (the K8s Job won't flip from Failed→Succeeded). Even if it did, `PhaseInitFailed → PhaseInitFailed` is a no-op MergeFrom. The cascade-13 bug is exclusively about the success path stomping FORWARD-progressed states, not the failure path.

**Why the test uses a real Reconcile cycle (not a direct call to handleInitJobCompletion):**

Calling `handleInitJobCompletion` directly is in-package legal, but it requires the test to construct a `*batchv1.Job` argument and pass it in — that bypasses the `reconcileProjectPhase2` Job-listing code at line 240-265 which is part of the contract under test. A real `Reconcile(ctx, request)` call exercises the entire path including the Job List + Phase-state read, which is closer to the runtime behavior the cascade describes. The existing tests in this file all use full Reconcile cycles for the same reason. Direct calls would also force adding a test-only helper for Job construction, which doesn't exist today.

**Why one envtest is sufficient (no second test for the Initialized → Initialized identity case):**

The Initialized → Initialized identity case (Phase already Initialized + isJobSucceeded fires again) is benign: MergeFrom with identical Phase produces no observable change. Writing a test for it would be a tautology. The OBSERVABLE bug is FORWARD-progressed states being clobbered, and one test covers the canonical forward-progressed case (Complete). The other three forward-progressed states (Running, PushLeaseFailed, PushLeakBlocked) follow by symmetric construction — the switch treats them identically. The grep assertion on the production code locks the contract for all four states.
