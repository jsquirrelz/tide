# Phase 53: Chart Config + Dashboard Provenance Surfacing - Research

**Researched:** 2026-07-21
**Domain:** Helm chart configuration surface (install-vs-upgrade state) + Go controller dispatch-gating + React/TS dashboard provenance display
**Confidence:** HIGH (all findings below are code-verified via direct file reads and `grep`/`helm` invocations against this repo; no external ecosystem research was needed ‚Äî this phase extends TIDE's own established patterns, not a third-party library)

## Summary

This phase has no new external dependencies, no new library, and no new architectural pattern ‚Äî it is a **mechanical extension of five patterns TIDE already uses elsewhere in the codebase**: (1) the `subagent.levels` ‚Üí env ‚Üí `ProviderDefaults` ‚Üí resolver chain (D-01/D-02), (2) the `signing-secret.yaml` `lookup`+`resource-policy: keep` first-install idiom (D-05), (3) the `hasVerificationContract`/`GateCommand`-gated dispatch chokepoints Phase 51/52 already built (D-04), (4) the `STATUS_TABLE`/`CONDITION_TABLE` additive-row vocabulary pattern (D-09), and (5) the existing `/api/v1/nodes/{kind}/{name}/artifacts` gitfetch endpoint (D-07). The work is real but almost entirely "plug into the existing seam," not new-pattern design.

Two load-bearing mechanics were **not fully documented in 53-CONTEXT.md** and must be surfaced to the planner explicitly:

1. **`charts/tide/values.yaml` and `charts/tide/templates/deployment.yaml` are NOT hand-edited.** They are regenerated by `make helm-controller` (`kustomize | helmify` then `hack/helm/augment-tide-chart.sh` re-applies hand-authored blocks from `hack/helm/tide-values.yaml` + Python-scripted marker-anchored insertions into `deployment.yaml`). CI's `helmify-verify`/`verify-chart-reproducible` job runs a fresh `make helm` and fails the build on any diff against the committed `charts/` tree. **Every value/env addition in this phase must be authored in `hack/helm/`, not directly in `charts/tide/`,** or the phase will pass local review and fail at the exact CI gate the v1.0.8 release-cascade lesson (STATE.md Blockers, D-10) warns about.
2. **`TIDE_VERIFIER_IMAGE` has ZERO chart wiring today.** `grep -c verifier charts/tide/values.yaml charts/tide/templates/deployment.yaml` returns 0. The kind test harness (`test/integration/kind/suite_test.go:487-500`) contains a comment that literally names this phase as the owner: *"the chart does not yet wire TIDE_VERIFIER_IMAGE (that env lands in Phase 53's chart-config surface, mirroring the existing TIDE_REPORTER_IMAGE/TIDE_IMPORT_IMAGE wiring). Until then, patch it onto the manager env post-install."* This phase must add the base `images.tideLanggraphVerifier` block AND the `TIDE_VERIFIER_IMAGE` env ‚Äî this is prerequisite plumbing underneath D-01/D-02, not something D-01/D-02 can assume already exists.

A third finding changes the shape of D-07: **the "existing gitfetch/artifacts API" does not yet serve Task findings.** `collectStageEnvelopes` (`internal/controller/artifact_push.go:84-147`) only emits `milestone`/`phase`/`plan`/`project` kind entries ‚Äî it has never emitted a `task` entry (the `tide-push` CLI-side JSON-staging logic for `kind=="task"` was built in Phase 49, but the controller-side trigger that would actually call it with a task entry was explicitly deferred: *"Nothing produces a 'task' entry yet; that is Phase 51's job"* ‚Äî and Phase 51/52 did not close this). Separately, `cmd/dashboard/api/artifacts.go`'s `artifactKinds` allowlist is `{project, milestone, phase, plan}` ‚Äî it 400s on `kind=task` today. Both gaps must close for D-07's "findings browsable through the existing artifacts API" claim to be literally true; this is additive wiring to the existing endpoint (no new route), consistent with OBS-04's "no new endpoint," but it is real, non-trivial work the planner must schedule, not a given.

**Primary recommendation:** Treat this phase as three independently-testable slices in dependency order: (A) chart mechanics ‚Äî `images.tideLanggraphVerifier` + `TIDE_VERIFIER_IMAGE` env, all authored in `hack/helm/`, proven by the existing static-grep chart-contract-test idiom; (B) the `subagent.verify` config surface + posture-marker ConfigMap + the one shared enablement-gate helper wired into the three known dispatch chokepoints; (C) dashboard provenance ‚Äî StatusBadge/ConditionBadge rows, payload extensions, and the `collectStageEnvelopes`+`artifactKinds` task-findings wiring. (A) blocks (B); (C) is independent of (A)/(B) except for the task-findings-staging sub-slice.

## Research Findings (numbered ‚Äî referenced throughout)

### Finding 1 ‚Äî Helm `lookup`/`IsInstall`/`IsUpgrade` semantics [VERIFIED: Context7 /helm/helm-www + local `helm template --help`, helm v3.16.3]

- `lookup` **always returns an empty response when executed via `helm template`** (no live-cluster read), regardless of `--is-upgrade`. Only `--dry-run=server` or a real `helm install`/`upgrade` gives `lookup` a live cluster to query. [CITED: helm-www chart_template_guide/function_list.mdx]
- `.Release.IsInstall` = true when the current operation is an install; `.Release.IsUpgrade` = true when upgrade or rollback. [CITED: helm-www topics/charts.mdx]
- `helm template` defaults to install semantics (`IsInstall=true`). The `--is-upgrade` flag ("set `.Release.IsUpgrade` instead of `.Release.IsInstall`") flips this for a no-cluster render ‚Äî confirmed present in the locally-installed `helm v3.16.3` (the exact version CI pins via `azure/setup-helm@v4` `version: 'v3.16.3'`, `.github/workflows/ci.yaml:199`).
- **This confirms D-06's render-pair test is directly implementable**: plain `helm template tide charts/tide` (no `lookup` match, `IsInstall=true`) ‚Üí the posture-marker-creation guard `(not $marker) and .Release.IsInstall` is true ‚Üí template should render the verify env ON. `helm template tide charts/tide --is-upgrade` ‚Üí `IsInstall=false` ‚Üí guard is false ‚Üí no marker rendered, verify env OFF. Both invocations are real, working commands today; no helm behavior is speculative here.
- **Real-cluster semantics for D-05's sticky marker** (not exercisable via `helm template`, only via `helm install`/`upgrade` against a live cluster, i.e. the kind test tier):
  1. Fresh v1.0.9 install: marker absent, `IsInstall=true` ‚Üí guard creates the marker (recorded posture=enabled). Env resolves ON.
  2. Later `helm upgrade` on that SAME release: marker now exists in-cluster, `lookup` finds it ‚Üí env resolves from the marker's recorded value (ON ‚Äî sticky).
  3. Pre-existing (pre-v1.0.9) install upgrading in-place to v1.0.9 for the first time: marker never existed (older chart had no such template), `lookup` finds nothing, AND `IsInstall=false` (this is a real `helm upgrade`, not install) ‚Üí guard is false ‚Üí no marker created ‚Üí env resolves to the "no marker, not installing" default (OFF).
- **Gotcha found and worth flagging to the planner**: Sprig's `default` function treats boolean `false` as "empty," so `default true .Values.verify.enabled` would silently coerce an explicit `enabled: false` override back to `true`. [CITED: helm-www versioned_docs/version-3/chart_template_guide/function_list.md] This is precisely why D-05 already specifies the override vocabulary as a **string enum** (`enabled|disabled|auto`) rather than a boolean ‚Äî confirmed as the correct choice, not just a style preference. Use `hasKey`/explicit string comparison, never `default true .Values.X` on a boolean value.

### Finding 2 ‚Äî `hack/helm/` is the actual source of truth for `charts/tide/` [VERIFIED: codebase]

`Makefile:806-816` (`helm-controller` target): `kustomize build config/default | helmify charts/tide` regenerates `values.yaml`/`Chart.yaml`/`deployment.yaml` from kubebuilder's Kustomize output, **wiping any hand-edit**, then `bash hack/helm/augment-tide-chart.sh` re-applies the project-specific augmentations:

- `hack/helm/tide-values.yaml` (473 lines, byte-identical to `charts/tide/values.yaml` ‚Äî confirmed via `diff`) is `cp`'d wholesale onto `charts/tide/values.yaml`. **Any `subagent.verify` block addition (D-01) must be authored here, not in `charts/tide/values.yaml` directly.**
- `hack/helm/augment-tide-chart.sh` contains a Python heredoc (lines 178-470) that injects marker-anchored blocks into `deployment.yaml` ‚Äî one numbered section per historical phase (`# phase2-args-injected`, `# phase3-env-injected`, `# phase9-reporter-env-injected`, `# phase28-import-env-injected`, `# phase36-agent-env-injected`, `# phase4-env-injected`, `# phase47-otlp-headers-env-injected`), each idempotently guarded (`if MARKER not in content`). **A new `# phase53-verify-env-injected` block following this exact pattern is required** for `TIDE_VERIFIER_IMAGE` + the verifier-model scalar + the structured per-level JSON var, anchored after the existing `# phase47-otlp-headers-env-injected` marker (the most recent one).
- New standalone templates (the D-05 posture-marker ConfigMap) follow the `cp "${HACK_DIR}/X.yaml" "${CHART_DIR}/templates/X.yaml"` idiom already used for `signing-secret.yaml`, `push-rbac.yaml`, `reporter-rbac.yaml`, `projects-pvc.yaml` (`augment-tide-chart.sh:139-168`) ‚Äî author `hack/helm/verify-posture-configmap.yaml` (or similar name), add one `cp` line.
- **CI enforcement**: `.github/workflows/ci.yaml` `helm-lint` job (lines 178-232) runs `make helm` fresh, then `git diff --quiet charts/` ‚Äî any drift between a hand-edit and a fresh `hack/helm/`-driven regeneration **fails the build**. `Makefile:856-864` `verify-chart-reproducible` is the equivalent local target. This is the exact mechanism the v1.0.8 release-cascade lesson (STATE.md Blockers) refers to ‚Äî a chart change that "looks done" locally but drifts from its `hack/helm/` source is invisible until this specific CI job runs.

### Finding 4 ‚Äî The three exact verifier-dispatch-activation chokepoints for D-04's enablement gate [VERIFIED: codebase]

All three existing "is verification active for this level?" decision points already gate on `GateCommand != "" && Phase == "Locked"` via `ResolveVerificationSpec`. The enablement check is a pure AND onto the SAME guard ‚Äî no new call sites, no restructuring:

1. **Task**: `internal/controller/task_controller.go:1556` ‚Äî `} else if hasVerificationContract(task) {` (routes an executor-believed-complete Task to `Verifying`+`dispatchVerifier`, vs `Succeeded` in the `else` branch). `hasVerificationContract` (`task_controller.go:2138`) is a pure function taking only `*Task` ‚Äî it has no `project`/chart-defaults parameter today, so the enablement AND needs either a new parameter or a sibling check at the call site.
2. **Plan (plan-check)**: `internal/controller/plan_controller.go:995` ‚Äî `if verification.GateCommand != "" && verification.Phase == verificationPhaseLocked && !planVerifyResolvedByOperator(plan) {` (routes a planner-completed Plan to `Verifying`, vs clearing to `""` in the `else`).
3. **Phase/Milestone/Project (shared)**: `internal/controller/level_verify.go:146` ‚Äî `levelVerifyDecision(spec, loopStatus, phase)`: `if spec.GateCommand == "" || spec.Phase != verificationPhaseLocked { return levelVerifyInactive }`. This ONE function is shared by all three levels via `maybeRunLevelVerify` (`level_verify.go:168`), called from `phase_controller.go`/`milestone_controller.go`/`project_controller.go`.

**Recommended shared helper placement**: beside `ResolveVerificationSpec` in `dispatch_helpers.go` (per D-04's instruction), signature roughly `verificationEnabledForLevel(project *Project, level string, chartDefaults VerifyDefaults) bool` implementing the precedence: `project.Spec.Verification.<Level> != nil` (pointer non-nil = explicitly authored at Project scope ‚Äî `VerificationDefaults` fields are `*VerificationSpec`, confirmed in `api/v1alpha3/project_types.go:72-82`, so nil-vs-authored is already distinguishable) ‚Üí else chart per-level `enabled` ‚Üí else `false`. Each of the three call sites ANDs this onto its existing `GateCommand != "" && Phase == Locked` check.

### Finding 5 ‚Äî `VerificationDefaults` is already pointer-typed, confirming D-04's "authored" precedence is implementable without new API surface [VERIFIED: codebase]

### Finding 6 ‚Äî `VerificationSpec.Evaluator` is currently write-only; D-02's "authored Evaluator wins" tier has no existing resolution logic [VERIFIED: codebase]

`grep -rn "\.Evaluator\b|EvaluatorRef" internal/controller/*.go pkg/dispatch/*.go` (excluding tests) shows `verification.Evaluator` is **only ever assigned** into `EvaluatorRef` on the outbound envelope/`LoopPolicy` (`task_controller.go:2340`, `plan_controller.go:1274`, `level_verify.go:412`, `dispatch_helpers.go:482`) ‚Äî it is never read back to select an actual container image or model anywhere in the Go codebase. There is no `Evaluator` CRD, no ConfigMap-of-named-evaluator-profiles, no lookup table. The verifier's dispatched image is unconditionally `r.Deps.VerifierImage` / `deps.VerifierImage` (one manager-wide value) at every one of the three dispatch sites, regardless of `verification.Evaluator`'s value.

**Implication for D-02**: "authored `Evaluator` > chart default > compiled default" precedence as literally stated requires NEW resolution logic (a string ‚Üí {image, model} mapping) that does not exist anywhere in the codebase today and has no obvious home (no registry pattern to extend). This is flagged as `[ASSUMED]` in CONTEXT.md's framing, not `[VERIFIED]` ‚Äî recommend the planner either (a) scope D-02 to just the chart-default ‚Üí compiled-default tiers this phase (leaving `VerificationSpec.Evaluator` reserved/unconsumed exactly as it is today, consistent with its current behavior), or (b) explicitly scope a minimal resolution (e.g., treat a non-empty `Evaluator` string as literally an image ref override) as new, separately-planned work. Given `Evaluator` is described everywhere else as a "same-namespace name ref" (not an image ref), option (a) is the lower-risk, spec-consistent choice for this phase.

### Finding 8 ‚Äî `TIDE_PRICING_OVERRIDES_JSON` precedent is a validated CLI flag on the MANAGER's own args, not a manager-container env var [VERIFIED: codebase]

CONTEXT.md D-01 cites `TIDE_PRICING_OVERRIDES_JSON` as "the precedent for structured config" riding "one structured JSON env var." The actual transport is more specific and worth getting right:
- Chart: `deployment.yaml:35-37` ‚Äî `{{- if .Values.pricing.overrides }} - --pricing-overrides-json={{ .Values.pricing.overrides | toJson }} {{- end }}` ‚Äî a manager **args entry** (conditionally emitted), not an env var on the manager's own container.
- Manager: `cmd/manager/main.go:177-178,245-249` reads `--pricing-overrides-json` via `flag.StringVar`, validates it at startup with `pkgdispatch.ParsePricingOverrides` (fail-fast: prints to stderr + `os.Exit(1)` on invalid JSON or out-of-range values), then carries the raw string opaquely on `Deps.PricingOverridesJSON`.
- The env var `TIDE_PRICING_OVERRIDES_JSON` only appears **downstream**, stamped onto the *dispatched subagent Job's* container env (`internal/dispatch/podjob/jobspec.go:516-521`) ‚Äî i.e. it's forwarded, not manager-consumed-as-env.

### Finding 9 ‚Äî `test/integration/kind/suite_test.go` uses `helm upgrade --install`, exercising real `IsInstall`/`IsUpgrade` semantics on reruns [VERIFIED: codebase]

`suite_test.go:511-513`: `helmCmd := exec.CommandContext(ctx, "helm", helmControllerArgs(chartDir, rolloutNonce)...)` where `helmControllerArgs` returns `["upgrade", "--install", "tide", chartDir, ...]`. ## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Install-vs-upgrade sticky state | A custom annotation/label scheme, a new controller-side "first reconcile" detector, or a bare `.Release.IsInstall` check | The `signing-secret.yaml` `lookup` + `if not` + `resource-policy: keep` idiom (D-05), applied to a small ConfigMap | This idiom is already proven correct in this exact chart for exactly this problem class (T-02-12-02/03); a bare `IsInstall` flips posture OFF on the very next `helm upgrade` (documented failure mode in CONTEXT.md D-05) |
| Structured per-level config transport | 15 discrete `TIDE_VERIFY_TASK_MAXITER`/`TIDE_VERIFY_PLAN_ONEXHAUSTION`/... env vars | One JSON blob + a `Parse*` validator function, mirroring `pricing.go`'s `ParsePricingOverrides` exactly (fail-closed at startup, ASVS V5) | Already the established pattern for "a map of per-key config too large for discrete env vars" in this codebase; a 15-var sprawl bloats the Deployment template for no benefit (D-01's own reasoning, now cross-checked as the codebase's real convention) |
| Verifier dispatch-enablement check | A per-call-site inline `if` at each of the 3 dispatch chokepoints | One shared helper beside `ResolveVerificationSpec` in `dispatch_helpers.go` | D-04's explicit instruction; also the ONLY way to keep the precedence rule (authored > chart > off) from drifting across 3 independently-maintained files |
| Status/condition vocabulary rendering | A bespoke component per new state | Additive rows on the EXISTING `STATUS_TABLE`/`CONDITION_TABLE` (`StatusBadge.tsx`/`ConditionBadge.tsx`) | These are explicitly designed as open, additive, defensively-coerced tables (`coerce()`/unknown‚Üínull philosophy already in place) ‚Äî the whole point of the pattern is that new states are table rows, not new components |

**Key insight:** Every piece of this phase has a same-shape precedent already merged and tested elsewhere in this codebase. The risk is not "design the right pattern" ‚Äî it's "find the SAME pattern's exact anchor points" (which is what Findings 1‚Äì15 above pin down) and "don't hand-edit the generated chart files" (Finding 2, the single highest-risk mistake this research uncovered).

## Common Pitfalls

### Pitfall 1: Hand-editing `charts/tide/values.yaml` or `charts/tide/templates/deployment.yaml` directly
**What goes wrong:** The change works locally, passes `git diff` review, and even passes a naive `helm template` smoke test ‚Äî then CI's `helmify-verify`/`helm-lint` job runs `make helm` fresh and the diff check fails, because the hand-edit isn't reflected in `hack/helm/tide-values.yaml` or `hack/helm/augment-tide-chart.sh`.
**Why it happens:** `charts/tide/` looks like a normal, directly-editable directory; nothing in the file itself signals "generated." Only the `augment-tide-chart.sh` header comment and the `verify-chart-reproducible` Makefile target name hint at this.
**How to avoid:** Every values.yaml/deployment.yaml change in this phase goes into `hack/helm/tide-values.yaml` (values) or a new marker-anchored Python block in `hack/helm/augment-tide-chart.sh` (deployment.yaml env/args), then run `make helm` locally and commit the regenerated `charts/tide/` output alongside the `hack/helm/` source change in the SAME commit.
**Warning signs:** `git diff charts/tide/` shows a change that has no corresponding `git diff hack/helm/` ‚Äî this is exactly what `verify-chart-reproducible` catches, but catching it locally before pushing is much cheaper.

### Pitfall 2: Using `default true`/`default false` with a boolean Values override
**What goes wrong:** An operator sets `--set subagent.verify.levels.task.enabled=false` intending to force-disable; the template's `{{ default true .Values.subagent.verify.levels.task.enabled }}` silently ignores the explicit `false` and renders `true` anyway (Sprig treats boolean `false` as "empty," so `default` overrides it).
**Why it happens:** This is a well-known but non-obvious Sprig/Helm gotcha ‚Äî `default`'s emptiness check is type-generic and boolean `false` qualifies as empty exactly like `""` or `0`.
**How to avoid:** D-05 already specifies the override vocabulary as a string enum (`enabled|disabled|auto`), not a boolean ‚Äî implement it that way. If any OTHER boolean chart value needs a real tri-state (unset/true/false) in this phase, use `hasKey .Values.subagent.verify "someBool"` to test presence explicitly, never `default`.
**Warning signs:** A `--set X=false` in a `helm template` render doesn't change the rendered output at all.

### Pitfall 3: Assuming `VerificationSpec.Evaluator` already resolves to an image/model
**What goes wrong:** Implementation time is spent building D-02's "authored Evaluator wins" precedence tier assuming `verification.Evaluator` is a live selector, only to discover (as this research found, Finding 6) that the field is currently write-only and there's no existing image/model registry to resolve it against.
**Why it happens:** CONTEXT.md's D-02 phrasing reads as if this tier already partially exists ("already resolved through the Task > Plan > Project walk") ‚Äî that phrase is true for the STRING VALUE's precedence walk (which field wins: Task's own, or Plan's, or Project's), but not for turning that string into an actual image/model.
**How to avoid:** Scope D-02 to the two REAL tiers (chart default ‚Üí compiled default) this phase; leave `VerificationSpec.Evaluator`/`EvaluatorRef` exactly as unconsumed as it is today, unless the planner explicitly decides to build the missing resolution logic as additional, separately-justified scope.
**Warning signs:** Trying to find "where does Evaluator become an image ref" and finding nothing ‚Äî that's the correct, verified answer (Finding 6), not a search failure.

### Pitfall 4: Assuming Task findings.json is already reachable via the artifacts endpoint
**What goes wrong:** D-07's dashboard "findings link" is wired to call `fetchNodeArtifacts("task", ...)`, but the request 400s (kind not in `artifactKinds`) or 200s with `state: "absent"` forever (nothing ever staged the findings.json onto the run branch) ‚Äî either way, the link never shows real data, and the bug is silent (no test failure) unless someone drives a real verifier-completed Task through a live/kind run.
**Why it happens:** CONTEXT.md's canonical refs describe this as "the 37-07 gitfetch endpoint findings browse rides ‚Äî unchanged," implying it's already wired end-to-end for tasks; Finding 10 shows it isn't (both the controller-side staging trigger and the dashboard-side allowlist are missing).
**How to avoid:** Treat "wire `collectStageEnvelopes` to include verifier-completed Task entries" and "add `task` to `artifactKinds`" as explicit, separately-verified tasks in the plan, not an assumed side-effect of the findings-link UI work.
**Warning signs:** A findings link that always renders the `absent` empty state even after a Task has genuinely gone through `VerifyHalted`/verified-`Succeeded`.

### Pitfall 5: Breaking the shared kind-suite `BeforeSuite` install with a second `helm upgrade`
**What goes wrong:** A D-06 kind sticky-posture test inserted into the existing suite (which shares ONE `helm upgrade --install` across 14+ specs, `Label("kind")`) runs a second `helm upgrade` mid-suite, which rolls the shared manager Deployment (new Pod, brief unavailability window) and can flake unrelated concurrently-running specs (`verifier_concurrency_test.go`, `level_verify_worktree_test.go`) that assume a stable Deployment.
**Why it happens:** The natural place to "add a kind test" is inside the existing `test/integration/kind` package, which strongly implies reusing the existing harness ‚Äî but that harness was built for ONE shared install, not a install/upgrade-transition proof.
**How to avoid:** Isolate the D-06 kind proof ‚Äî either its own release name/namespace with its own install/upgrade lifecycle (cleaned up in its own `AfterEach`), or the marker-ConfigMap-deletion trick (Finding 7) against the shared install performed LAST in a way that doesn't roll the Deployment for unrelated specs, or as a genuinely separate `go test` target outside the shared suite.
**Warning signs:** Previously-stable kind specs (concurrency/worktree tests) start flaking only after the D-06 test is added, with timing that correlates to a Deployment rollout.
