# Phase 04 — UI Design Contract

> Visual and interaction contract for the **read-only TIDE Dashboard** (DASH-01..05). Gates, observability, and CLI work in this phase have no UI surface — the dashboard is the only frontend. Generated by gsd-ui-researcher; will be verified by gsd-ui-checker.

> **Audience:** Kubernetes operators who already use `kubectl`, `k9s`, `helm`, `argocd`. Dense, status-first, monospace-friendly — *not* a SaaS marketing surface. Aesthetic peers: Argo CD UI, k9s, Headlamp, Lens.

> **Locked upstream:** All `D-D1..D-D6` decisions in `04-CONTEXT.md` (side-by-side split, read-only end-to-end, SSE-only, clipboard-copy actions, <500KB bundle, React 18 + @xyflow/react v12 + dagre + Tailwind v4 + Vite). Do NOT re-derive these. This spec extends them into pixel-level prescription.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Tailwind v4 raw, no shadcn) |
| Preset | not applicable — Vite + Tailwind v4 single-app, no component library registry |
| Component library | none — bespoke 15-component inventory below (DOM/SVG primitives via Tailwind utility classes; @xyflow/react v12 supplies graph primitives only) |
| Icon library | `lucide-react` (latest stable; tree-shakeable, ~1KB per icon, well-covered ops vocabulary — used by Headlamp, shadcn, Coder, OpenStatus) |
| Font | System font stack only (NO bundled web fonts — bundle weight gate per D-D5) |

**Rationale for "no shadcn":**

- This is a single-purpose ops dashboard, not a multi-surface product. The bundle-size gate (<500KB gzipped) makes shadcn's CSS-in-JS variance and registry sprawl a liability.
- The component count is ~15 components, mostly bespoke graph affordances (custom @xyflow nodes, wave bands, log streamer) that shadcn does not cover.
- Tailwind v4's `@theme` directive + a small `src/ui/` directory of typed React primitives (Button, Badge, Toast, Drawer, Dropdown) is the smaller, faster path. Re-evaluate in v1.x if component count crosses ~30.

**Font stack (verbatim CSS, copy into `src/index.css` under `@theme`):**

```css
--font-sans: ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, "Helvetica Neue", Arial, sans-serif;
--font-mono: ui-monospace, SFMono-Regular, "SF Mono", Menlo, Consolas, "Liberation Mono", monospace;
```

`--font-mono` is used wherever the operator's eye expects fixed-width: resource names (CRD names, namespaces, UIDs), status badges, log streams, task IDs, wave numbers, the connection-status pill. Everything else is `--font-sans`.

## Spacing Scale

8-point base scale (Tailwind v4 defaults are honored; we declare named tokens to forbid arbitrary `mt-[13px]` shapes). Multiples of 4 only.

| Token | Value | Tailwind utility | Usage |
|-------|-------|------------------|-------|
| `space-1` | 4px | `gap-1`, `p-1` | Icon-to-label gap, inline badge padding |
| `space-2` | 8px | `gap-2`, `p-2` | Compact element spacing (badge internal, drawer-row vertical) |
| `space-3` | 12px | `gap-3`, `p-3` | Card internal padding, node internal padding |
| `space-4` | 16px | `gap-4`, `p-4` | Default element spacing, drawer section gap |
| `space-6` | 24px | `gap-6`, `p-6` | Section padding, drawer outer padding |
| `space-8` | 32px | `gap-8`, `p-8` | Layout gaps between panes (when un-resized) |
| `space-12` | 48px | `gap-12`, `p-12` | Major section breaks, empty-state hero spacing |
| `space-16` | 64px | `gap-16`, `p-16` | Page-level top padding for full-screen error states |

**Exceptions (the only allowed off-scale values):**

- **Resizable divider grip:** 6px wide (between panes — finger/cursor target). Hit area expands to 12px on hover via `before:` pseudo-element.
- **Status badge icon:** 14px square (one notch under `space-4`). Pairs visually with 12px body text without overpowering.
- **Pod-log line spacing:** `line-height: 1.4` × 13px monospace = 18.2px per line. Lines are *tight* on purpose (operator scanning ergonomics).

Arbitrary spacing values like `mt-[13px]`, `gap-[7px]`, `w-[123px]` are **forbidden** and rejected by ESLint rule `tailwindcss/no-arbitrary-value`.

## Typography

Exactly **3 sizes + 1 monospace variant**, **2 weights**. The dashboard is dense; we resist the urge to add more.

| Role | Size | Weight | Line height | Family | Usage |
|------|------|--------|-------------|--------|-------|
| Body | 14px | 400 (regular) | 1.5 (21px) | sans | Default text, drawer content, empty-state body copy, toast body |
| Label | 12px | 600 (semibold) | 1.4 (16.8px) | sans | Form labels, section headers inside drawer, table column headers, connection-status pill |
| Heading | 18px | 600 (semibold) | 1.3 (23.4px) | sans | Drawer title, full-screen error heading, project picker label |
| Mono | 13px | 400 (regular) | 1.4 (18.2px) | mono | Resource names, CRD UIDs, status badge text, log stream, task IDs, namespace prefix |

**Density rationale (14px body, not 16px):**

Operator dashboards optimize for *information density per viewport*, not reading comfort over long-form prose. Argo CD UI uses 13–14px; k9s uses 12–13px terminal sizing; Lens uses 14px. The dashboard inherits this convention. 16px body would force horizontal scrolling on dense DAGs.

**Heading rationale (one size, 18px):**

The dashboard has no marketing hierarchy — no H1/H2/H3 ladder, no display type. The only headings appear in drawer titles and full-screen empty/error states. One size suffices.

**Forbidden:**

- Bold weight (700+) — semibold (600) is the heaviest weight permitted; 700+ produces visual noise at 12–14px sizes.
- Italics — operator dashboards never need them. Distracting at small sizes.
- Underlines on body text — reserved for the *single* anchor element pattern (drawer's "view in kubectl" link).
- Text shadows, letter-spacing tweaks, font-feature-settings — adds bytes, fights system-font defaults.

## Color

The dashboard ships with **dark mode as default** and a light-mode variant. Dark is default because:

- Argo CD UI, k9s, Lens, OpenLens, Octant, KubeShark, KubeView all default to dark. Operator muscle memory expects it.
- Status colors (red/yellow/green) have better legibility on dark backgrounds for the long-glance scanning pattern operators use.
- Reduces eye strain during the multi-hour "watch a run go" workflow.

A user can toggle to light via a system-preference media query OR a header gear-icon toggle (persists to `localStorage`).

| Foreground muted | `--text-muted` | `#4B5563` (slate-600) |
| Border subtle | `--border-subtle` | `#D1D5DB` (slate-300) |

**Accent reserved-for list (10% rule — EXPLICIT, never "all interactive elements"):**

1. Primary CTA button background (e.g., "Copy `tide apply` command" empty-state CTA)
2. Currently-selected node ring (2px outer ring, accent color)
3. Currently-active wave-band edge highlight (only the band whose tasks are dispatching)
4. SSE-connected status dot (header connection indicator)
5. Focus ring outline on keyboard focus (accessibility — never disabled, never just `outline:none`)
6. Connection-status pill background when state is "connected"

That is the entire list. Buttons, links, hovered nodes, drawer headers, badges, the resizable-divider grip — **none of these use accent**. Use `--text-primary` on `--surface-overlay` for hover, `--border-strong` for selection-without-action.

**Color contrast (WCAG 2.1 AA — non-negotiable):**

| Foreground / Background | Ratio | Status |
|-------------------------|-------|--------|
| `--text-primary` / `--surface-base` (dark) | 14.8:1 | Passes AAA |
| `--text-muted` / `--surface-base` (dark) | 5.7:1 | Passes AA |
| `--text-primary` / `--surface-base` (light) | 17.0:1 | Passes AAA |
| `--text-muted` / `--surface-base` (light) | 7.1:1 | Passes AA |
| `--accent` / `--surface-base` (dark — as button bg with white text) | 4.6:1 | Passes AA (large text) |
| All status badge foreground / badge background pairs | ≥ 4.5:1 | Verified per status (see Status Vocabulary) |

## Component Inventory

**15 components.** Each has a one-paragraph contract: what state it owns, what events it emits, what slots/props it exposes. Planner uses this list as task DAG seeds; executor reads each row as a component-shaped task brief.

---

### 1. `<AppShell>`

The top-level layout. Owns no domain state. Renders the persistent header bar (height `space-12` = 48px), the resizable two-pane split below, and the `<ToastContainer>` portal mount.

**Props:** none. **State:** `splitRatio: number` (0.3 — 0.7, default 0.5; persisted to `localStorage` key `tide.dashboard.split-ratio`).

**Slots:** `<Header />` (always), `<PlanningDAGView />` left, `<ExecutionDAGView />` right, `<TaskDetailDrawer />` (conditionally mounted as right-side slide-in overlaying both panes, *not* a third pane).

**Events emitted:** none — child components emit upward via React Context (`AppShellContext`).

**Layout primitives:** CSS Grid for header-vs-body; CSS Flex with `flex-basis` driven by `splitRatio` for the two panes.

### 2. `<Header>`

The persistent top bar. Renders left-to-right:

1. **TIDE wordmark** (text-only `TIDE` in mono, 18px semibold, no logo image — saves bytes). Subtle wave-shaped underline via SVG `<path>` 1px stroke in `--accent` (only place a "wave" decoration appears in chrome).
2. **Project picker** (`<ProjectPicker>` component) — flex-grow center-left.
3. **Connection status pill** (`<ConnectionStatusIndicator>`) — center-right.
4. **Theme toggle** (gear icon → dropdown with `system | dark | light` radio group).

**Props:** none. **State:** none (delegates).

**Height:** 48px fixed. **Border-bottom:** 1px `--border-subtle`. **Background:** `--surface-raised`.

### 3. `<PlanningDAGView>` (left pane)

Renders the Planning DAG for the currently-selected Project: `<ProjectNode>` → `<MilestoneNode>` (nested) → `<PhaseNode>` (nested) → `<PlanNode>`.

**Layout:** dagre top-down (`rankdir: TB`). Hierarchy is shown via nested @xyflow `parentNode` containment — Milestones nest inside their Project, Phases nest inside their Milestone, Plans nest inside their Phase. Per D-D1 the README spec calls for "nested `subgraph` containment for the planning graph"; React Flow v12's `parentNode` + `extent: 'parent'` is the implementation.

**Props:**
```ts
type Props = {
  projectName: string;
  onPlanClick: (planName: string) => void; // swaps right pane
}
```

**State (local):** dagre node positions (recomputed on data change). **Subscribes to:** `useProjectEvents(projectName)` hook (the SSE source).

**Empty state (rare — Project exists but no Milestone yet):** centered text "Awaiting initial milestone — orchestrator is composing the rising tide…" in `--text-muted`.

**Selected-plan affordance:** the Plan node corresponding to `selectedPlanName` (parent state) gets a 2px accent ring (`--accent`) AND its parent Phase node gets a 1px `--border-strong` ring (breadcrumb effect).

---

### 4. `<ExecutionDAGView>` (right pane)

Renders the Execution DAG for the currently-selected Plan: `<TaskNode>` × N, grouped into wave bands. Cross-wave edges visible.

**Layout:** dagre left-right (`rankdir: LR`). Wave bands implemented as `<WaveBandBackground>` components (see #6) — *not* @xyflow `parentNode` containment, because tasks belong to a wave by computation (`pkg/dag.ComputeWaves`), not by parent-child. Bands are presentational; tasks remain peers.

**Props:**
```ts
type Props = {
  planName: string;
  onTaskClick: (taskName: string) => void; // opens TaskDetailDrawer
}
```

**State (local):** dagre positions, wave-band rectangles (computed from task positions). **Subscribes to:** `useProjectEvents` (filtered to the Plan's tasks).

**Cross-wave edges** are visually distinct from intra-wave edges:
- Intra-wave edges: 1px `--border-subtle` straight line.
- Cross-wave edges: 1.5px `--text-muted` *bezier* line with subtle arrowhead. The bezier curve signals "this edge crosses a wave boundary."
- Active-dispatch edges (the wave currently dispatching downstream tasks): 1.5px `--accent` (only the few edges feeding the active wave).

**Empty state (Plan exists, no tasks yet — admission webhook still computing):** "Plan accepted — computing wave structure…" in `--text-muted`.

### 6. `<WaveBandBackground>`

Renders a single wave's background band inside the Execution DAG. Pure SVG `<rect>` element drawn behind the dagre-laid-out task nodes.

**Props:**
```ts
type Props = {
  waveIndex: number;       // 0-based
  bounds: { x, y, w, h };  // computed from member task positions + padding
  isActiveDispatch: boolean; // currently dispatching this wave
  taskCount: number;
}
```

**Visual:**

- **Inactive bands** (already-succeeded, not-yet-dispatched): `fill: --surface-overlay; opacity: 0.4;`
- **Active band** (currently dispatching): `fill: --surface-overlay; opacity: 0.6; stroke: --accent; stroke-width: 1px; stroke-dasharray: 4 2` (subtle dashed border).
- **Failed band** (any task in this wave is `Failed` or `PushLeakBlocked`): `fill: --destructive-muted; opacity: 0.3;`

**Label:** Top-left corner of each band, label-size mono text: `WAVE <N> · <count> tasks` in `--text-muted`. Click on label opens a small popover with the full task list (defer to v1.x if click target is hard to reach with overlapping nodes — read-only label is sufficient for v1.0).

**Composition with @xyflow/react v12:** Wave bands render in a custom `<Background>` slot via the `<Background>` component override pattern, ensuring they sit below nodes and edges in z-order. Implementation reference: React Flow v12 "Custom Background" recipe.

### 7. `<TaskDetailDrawer>`

Slide-in panel from the **right edge** of the viewport. Overlays both panes (does NOT push them). Width: 420px (fixed); height: 100% viewport height minus header.

**Animation:** translateX from `100%` to `0` over 180ms, `cubic-bezier(0.4, 0, 0.2, 1)`. Backdrop is a `--surface-base / 0.6` semi-transparent layer that closes the drawer on click.

**Props:**
```ts
type Props = {
  taskName: string | null; // null = closed
  onClose: () => void;
}
```

**Internal layout (top to bottom):**

1. **Header** (`space-6` padding): drawer title `task/<name>` (heading 18px), close button (`X` icon, top-right, 24px hit area).
2. **Status row** (`space-4` margin top): `<StatusBadge>` + elapsed-time text ("running for 4m 12s") in mono.
3. **Metadata grid** (label/value pairs): namespace, attempt (`2 of 3`), pod name (mono), exit code (if completed), wave index, scheduled-at (`2026-05-16 14:32:01 UTC`), envelope path (PVC path, mono, truncated with tooltip showing full path).
4. **Actions row**: row of clipboard-copy buttons relevant to current status — see "Clipboard-copy action affordance" (#10) below.
5. **Conditions list**: bulleted list of CRD `Status.Conditions[]` with type + reason + age.
6. **"Open log stream" button**: full-width button at bottom-of-content, accent-color border + accent-color text, opens `<PodLogStreamer>` inline below.

**Accessibility:** `role="dialog"`, `aria-modal="true"`, `aria-labelledby` pointing at drawer title. Focus traps inside drawer when open; Escape key closes; focus returns to triggering node on close.

### 8. `<PodLogStreamer>`

The DASH-04 surface — opens inline inside `<TaskDetailDrawer>` (NOT a separate page). Renders SSE-streamed pod log lines.

**Props:**
```ts
type Props = {
  taskName: string;
  onClose: () => void;
}
```

**Internal state:** `lines: string[]` (ring-buffer-capped at 5000 lines — see Bundle gates rationale); `autoScroll: boolean` (default `true`); `paused: boolean` (auto-set true if user scrolls up).

**Visual:**

- **Container:** background `--surface-overlay`, border `1px --border-subtle`, mono font, font-size 13px, line-height 1.4.
- **Toolbar** (top, 32px tall): pause/resume button (auto-derived from `paused`), "Copy logs" button (copies `lines.join('\n')` to clipboard), line-wrap toggle (off by default — operators want to see real line lengths), close button.
- **Viewport** (scrollable, fills remaining drawer height): each line rendered as a `<div>` with `white-space: pre` (or `pre-wrap` when wrap-toggle on). ANSI color escape sequences parsed via a bespoke 80-line parser (`src/lib/ansi.ts`) — supports SGR codes 30–37, 90–97, 40–47, 100–107, 1 (bold), 22 (no-bold), 39 (default fg), 49 (default bg), 0 (reset). No other SGR codes — anything else is stripped before render.

**Pitfall 22 mitigation (client side):** When component unmounts (drawer close, task switch, navigation away), the EventSource is `.close()`d in `useEffect` cleanup. The backend already handles its side (5-min idle close + client-disconnect-cleanup defer per D-D4).

**Connection state visual:**

- Connecting: spinner icon (lucide `Loader2`, animate-spin) + "Connecting to log stream…" placeholder in `--text-muted`.
- Connected, no lines yet: "Waiting for output…" in `--text-muted`.
- Disconnected (5-min idle close, or backend dropped): yellow `AlertTriangle` icon + "Log stream closed by backend (5 min idle). [Reconnect]" — clicking Reconnect opens a new EventSource.

**No search/filter in v1.0** — deferred to v1.x. Operators use `Cmd-F` browser-native find for now.

### 9. `<ProjectPicker>`

Header dropdown. Renders the currently-selected Project name + a chevron-down icon. Click opens a dropdown menu listing all Projects the dashboard SA can read (one entry per `Project.metadata.name`).

**Props:** none. **State:** `selectedProjectName: string`, `isOpen: boolean`.

**Behavior:**

- If `count === 0`: dropdown is disabled, shows `"No projects"` in `--text-muted`.
- If `count === 1`: dropdown still renders (no auto-select-only) but is a single-item list — keeps the UX consistent across single- and multi-Project clusters.
- If `count > 1`: full dropdown list with namespace prefix (e.g., `default/my-project`, `team-a/staging-tide`).

**Keyboard shortcut:** `/` key (when no input is focused) opens the picker focused on a search input. Search filters by substring match on `<namespace>/<name>`.

**No separate "Projects listing page" in v1.0** — deferred. The picker is the only Projects surface.

### 10. Clipboard-Copy Action Affordance

This is the **D-D6 surface** — buttons that copy `tide` CLI commands to the user's clipboard and emit a confirmation toast. NOT a single component; a **pattern** with a shared `<CopyCommandButton>` primitive.

**`<CopyCommandButton>` props:**
```ts
type Props = {
  command: string;     // exact CLI text to copy
  label: string;       // button label (e.g., "Approve project")
  variant?: 'primary' | 'destructive' | 'secondary'; // styling
  description?: string; // optional helper text below button
}
```

**Visual:**

- `primary`: background `--accent`, foreground `#000`, 13px label-weight text.
- `destructive`: background transparent, border `1px --destructive`, foreground `--destructive`.
- `secondary`: background transparent, border `1px --border-subtle`, foreground `--text-primary`.

**Click behavior:**

1. Write `command` to clipboard via `navigator.clipboard.writeText`.
2. On success: emit toast (#11) with copy `Copied: <command> — paste in your terminal to run.`
3. On failure (clipboard API unavailable / blocked): toast with error variant: `Could not copy to clipboard. Command: <command>`. Toast remains visible 8s instead of 4s in this path.

**Placement (v1.0 — drawer-only to avoid clipboard-spam):**

All Copy-Command buttons live exclusively inside `<TaskDetailDrawer>`'s "Actions" row. Status badges in nodes/headers do NOT have inline copy buttons — clicking the node opens the drawer, copy buttons live there.

## Status Vocabulary

Ten CRD `.status.phase` values, mapped to color + icon + label + screen-reader description. Used by `<StatusBadge>` (a primitive that all node and drawer components consume).

**`<StatusBadge>` shape:**

```
┌─────────────────┐
│ ◐ Running       │   ← 14px lucide icon + label-size mono text
└─────────────────┘
```

- Inline-flex, padding `space-1` `space-2`, border-radius 4px, font 12px mono, semibold (600).
- Background: `<status-color> / 0.15` (rgba w/ 15% alpha — subtle tinted fill).
- Border: 1px `<status-color> / 0.4`.
- Foreground: `<status-color>` direct (full saturation for legibility).
- Icon: 14px, current color (inherits foreground).

| CRD Status | Color | lucide Icon | Label | Animation | SR Description |
|------------|-------|-------------|-------|-----------|----------------|
| `Pending` | `--status-pending` | `Circle` (hollow) | `Pending` | none | "Pending dispatch" |
| `Dispatching` | `--status-running` | `Loader2` | `Dispatching` | `animate-spin` | "Dispatching — Job creation in progress" |
| `Running` | `--status-running` | `CircleDot` | `Running` | `animate-pulse` | "Running" |
| `AwaitingApproval` | `--status-warning` | `Hand` | `Awaiting approval` | none | "Awaiting human approval — run `tide approve` to advance" |
| `Paused` | `--status-warning` | `Pause` | `Paused` | none | "Paused at slack tide — run `tide resume` to advance" |
| `Succeeded` | `--status-success` | `CircleCheck` | `Succeeded` | none | "Succeeded" |
| `Failed` | `--status-error` | `CircleX` | `Failed` | none | "Failed — see logs and Conditions for details" |
| `PushLeaseFailed` | `--status-error` | `LockKeyhole` | `Push lease failed` | none | "Push lease failed — concurrent push detected by force-with-lease" |
| `PushLeakBlocked` | `--status-blocked` | `ShieldAlert` | `Push leak blocked` | none | "Push blocked by gitleaks — a secret pattern was detected in the diff" |
| `Rejected` | `--status-error` | `Ban` | `Rejected` | none | "Rejected by operator — run `tide resume` to clear" |

**Color-blindness considerations:**

- Every badge has icon + label + color. **Never color-only.**
- Status families that share a color hue (e.g., `Failed` + `PushLeaseFailed` + `PushLeakBlocked` all red/purple-adjacent) have distinct icons (`CircleX`, `LockKeyhole`, `ShieldAlert`).
- The `--status-blocked` purple-500 is deliberately distinct from `--status-error` red-500 for protanopia/deuteranopia distinguishability (both have very different luminance values).

**Animation rules:**

- `animate-spin` and `animate-pulse` respect `prefers-reduced-motion: reduce` — when set, both reduce to no animation. The icon still displays; only the rotation/pulse stops.

## Layout & Interaction Specs

### Resizable Divider

- 6px wide visible track between the two panes; expands hit area to 12px on hover/focus.
- Cursor: `col-resize`.
- Constraints: `splitRatio` clamped to `[0.3, 0.7]` — neither pane goes below 30% of viewport width. Below 1024px viewport, the divider locks at `splitRatio = 0.5` and dragging is disabled (small viewports are a v1.x concern; primary target is ≥1280px operator laptops).
- Keyboard: when focused (Tab to grip), Left/Right arrow keys adjust by ±0.02 per press; Home/End jump to clamp limits.
- Persistence: `splitRatio` writes to `localStorage['tide.dashboard.split-ratio']` on `mouseup` (debounced). Restores on next session.

### Drawer Slide

- Slides in from **right edge** (translateX 100% → 0). 420px fixed width.
- Backdrop fades in over the same 180ms duration. Backdrop covers both panes (NOT the header — header remains interactive while drawer is open).
- Close triggers: backdrop click, Escape key, X button. Focus returns to the triggering `<TaskNode>` on close.

### Plan-click-swaps-right-pane

- Click on a `<PlanNode>` in the left pane:
  - Right pane data fetch begins immediately (no transition delay).
  - Right pane fades out over 100ms, fades in over 100ms when new graph is ready.
  - URL hash updates to `#/plan/<plan-name>` (browser-native History API; no router library). Deep-linkable.

### Scroll Behavior

- **Outside panes** (page level): no scroll. Header + body fill 100% viewport; panes scroll internally as needed.
- **Inside left/right panes:** @xyflow/react owns pan/zoom (mouse wheel = pan vertical with `Shift` for horizontal; `Cmd/Ctrl + wheel` = zoom; clicking empty space + drag = pan). Native browser scroll disabled within pane bounds.
- **Inside drawer:** vertical scroll if content exceeds drawer height.
- **Inside log streamer:** vertical scroll; scrolling up auto-disables `autoScroll`; scrolling to bottom re-enables.

## Accessibility

WCAG 2.1 AA baseline. Non-negotiable.

### Keyboard Map (global)

| Key | Action | Context |
|-----|--------|---------|
| `Tab` | Focus next interactive element | Global. Follows visual order: Header → Project picker → Connection pill → Theme toggle → Left pane (depth-first into nodes) → Right pane (depth-first) → Drawer (when open). |
| `Shift+Tab` | Focus previous | Global |
| `Enter` | Activate focused element (click action) | Buttons, nodes, dropdown items |
| `Space` | Same as Enter for buttons | Buttons only |
| `Escape` | Close drawer / dropdown / dismiss toast | Drawer, dropdown, toast |
| `Arrow keys` | Navigate to adjacent node in current pane | Pane focused on a node — uses `@xyflow/react` v12's built-in `useKeyPress` hook + a custom "find-adjacent-node-by-position" helper |
| `/` | Open Project picker with search focused | Global, when no input is focused |
| `?` | Open keyboard shortcuts help modal | Global (the modal is the only "help" surface in v1.0; lists this exact table) |
| `Left` / `Right` arrows (with grip focused) | Resize divider | Resizable divider grip focused |

### Focus Rings

- **All interactive elements** show a visible focus ring on `:focus-visible`. Ring: 2px solid `--accent`, 2px offset.
- Focus rings are NEVER removed (no `outline: none` without replacement).
- Mouse-only interaction does not show focus ring (`:focus-visible` semantics).

### Screen Reader Contract

- **Status badges:** always include text label (not icon-only). The screen-reader description column in the Status Vocabulary table is the `aria-label`.
- **Live region:** the right pane and the connection-status indicator have `aria-live="polite"` so SSE-driven state changes are announced without interrupting the user.
- **Drawer:** `role="dialog" aria-modal="true" aria-labelledby="<drawer-title-id>"`.
- **Toasts:** `role="status"` (info/success) or `role="alert"` (warning/error) — error variants interrupt; success variants don't.
- **DAG nodes:** each node has `aria-label="<kind> <name>, status <status>"` and `role="button"` since they're clickable.
- **Custom @xyflow elements:** edges are decorative (`aria-hidden="true"`) — the graph relationship is conveyed via the node aria-labels and the layout direction.

### Color Contrast

Already declared in the Color section — every foreground/background pair in the design meets WCAG AA (4.5:1 for body, 3:1 for large text and graphics). CI lint runs `axe-core` against rendered Storybook (or equivalent) on every PR.

### Motion

`prefers-reduced-motion: reduce` disables all spin/pulse/slide animations. State changes still happen — they just don't animate.

**Researcher-identified UI deferrals (v1.x candidates):**
