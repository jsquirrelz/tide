Output:
  - Two new GET-only dashboard backend endpoints that surface the rich Plan
    + Task detail the existing minimal SSE projection cannot carry:
      * `GET /api/v1/plans/{name}?namespace=foo` → plan + child task array
        with `(name, phase, waveIndex, attempt, dependsOn[])` per task
        (waveIndex resolved by looking up the owning Wave CRDs by
        Spec.PlanRef + Spec.WaveIndex; task list filtered by Spec.PlanRef).
      * `GET /api/v1/tasks/{name}?namespace=foo` → rich task detail matching
        TaskDetailData (status, attempt, attemptMax, podName, exitCode,
        waveIndex, scheduledAt, envelopePath, elapsedText, conditions[]).
  - Three new frontend hooks:
      * `useProjects()` — wraps the existing `fetchProjects()` helper
        in `lib/api.ts` with React state + `refetch` callback. NOT
        SSE-driven (the project list is stable during a session; refetch
        button satisfies any need).
      * `useTasks(projectName, planName)` — composes `useSSEStream` against
        `/api/v1/projects/{name}/events` and `fetchPlan(planName)`. The SSE
        events are a "refresh trigger" only (the minimal projection lacks
        DAG fields); on each event whose payload `kind` is "Plan" or
        "Task" AND payload `planRef === planName` (or `name === planName`
        for the Plan itself), the hook calls `fetchPlan(planName)` and
        rebuilds the ExecutionPlanData shape. Initial fetch happens on
        planName change (and on `projectName` change reset).
      * `useTaskDetail(projectName, taskName)` — same pattern; SSE refresh-
        trigger + `fetchTask(taskName)`. Returns `null` when `taskName` is
        null. Filter: payload `kind === "Task"` AND `name === taskName`.
  - `App.tsx` modifications: replace placeholder defaults, mount
    `<ProjectPicker>` into `<Header projectPicker={...}>`, wire the three
    hook outputs into the existing component props (no signature changes
    needed — ExecutionDAGView/TaskDetailDrawer already accept `plan` and
    `task` props sized for these shapes per plans 04-13 + 04-15).
  - Tests for every new hook + new endpoint following the existing patterns
    (Vitest + FakeEventSource + fetch stub; Go table-driven tests with the
    controller-runtime fake.NewClientBuilder).

The plan is intentionally scoped to the WIRING; it does NOT touch
`cmd/manager` (out of scope), does NOT modify `charts/tide/values.yaml`
