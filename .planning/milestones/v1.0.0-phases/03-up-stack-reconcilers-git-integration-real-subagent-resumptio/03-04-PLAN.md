<objective>
Wave 2 utility package — `internal/gitleaks/` — a thin Go-library wrapper around `github.com/zricethezav/gitleaks/v8/detect` (v8.30.1). Scans git diffs for known secret patterns; returns structured findings. Default rule set is embedded at build time via `go:embed`; per-Project overrides land via ConfigMap at runtime (the ConfigMap-mount mechanics live in the push Job spec — plan 03-06).

Purpose: Defense-in-depth at the push boundary (D-B3). `internal/harness/redact/` (Phase 2 D-F4) is the harness-side first line; gitleaks is the push-side second line — catches anything the harness redact missed, plus broader patterns gitleaks ships.

Output: Three Go files + one embedded TOML + unit tests proving fixed-pattern detection works end-to-end. Pure-Go; no shell-out; no K8s API surface.

Why a separate plan from 03-06 (the push Job binary): keeps the scanner unit-tested in isolation; allows plan 03-06 to focus on Job lifecycle, envelope IO, and push orchestration without intermixing scanner-correctness tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Push Job process → outbound git remote | gitleaks scans the staged diff BEFORE the go-git Push call. A finding short-circuits the push (push Job exits non-zero) — secret never leaves the cluster. |
| External (kubectl user) → ConfigMap override | Per-Project ConfigMap (referenced by Project.Spec.git.leaksConfigRef) carries custom rule TOML. The ConfigMap MUST live in the same namespace as the Project (cross-namespace ConfigMap refs are NOT permitted in v1.0 — enforced at the push Job spec level in plan 03-06). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-304 (Tampering — gitleaks rule overrides) | Per-Project ConfigMap | mitigate (partial here, full enforcement in 03-06) | This plan ships the loader; it accepts a TOML path and trusts the file contents (the loader is content-agnostic). The boundary enforcement — ConfigMap MUST live in the Project's namespace, Project owner cannot escalate via override — is enforced by the push Job spec (plan 03-06) which restricts the ConfigMap volume mount to a single namespace-scoped ref. |
| Defense-in-depth | Information Disclosure | gitleaks default ruleset | mitigate | Default rules are embedded at build time (go:embed) — cannot be tampered with at runtime. Custom rules via TOML COMPOSE with defaults (useDefault=true) — they can ADD rules but cannot REMOVE the embedded defaults. Test 5 of Task 1 asserts this composition property. |
</threat_model>
