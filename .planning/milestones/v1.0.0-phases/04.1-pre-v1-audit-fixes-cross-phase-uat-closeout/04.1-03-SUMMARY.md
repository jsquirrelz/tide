# Phase 04.1 Plan 03: P1.3 DefaultCaps Helper — 300s/60s Deadline Mismatch Closed

Introduced `DefaultCaps(caps, kind JobKind)` in `internal/dispatch/podjob/caps.go` — the single shared helper that applies a Kind-appropriate wall-clock floor. Both executor consumers (token mint in `task_controller.go`, activeDeadlineSeconds in `jobspec.go`) now route through it, making the 300s↔60s deadline drift class structurally impossible.

## What Was Done

**Bug closed (P1.3):** When `task.Spec.Caps` was nil or `WallClockSeconds <= 0`:
- Token mint (`task_controller.go`) applied the 300s floor → signed token valid for 360s
- `BuildJobSpec` (`jobspec.go`) read caps directly with no floor → `activeDeadline = 0 + 60 = 60s`

Result: Pod killed at t=60s while the token remained valid until t=360s — a misleading "auth" error on cold clusters.

**Fix:** Both call sites now route through `DefaultCaps(caps, JobKindExecutor)`. The old `const defaultWallClockFloorSeconds int32 = 300` at `task_controller.go:64-68` is deleted — one source of truth in the `podjob` package.

## Files

### Created

**`internal/dispatch/podjob/caps.go`** (new, 75 lines)
- `JobKind` string type with `JobKindExecutor` and `JobKindPlanner` constants
- `executorCapsFloorSeconds int32 = 300` — covers image pull + scheduler delay + init container
- `plannerCapsFloorSeconds int32 = 600` — adds Anthropic API call latency on top of pod startup
- `DefaultCaps(caps *tidev1alpha1.Caps, kind JobKind) *tidev1alpha1.Caps` — returns caps unchanged if `WallClockSeconds > 0`, otherwise returns a new `*Caps` with the Kind-appropriate floor applied, preserving non-zero Iterations/InputTokens/OutputTokens fields

**`internal/dispatch/podjob/caps_test.go`** (new, 142 lines)
- `TestDefaultCaps` — 11 table-driven subtests covering nil/zero/negative/positive WallClockSeconds for both JobKindExecutor (300s floor) and JobKindPlanner (600s floor); verifies non-wall-clock field preservation
- `TestDefaultCaps_NilCapsDeadlineMatch` — proves executor (360s) and planner (660s) nil-caps deadlines are tautologically identical per Kind, plus drift guard asserting planner floor > executor floor

### Modified

**`internal/dispatch/podjob/jobspec.go`** — `BuildJobSpec` now derives `wallClockSeconds` via `DefaultCaps(opts.Task.Spec.Caps, JobKindExecutor).WallClockSeconds` instead of reading caps directly. The `activeDeadline` math (`wallClockSeconds + DefaultWallClockGraceSeconds`) is unchanged.

**`internal/controller/task_controller.go`** — Token mint at step 8 now uses `podjob.DefaultCaps(task.Spec.Caps, podjob.JobKindExecutor).WallClockSeconds`. The old `const defaultWallClockFloorSeconds int32 = 300` is deleted.
