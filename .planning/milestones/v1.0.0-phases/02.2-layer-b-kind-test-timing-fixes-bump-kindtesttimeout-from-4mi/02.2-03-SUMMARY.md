## Issues Encountered

### NEW pre-existing chart/binary flag mismatch (out-of-scope; routed to planner)

**1. `charts/tide/values.yaml:20` passes `--metrics-bind-address=:8443` as a manager-binary CLI flag, but `cmd/manager/main.go` does NOT define that flag.** This caused the `tide-controller-manager` Pod to enter `CrashLoopBackOff` on both the clean run and the rerun (after Plan 02.2-03's chart fix unblocked the PVC issue that was masking this earlier). The Pod's container exits 2 with `flag provided but not defined: -metrics-bind-address`, kubelet restarts, the flag is still undefined, and the cycle continues with exponential backoff. helm install --wait blocks indefinitely waiting for Pod Ready; outer `timeout 300s go test` fires.

This is a pre-existing bug latent since Phase 1 Plan 11 (commit `a6e16c6 feat(01-11): helmify-driven Helm chart pair`). It was never caught because:
- Phase 1 was envtest-only (no kind tests; the manager binary's `flag.Parse()` never ran against the chart's `manager.args` list)
- Phase 02 added Layer B kind tests but the PVC issue in Plan 02.2-01 masked Pod startup (Pod stuck in Pending, never reached container start)
- Plan 02.2-01 fixed cert-manager + helm --replace + cleanupKindCluster but never exercised the manager binary's flag parsing
- Plan 02.2-03 fixed the PVC issue → first time the manager binary's `flag.Parse()` ran against the chart's args at runtime → flag mismatch revealed

**Routed to planner** via the `02.2-03-VERIFICATION.md` Section 5 "Fix landscape" (Options A through D). The cheapest path is likely Option A (define `--metrics-bind-address` in `cmd/manager/main.go`); a thorough audit (Option C) is also recommended to catch any other latent chart/binary flag mismatches.

This is a **Rule 4 (architectural)** scope-out, not a Rule 1/2/3 auto-fix candidate. Plan 02.2-03's `<files>` is strictly limited to chart PVC + test override + verification artifact + Plan 02.2-02 edit; touching `cmd/manager/main.go` (or removing `--metrics-bind-address` from `charts/tide/values.yaml`) would constitute scope creep outside the plan's explicit boundary. The new blocker is documented in the verification artifact with sufficient root-cause analysis for the next debug-slice planner to author a fix.

### Layer A envtest ART-01 flake on clean run only (not a regression)

**2. `ART-01: init Job completion sets Project phase to Initialized` envtest spec timed out on the clean run.** `Eventually(... should be created).WithTimeout(20s)` fired with `jobs.batch tide-init-{uid} not found`. Upstream error message in the same log block: `Operation cannot be fulfilled on projects.tideproject.k8s ... the object has been modified; please apply your changes to the latest version and try again` (ResourceVersion conflict). The rerun showed all 18/18 envtest specs PASS in 20.176s. This is consistent with a residual race condition under load in the init reconciler harness — a known-flaky area last touched in commit `523382a fix(02-13): make integration envtest suite deterministically green`.

**Not a Plan 02.2-03 regression** — this plan only touched `charts/tide/templates/projects-pvc.yaml`, `charts/tide/values.yaml`, and `test/integration/kind/suite_test.go`. None of these affect Layer A envtest. Surfaced coincidentally during this runtime verification. A follow-up patch in the appropriate phase could bump ART-01's `Eventually(...).WithTimeout(20s)` to 30s or tighten the polling interval.
