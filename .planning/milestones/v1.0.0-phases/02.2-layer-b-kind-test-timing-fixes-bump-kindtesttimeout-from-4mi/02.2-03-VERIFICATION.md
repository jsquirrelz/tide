### Root-Cause Summary (for the next debug plan)

1. **Chart's `values.yaml:18-24` `manager.args` block passes `--metrics-bind-address=:8443`** as a CLI flag to the manager binary. The chart was generated by `helmify v0.4.17` in Phase 1 Plan 11 (commit `a6e16c6`) from a Kustomize base that included controller-runtime's standard `--metrics-bind-address` flag — but the standard kubebuilder scaffold defines that flag inside the Go binary, while TIDE's `cmd/manager/main.go` does NOT include the kubebuilder-standard flag set. TIDE's custom main.go defines a smaller flag set (`--config`, `--leader-elect`, `--watch-namespace`, `--subagent-image`, `--credproxy-image`, `--default-file-touch-mode`, `--rate-limit-default-rpm`, `--rate-limit-default-burst`, plus opts.BindFlags for zap-log-level). **No `--metrics-bind-address` definition exists.**

2. **The manager binary exits immediately on the unrecognized flag** — Go's `flag.Parse()` calls `Usage()` and exits 2 when it encounters an undefined flag. Container's exit-2 → kubelet restarts the Pod → flag still undefined → restart loop → `CrashLoopBackOff` with exponential backoff.

3. **kube-scheduler does NOT mark the Pod NotReady until kubelet reports the failure** — for Helm `--wait`'s purposes, the Pod is `0/1 Ready` and stays that way. helm install --wait blocks on its 5m budget; meanwhile the outer `timeout 300s go test` fires at 5m and SIGTERMs the test process.

4. **This is a pre-existing chart bug since Phase 1 Plan 11** (`charts/tide/values.yaml:18-24`). It was never caught because:
   - Phase 1 was envtest-only (no kind tests; main.go's flag set never ran against the chart's args)
