**Corrected root cause citation (per `.planning/debug/credproxy-dispatch-guard.md` §Reasoning Checkpoint):**

Cascade-5 was originally hypothesized in `02.2-06-VERIFICATION.md` §Section 5 as a "parent Wave Ready check" gating dispatch. That hypothesis was WRONG. The corrected root cause (from `.planning/debug/credproxy-dispatch-guard.md` §Reasoning Checkpoint) is that `task_controller.go:217` (Step 3 of `reconcileDispatch`) calls `resolveProject(ctx, task)` which returned `"no project found in namespace credproxy-test"` because the credproxy fixture created only Plan+Task without an accompanying Project CRD. **CASCADE-6 ROOT CAUSE:**

Two distinct failures surface as cascade-6:

**Failure A — credproxy_test.go HARN-03 (60s pod-wait timeout):**
- Classification: **spec-flake** (timeout budget too tight)
- The `credproxy-task` does NOT get a Job/Pod created within 60s because of a `ResourceVersion` conflict at first reconcile: the task controller receives the Task at 15:50:05 and tries to update status, but the Task was concurrently modified (by PlanReconciler's `stampTaskLabels` / wave materialization). The conflict causes an immediate error + requeue. The retry resolves the conflict, but the retry + dispatch path (Job create) takes > 60s total.
- Evidence: `"error":"Operation cannot be fulfilled on tasks.tideproject.k8s \"credproxy-task\": the object has been modified; please apply your changes to the latest version and try again"` in manager logs at 15:50:05; `task cleanup` at 15:51:10 (AfterEach ran deleteNamespace 65s after spec start)
- Fix shape: increase the `Eventually(_, 60*time.Second, ...)` for the Pod-wait assertion to `90*time.Second` or `120*time.Second` to allow for the ResourceVersion conflict retry cycle

**Failure B — failure_test.go AC3 (120s task-phase timeout):**
- Classification: **spec-flake** (timeout budget too tight)  
- The `beta-fail` task (testMode=fail-exit-1) does not reach `Failed` phase within 120s. This spec hasn't run end-to-end since Phase 02.1. The dispatch cycle for a real Job (create → Pod schedule → container start → exit-1 → Job Failed → Task status update) takes more than 120s in the kind cluster.
- Fix shape: increase the `Eventually(_, 120*time.Second, ...)` for the failed-phase assertion to `180*time.Second` or `240*time.Second`

**Why this was not predictable from prior cascades:** All 5 prior cascades were structural issues (PVC RWX/RWO, missing chart flags, fixture incompleteness). Cascade-6 is a timing/flakiness issue in specs that have not run end-to-end since Phase 02.1. The `60s` and `120s` `Eventually` timeouts were set during Phase 02.1 when dispatch was faster (no ResourceVersion conflict because no hierarchy parents existed to concurrently modify).
