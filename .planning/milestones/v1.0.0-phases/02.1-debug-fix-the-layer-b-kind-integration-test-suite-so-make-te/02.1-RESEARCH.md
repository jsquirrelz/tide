# Phase 02.1: Debug + Fix Layer B Kind Integration Test Suite — Research

**Researched:** 2026-05-13
**Domain:** K8s integration test infrastructure (kind cluster bootstrap, kustomize-vs-helm parity, cert-manager-gated webhooks, controller-manager image+secret wiring, Ginkgo Layer B specs)
**Confidence:** HIGH for every root cause flagged below — they are verified against source files, kustomize output, and helm-template diffs, not extrapolated from training data.

## Summary

Phase 2 Plan 13 (`02-13-PLAN.md`) shipped the Layer B kind integration test files, the `make test-int` target, the kind cluster.yaml, and the CI workflow. The plan's own summary admits: *"Tests were written and verified to compile on dev machine. Actual runtime timing requires envtest binaries (`make setup-envtest`) and Docker/kind for Layer B."* — i.e. **Layer B has never actually been run green.** Phase 02.1's job is to make it run green for the first time.

The defects are not in the Ginkgo spec bodies (those are mostly correct). The defects are in the **install path** the kind suite uses to deploy the controller: the suite installs via `kustomize build config/default | kubectl apply -f -`, but the controller's Phase 2 wiring REQUIRES three resources that **only the Helm chart provides** — the `tide-signing-key` Secret, the `tide-projects` PVC, and the `tide-subagent` ServiceAccount — plus three Deployment-arg flags (`--subagent-image`, `--credproxy-image`, `--default-file-touch-mode`) that only the chart's Deployment template renders. The result: the manager Pod CrashLoopBackOffs on startup with `TIDE_SIGNING_KEY env var is required (HARN-03)` before any test spec ever runs, the readiness probe never goes green, the webhook Service has no endpoints, and every test that calls `skipIfCRDsOnlyMode()` either skips silently or fails on `Eventually(Task.Status.Phase=Succeeded)` because no controller is reconciling.

There is also one true code-level bug: `credproxy_test.go` greps for the literal string `credproxy listening on 127.0.0.1:8443` in container logs, but no log statement in `cmd/credproxy/main.go` or `internal/credproxy/server.go` emits that string. The closest is `log.Info("starting credential proxy", "listenAddr", ...)` (structured zap JSON, not a plaintext line). That test would fail even if the controller were healthy.

**Primary recommendation:** Replace the `kustomize build config/default` install path in `test/integration/kind/suite_test.go` with `helm install tide ./charts/tide --create-namespace -n tide-system`. The Helm chart already provides every missing piece. This is also more honest: the spec's eventual acceptance test (BOOT-04) is `helm install tide` — testing against the Helm chart at Phase 02.1 catches drift between dev-loop kustomize and production-shape helm now, not at Phase 5. Then fix the credproxy log line as a tiny code change.

The architectural shift is: **stop installing via kustomize in the Layer B suite; install via Helm**. Same chart used by every other path (manual install, BOOT-04 acceptance test, future Helm CI test). One install surface, one place to fix bugs.

### Pattern 1: Helm-install-as-test-bootstrap
**What:** BeforeSuite uses `helm install --wait` instead of `kustomize build | kubectl apply`.
**When to use:** When the application has a Helm chart that is the production install surface AND the chart provides resources the test-target controller depends on (Secrets it `envFrom`s, PVCs it mounts, sidecar SAs).
**Example:**
```go
// Source: cert-manager E2E + kubebuilder book pattern
helmCmd := exec.CommandContext(ctx, "helm", "install", "tide", "./charts/tide",
    "--create-namespace", "-n", "tide-system",
    "--set", "controllerManager.manager.image.tag=test",
    "--set", "images.stubSubagent.tag=test",
    "--set", "images.credProxy.tag=test",
    "--set", "controllerManager.manager.image.pullPolicy=IfNotPresent",
    "--set", "images.stubSubagent.pullPolicy=IfNotPresent",
    "--set", "images.credProxy.pullPolicy=IfNotPresent",
    "--wait", "--timeout", "120s",
)
helmCmd.Dir = filepath.Join("..", "..", "..")
out, err := helmCmd.CombinedOutput()
Expect(err).NotTo(HaveOccurred(), "helm install failed: %s", out)
```

### Pattern 2: imagePullPolicy=IfNotPresent for kind-loaded images
**What:** All images loaded via `kind load docker-image` MUST set `imagePullPolicy: IfNotPresent` (or `Never`). Default for non-`:latest` tags is already `IfNotPresent`, but the existing suite explicitly patches the YAML string to set it — fragile. Helm values give a clean override surface.
**When to use:** Always, in any test that uses `kind load`.
**Example:** `--set controllerManager.manager.image.pullPolicy=IfNotPresent` in helm install args.
[VERIFIED: kind quick-start docs via ctx7]

### Pattern 3: helm --wait replaces bespoke readiness polling
**What:** `helm install --wait` blocks until all chart resources (Deployments, StatefulSets, Jobs, PVCs) report ready. Eliminates the brittle `waitForControllerReady` poll loop.
**When to use:** When the chart includes the workloads under test.
**Example:** `helm install ... --wait --timeout 120s`.

### Anti-Patterns to Avoid
- **Don't add the missing Secret/PVC/SA to `config/default/`.** That moves the helm-chart-only resources into the kustomize overlay and breaks the design rule that the chart is the production install surface (DIST-01). The overlay is for dev-loop `make deploy`; it intentionally omits chart-provisioned state.
- **Don't emit `fmt.Println` for the credproxy log line just to satisfy the test.** Use structured zap (`log.Info("listening", "addr", ...)`) and update the test grep to match the JSON shape. Plaintext log lines for assertion purposes are a smell. (Or: emit a single human-readable status line on top of zap — many controllers do this for boot-banner readability.)
- **Don't tighten the cert-manager readiness wait without bumping the version.** v1.16.2 has known slow webhook startup on kind under Docker Desktop. v1.20+ is faster. [ASSUMED — needs measurement, but version freshness is a no-cost improvement]
- **Don't add a `Wait()` for "endpoints exist" via custom code.** `helm install --wait` covers this; the chart's Deployment `readinessProbe` ensures the Service backing the webhook has at least one ready endpoint before helm returns.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Wait for tide-controller-manager Ready | Bespoke poll loop checking `readyReplicas==1` | `helm install --wait --timeout 120s` | Helm tracks every chart resource's readiness in one call; correct retry semantics; emits structured failure output |
| Apply controller Deployment | `kubectl apply -k config/default` | `helm install tide ./charts/tide` | Helm has the missing Secret/PVC/SA/args; same chart is the production install path |
| Discover the right webhook Service to wait for | Hardcoded service name + custom Endpoints check | Helm chart's `--wait` flag (it waits on Service endpoints behind ready Deployment) | Less code; survives chart refactor |
| Make tide-controller-manager image available in kind | `imagePullPolicy: IfNotPresent` string-patch after kustomize emit | Helm values: `--set controllerManager.manager.image.pullPolicy=IfNotPresent` | First-class chart surface; no fragile string manipulation |
| Verify credproxy bound the listener | Plaintext log-line grep on container logs | (a) Use existing structured zap JSON + match key/value, OR (b) emit one `fmt.Fprintln(os.Stdout, "credproxy listening on 127.0.0.1:8443")` next to the existing `log.Info` for the test to grep | Plaintext-log-for-test-assertion is fragile; either accept the fragility honestly (single boot-banner line in addition to structured logs — fine, common pattern) or update the test to match the structured shape (cleaner but more code) |

**Key insight:** every "don't hand-roll" here points to the same root cause — *the Layer B suite was built as a one-off bootstrap path that diverged from the Helm chart's install path*. Closing that divergence fixes 5 of the 7 specs at once.

## Common Pitfalls

### Pitfall 1: Kustomize-vs-helm parity drift
**What goes wrong:** The kustomize overlay (`config/default`) is the *dev-loop* install path (`make deploy`); the Helm chart is the *production* install path. Phase 2 schema additions (signing-key, projects-PVC, subagent-SA, Phase 2 args) live ONLY in the chart's `augment` layer — they are not in the kustomize overlay. A test that uses kustomize gets a partial install.
**Why it happens:** Helmify regenerates the chart from kustomize output but the `hack/helm/augment-tide-chart.sh` script re-adds project-specific resources on top. Those additions never flow back into kustomize.
**How to avoid:** Always install via Helm in any integration test that exercises Phase 2+ behavior.
**Warning signs:** Manager Pod CrashLoopBackOff with `TIDE_SIGNING_KEY env var is required`; Pods stuck in `ErrImageNeverPull` for `controller:test`; Tasks stuck in `Pending` with "no PVC bound."

### Pitfall 2: Manager image not loaded into kind
**What goes wrong:** `kustomize edit set image controller=controller:test` makes the manifest reference `controller:test`, but if the image was never `kind load`ed, the kubelet errors with `ImagePullBackOff` trying to pull from a public registry.
**Why it happens:** Easy to miss if the manager Dockerfile was added after the initial test setup.
**How to avoid:** `test-int-kind-prep` already builds + loads `controller:test`. Verify the load succeeded with `kind get nodes --name tide-test | xargs -I {} docker exec {} crictl images | grep controller`.
**Status:** Recent commit `548355f fix(02-13): build + load manager image for kind suite; trim invalid testdata` addresses this. Should be working now.

### Pitfall 3: cert-manager not ready when webhook Certificate reconciles
**What goes wrong:** `kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/.../cert-manager.yaml` returns immediately, but cert-manager's own webhook needs ~30-60s to become Ready before it can admit the `Certificate` resource from `config/certmanager/`.
**Why it happens:** Race between cert-manager install and the kustomize bundle's Certificate apply.
**How to avoid:** Wait for `cert-manager-webhook` Deployment Ready before applying anything that creates Certificates. The current suite does this via `kubectl rollout status deployment/cert-manager-webhook -n cert-manager --timeout=120s`. Verify that step still runs first.
**Status:** Existing `installCertManager()` does the right thing.

### Pitfall 4: Webhook Service has no endpoints because manager Pod is unhealthy
**What goes wrong:** `tide-webhook-service` selects Pods with `control-plane=controller-manager`. If the manager Pod is CrashLoopBackOff, the Service has zero endpoints. Plan admission webhook calls timeout with `service "tide-webhook-service" not found` or `connection refused`.
**Why it happens:** Symptom of Pitfall 1 (manager missing TIDE_SIGNING_KEY etc).
**How to avoid:** Fix Pitfall 1 first. The Service is correctly wired in kustomize; the failure is downstream of manager Pod health.

### Pitfall 5: TaskReconciler injects empty subagent/credproxy image
**What goes wrong:** Without `--subagent-image` / `--credproxy-image` flags, `TaskReconciler.SubagentImage` and `.CredproxyImage` are empty strings. `BuildJobSpec` emits a `corev1.Container` with `Image: ""`. Kubelet rejects with `Failed to apply default image tag "": couldn't parse image reference ""`.
**Why it happens:** The kustomize overlay's manager Deployment doesn't append these flags. The Helm chart's `deployment.yaml` does (see lines 27-31: `- --subagent-image=...`).
**How to avoid:** Helm install path provides the flags. If you stay on kustomize for any reason, add the args to `config/manager/manager.yaml` (NOT recommended — pollutes dev overlay).

### Pitfall 6: stub-subagent ServiceAccount missing → Pod create fails
**What goes wrong:** Every Task Job's PodSpec sets `serviceAccountName: tide-subagent` (constant in `jobspec.go:43`). If that SA doesn't exist in the test fixture namespace (`tide-int-test`, `failure-test`, etc.), the K8s API server rejects Pod creation with `serviceaccount "tide-subagent" not found`.
**Why it happens:** The Helm chart creates `tide-subagent` in `.Release.Namespace` only (= `tide-system`). The tests apply Tasks into per-test namespaces.
**How to avoid:** Either (a) chart-create the SA in every namespace via a kustomize-loop or `helm install` post-render hook, OR (b) have the BeforeEach create the SA in each per-test namespace, OR (c) make the SA cluster-scoped... no, ServiceAccounts are namespace-scoped — (a) or (b) only.
**Recommendation:** Add to test BeforeSuite a helper `ensureSubagentSA(namespace)` that creates the SA in any namespace before applying Tasks. Trivial — empty SA, no RBAC needed (D-A4: zero verbs).

### Pitfall 7: Plan/Phase/Milestone referential integrity is incomplete in fixtures
**What goes wrong:** Some test fixtures (e.g., `failure_test.go`) create only a `Plan` with `phaseRef: fail-phase` and Tasks, but no actual Phase/Milestone/Project. PlanReconciler's logic may need a resolvable parent chain to materialize Waves.
**Why it happens:** The Layer B specs were authored before verifying full hierarchy was needed.
**How to avoid:** Either (a) extend each fixture to include the full Project→Milestone→Phase→Plan chain, OR (b) verify that PlanReconciler handles the no-parent case gracefully (no-op + condition message), then make the test fixtures match.
**Status:** The `wave_test.go` fixture (`three-task-wave.yaml`) DOES include the full chain. `failure_test.go`, `caps_test.go`, `output_test.go`, `credproxy_test.go` do NOT — they only inline a `Plan` + `Task`s. Need to verify whether reconciliation succeeds with the truncated hierarchy. [ASSUMED — must verify by reading PlanReconciler.resolveProject behavior]

### Pitfall 8: credproxy startup log line not emitted in plain text
**What goes wrong:** `credproxy_test.go` greps for the literal string `credproxy listening on 127.0.0.1:8443` in container logs. The actual code calls `log.Info("starting credential proxy", "listenAddr", listenAddr, ...)` which emits JSON like `{"level":"info","msg":"starting credential proxy","listenAddr":"127.0.0.1:8443",...}`.
**Why it happens:** Plan 13 author wrote the test against a specification, not against the running code.
**How to avoid:** Pick one:
  - **(a) Emit a one-line boot banner** in `cmd/credproxy/main.go` directly to stdout: `fmt.Fprintf(os.Stdout, "credproxy listening on %s\n", listenAddr)` immediately before `srv.ListenAndServeTLS`. Two-line change. Test passes unchanged.
  - **(b) Update the test grep** to match the structured zap shape: `strings.Contains(logs, `"msg":"starting credential proxy"`)`. Slightly more brittle to log format changes but no production code change.
**Recommendation:** (a) — boot banners are a common pattern, low cost, no flakiness.

### Pitfall 9: 90s `waitForControllerReady` ceiling too tight for cold-start kind + image-load + cert-manager
**What goes wrong:** First-time `make test-int` on a slow CI runner: docker build (60s) + kind create (45s) + cert-manager install + ready wait (60s) + manager image pull (0s — already loaded) + manager Pod startup (10s) + readiness probe wait (15s after `initialDelaySeconds=5`) → ~90s passes between BeforeSuite start and the `waitForControllerReady` call, leaving only 0s of margin.
**Why it happens:** Each component's wait is bounded; together they bust the budget.
**How to avoid:** Move the long waits into `make test-int-kind-prep` (already does cluster + image load); use `helm install --wait --timeout 120s` for the controller readiness (replaces the bespoke poll); keep the 90s ceiling in `waitForControllerReady` as a fallback but expect it to no-op because helm already waited.

### Helm-install BeforeSuite (Pitfall 1 fix)

```go
// test/integration/kind/suite_test.go — replace applyController()

// applyController installs the TIDE controller via helm.
// Helm is the canonical install path (BOOT-04) AND it provides the
// chart-only resources (signing-key Secret, projects PVC, subagent SA,
// Deployment args) that kustomize omits. helm --wait blocks until every
// chart resource is Ready, replacing the bespoke waitForControllerReady poll.
func applyController() {
    if _, err := exec.LookPath("helm"); err != nil {
        GinkgoWriter.Println("helm not in PATH; falling back to CRDs-only mode")
        return
    }
    chartDir := filepath.Join("..", "..", "..", "charts", "tide")
    if _, err := os.Stat(chartDir); os.IsNotExist(err) {
        GinkgoWriter.Println("charts/tide not present; falling back to CRDs-only mode")
        return
    }
    helmCmd := exec.CommandContext(ctx, "helm",
        "install", "tide", chartDir,
        "--create-namespace", "-n", kindControllerNamespace,
        "--kubeconfig", kubeconfigPath,
        "--set", "controllerManager.manager.image.repository=controller",
        "--set", "controllerManager.manager.image.tag=test",
        "--set", "controllerManager.manager.image.pullPolicy=IfNotPresent",
        "--set", "images.stubSubagent.repository=ghcr.io/jsquirrelz/tide-stub-subagent",
        "--set", "images.stubSubagent.tag=test",
        "--set", "images.stubSubagent.pullPolicy=IfNotPresent",
        "--set", "images.credProxy.repository=ghcr.io/jsquirrelz/tide-credproxy",
        "--set", "images.credProxy.tag=test",
        "--set", "images.credProxy.pullPolicy=IfNotPresent",
        "--wait", "--timeout", "120s",
    )
    out, err := helmCmd.CombinedOutput()
    if err != nil {
        GinkgoWriter.Printf("helm install failed: %v\n%s\n", err, out)
        if os.Getenv("TIDE_REQUIRE_CONTROLLER") == "1" {
            Fail(fmt.Sprintf("helm install failed (TIDE_REQUIRE_CONTROLLER=1): %v", err))
        }
        return
    }
    GinkgoWriter.Printf("helm install completed\n%s\n", out)
}
```

### Subagent SA in per-test namespace (Pitfall 6 fix)

```go
// test/integration/kind/suite_test.go — new helper

// ensureSubagentSA creates the tide-subagent ServiceAccount in the given namespace.
// Every Task Job references this SA by name (jobspec.go:43); without it Pod
// creation fails with `serviceaccount "tide-subagent" not found`.
// The SA has zero RBAC (D-A4: subagent pods have no K8s API verbs).
func ensureSubagentSA(ns string) {
    saYAML := fmt.Sprintf(`
apiVersion: v1
kind: ServiceAccount
metadata:
  name: tide-subagent
  namespace: %s
automountServiceAccountToken: true
`, ns)
    _ = applyYAML(saYAML)
}
```

Each test's `BeforeEach` (or each test's `createNamespace`) should call `ensureSubagentSA(ns)` after creating the namespace.

Closes the phase. |

Alternative plan shapes considered and rejected:
- **One plan per Ginkgo spec (7 plans):** rejected because 5 of 7 specs share the install-path root cause; fixing them separately would mean 5 plans that each can't be verified without first applying the install-path fix to suite_test.go. Wrong granularity.
- **One mega-plan (1 plan):** rejected because the credproxy code fix and the suite_test.go install-path fix touch unrelated files and have independent verification surfaces; bundling them defeats wave parallelism.
- **Per-failure-type grouping (deployment-ready, webhook-endpoints, specs-pass):** rejected because "deployment-ready" and "webhook-endpoints" collapse into the same Helm-install fix; there's no parallel work to do.

Wave structure for the 4-plan shape:
- **Wave 1:** 02.1-01 (baseline) — alone, gates everything
- **Wave 2:** 02.1-02 (install path) + 02.1-03 (credproxy banner) — independent, parallelizable
- **Wave 3:** 02.1-04 (verify + close) — depends on Wave 2

The Wave 2 parallelism is real because the install-path fix touches `test/integration/kind/suite_test.go` only and the credproxy fix touches `cmd/credproxy/main.go` only — no file overlap, no shared compilation unit.
