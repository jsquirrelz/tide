## Issues Encountered

### 1. Clean run hit 1187s wall-time (3.96× the 300s budget)

- **Observed:** `make test-int` wall-time was 1187s on a cold-cache dev laptop. The Makefile's `timeout 300s` only bounds the `go test ./test/integration/...` invocation (≈300s consumed there); the dependency chain `manifests → generate → fmt → vet → setup-envtest → test-int-kind-prep` consumed ~880s of additional wall-time, the bulk in docker image builds + `kind create cluster` + `kind load docker-image` (the latter 3 times).
- **Impact:** The phase's "≤ 300s wall-time per run" goal as literally written cannot be achieved on a cold-cache dev laptop regardless of any source-code fix to the suite. The goal needs re-scoping (probably to "go test wall-time only" — but that's a planning-layer decision).
- **Documented in:** VERIFICATION.md Section 4 + Section 5 Root-Cause Summary item 4.

### 2. Helm install killed at 3m7s by suite's outer 4-min context

- **Observed:** Log line 957 of clean run: `helm install failed after 3m7.733056248s: signal: killed`. The killer is `suite_test.go:88`'s `context.WithTimeout(context.Background(), kindTestTimeout)` where `kindTestTimeout = 4 * time.Minute` at line 59.
- **Root cause:** The parent context's 4-min budget cannot accommodate cert-manager-install (~43s) + helm-install-with-wait (which itself needs ~3-4 min on cold-cache to install + readiness-probe the controller-manager Deployment). The `helm --timeout 5m` flag in `suite_test.go:336` is dead code — the parent context's `signal: killed` fires first.
- **Documented in:** VERIFICATION.md Section 1 + Section 5 Root-Cause Summary items 1-2.

### 3. Rerun failed in `test-int-kind-prep` on a zombie kind control-plane container

- **Observed:** Clean run's AfterSuite tried `kind delete cluster --name tide-test` and got `failed to delete nodes: command "docker rm -f -v tide-test-control-plane" failed with error: exit status 1` ... `cannot remove container "tide-test-control-plane": could not kill container: tried to kill container, but did not receive an exit event`. The kind cluster ended up half-deleted: `kind get clusters` still listed `tide-test`, but the Docker container had stopped responding. Rerun's `kind load docker-image` then failed to exec into the dead control-plane.
- **Root cause:** AfterSuite's cluster-delete logic doesn't handle the half-installed-cluster case (e.g., after a half-failed helm install + a controller-runtime cert-watcher cleanup error chain). When the docker rm fails, AfterSuite ignores it and reports the cluster "deleted" — but the next test-int run gets a broken cluster.
- **Documented in:** VERIFICATION.md Section 2 + Section 5 Root-Cause Summary item 3.

### 4. Pre-existing STATE.md inconsistencies surfaced (out of scope to fix)

- **Observed (during plan read-first step):** STATE.md frontmatter `completed_phases: 2` vs ROADMAP.md top-level Phases checklist showing Phases 1+2 still `[ ]` unchecked. STATE.md frontmatter `percent: 86` vs body Progress bar `[████████░░] 83%`.
- **Impact:** Plan 04 Task 3 was supposed to add a `Known inconsistency` entry to STATE.md documenting these. Because Task 3 did not run, that documentation was NOT added in this plan. Carrying forward to the debug plan / next planner.
- **Documented in:** Task 3 design (Plan 04 PLAN.md) anticipated this; the actual values seen during this run were `completed_phases: 2`, `total_plans: 28`, `percent: 86`, body `[████████░░] 83%`.
