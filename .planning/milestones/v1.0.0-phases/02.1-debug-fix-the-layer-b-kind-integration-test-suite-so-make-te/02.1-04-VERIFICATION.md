### Root-Cause Summary (for the debug plan)

1. **`kindTestTimeout = 4 * time.Minute` (suite_test.go:59) is too tight** — cert-manager install (~43s) + helm-install-with-wait + readiness-probe wait + tide-controller-manager startup all need to fit inside this context, which is impossible on a cold-cache dev laptop where helm install alone needs ~190s+.
2. **`helm --timeout 5m` (suite_test.go:336) is dead code** — it exceeds the parent context's 4-min budget, so the parent context kills the helm subprocess before helm's own timeout can fire. Either the parent context needs to grow (≥6m) or the helm timeout needs to shrink (≤3m, but that's risky on slow runners).
3. **`AfterSuite` cluster delete is fragile** — when BeforeSuite leaves the cluster in a half-installed state (helm release "failed" but partial resources rendered), AfterSuite's `kind delete cluster` can leave a zombie Docker container that breaks the next `kind load` call. This is the idempotency failure mode.
4. **The Makefile's `timeout 300s` does not bound the full `make test-int` invocation** — it only bounds the `go test` step. `test-int-kind-prep` (image builds + kind cluster create + image loads) is unbounded and consumed 880s in the clean run. The phase's "300s wall-time on `make test-int`" goal as written cannot be achieved on a cold-cache dev laptop regardless of any source-code fix to the suite.

The fix landscape (architectural, requires plan re-design — Rule 4 territory):
- Option A: increase `kindTestTimeout` to ≥6min AND keep `helm --timeout 5m` (gives helm enough room to actually complete)
- Option B: split BeforeSuite into a "prep cluster + cert-manager" phase under one context and a "helm install + wait" phase under a separate, longer context
- Option C: bump cert-manager from v1.16.2 to v1.20 (RESEARCH.md flagged this as a no-cost improvement; may shave 10-20s off the BeforeSuite budget)
- Option D: hard-fix AfterSuite to handle the half-installed-cluster zombie-container case so idempotency is genuine
- Option E: re-scope Phase 02.1's "300s wall-time" goal to mean "go test wall-time only" (300s for the test invocation; test-int-kind-prep is separate) and document this in the success criteria

These decisions belong to a planner re-evaluation, not to Plan 04's executor. The debug routing should surface them.
