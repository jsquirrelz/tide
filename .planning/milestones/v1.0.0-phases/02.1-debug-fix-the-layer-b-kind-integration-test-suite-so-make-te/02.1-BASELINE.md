## Root-Cause Confirmation

The RESEARCH.md hypothesis was: *"controller Pod CrashLoopBackOff with `TIDE_SIGNING_KEY env var is required (HARN-03)` because the kustomize overlay omits chart-only resources (signing-key Secret, projects PVC, subagent SA) and the chart-only Deployment args (`--subagent-image`, `--credproxy-image`)."*

### Refinement of the hypothesis (carry to Plans 02 + 03)

The hypothesis is **CONFIRMED in its structural form** (kustomize-vs-helm install-drift is the real cause) but **REFUTED in its precise CrashLoop prediction** (because the inner timeout fires before the controller Pod is ever created). The practical implication:

1. The fix in Plan 02 (switch BeforeSuite to `helm install`) is still the right fix — for the structural drift reason RESEARCH.md identified.
2. The fix also collaterally addresses an additional problem the baseline revealed: the **300s inner timeout is too tight** for the current sequence (Layer A 40s + kind create cluster + kind load images already done at line 220 of the log + cert-manager install + cert-manager readiness wait + applyController + waitForControllerReady). Either the timeout needs to grow (RESEARCH.md Pitfall 9 anticipated this — though estimated ≥90s slack; reality is much tighter) OR the wait sequence needs to be pruned (`helm install --wait` collapses several discrete waits into one, which is itself a budget improvement).
3. Plan 04 (CI verification) must verify that `make test-int` completes WITHIN the inner timeout — not merely "exits 0." A green run that completes in 295s is not the right finish line; 295s is one CI machine away from a 305s flake. Build slack into the timeout.

**No planner re-evaluation is required.** The structural diagnosis is correct; the precise crashloop prediction was wrong about timing of when the failure surfaces, but the cure (use Helm) is the same.

### D-01 — Credproxy boot-banner approach

**Decision:** **Approach (a)** — emit a single-line plaintext banner to stdout via `fmt.Fprintf(os.Stdout, "credproxy listening on %s\n", listenAddr)` in `cmd/credproxy/main.go`, **keeping** the existing structured zap `log.Info("starting credential proxy", "listenAddr", listenAddr, ...)` line alongside it.

**Rationale:**
- Minimal code change (literally two lines: one `fmt.Fprintf` and one comment explaining the exception).
- The existing test grep at `test/integration/kind/credproxy_test.go:199,208` (`strings.Contains(logs, "credproxy listening on 127.0.0.1:8443")`) passes unchanged — no test edit required.
- Boot banners are a common, intentional convention in K8s operators. The structured zap log stays for log-aggregator consumption; the banner is the operator-readable "yes, the listener bound" signal that survives log-formatter refactors.
- Per RESEARCH.md Pitfall 8 recommendation + PATTERNS.md `cmd/credproxy/main.go` entry.

**Rejected: (b)** update the test grep to match zap's JSON shape (`strings.Contains(logs, `"msg":"starting credential proxy"`)`) — more brittle to log-format refactors (any future move off zap or any change to its config breaks the assertion); requires test changes that don't align with the spec's eventual operator-readable boot signal.

**Implemented by:** **Plan 03 (`02.1-03-PLAN.md`) — verbatim**, no further design discretion.

### D-02 — Per-test subagent-SA strategy

**Decision:** **Approach (a)** — add an `ensureSubagentSA(ns string)` helper in `test/integration/kind/suite_test.go` (analog: the existing `applyYAML` helper at lines 385–394), called from `createNamespace` in `test/integration/kind/failure_test.go:158-166` and from any future fixture that creates a per-test namespace.

**Rationale:**
- The chart's `charts/tide/templates/serviceaccount-subagent.yaml` only templates the `tide-subagent` SA in `.Release.Namespace` (i.e., `tide-system`). Tests apply Tasks into per-test namespaces (`tide-int-test`, `failure-test`, `caps-test`, `output-test`, `credproxy-test`).
- Without the SA in each per-test namespace, Pod creation fails with `serviceaccount "tide-subagent" not found` — verified by reading `internal/dispatch/podjob/jobspec.go:43-44` where `ServiceAccountSubagent = "tide-subagent"` is unconditionally set as the PodSpec's `ServiceAccountName`.
- The helper YAML is identical to the chart template minus the `tide.labels` block and the `{{ .Release.Namespace }}` substitution — D-A4 (zero K8s verbs on subagent SA) is preserved exactly.

**Rejected: (b)** cluster-scoped SA — ServiceAccounts are namespace-scoped by K8s API design; this is not implementable.
**Rejected: (c)** chart post-render hook to template the SA in arbitrary namespaces — out of scope for this phase; tests own their per-namespace setup.
**Rejected: (d)** use a single shared namespace for all tests — breaks test isolation, would cause cross-test resource leaks and flakiness.

**Implemented by:** **Plan 02 (`02.1-02-PLAN.md`)**. Plan 02 adds the helper alongside the install-path switch (both touch the same file, `suite_test.go`).

### D-03 — `TIDE_REQUIRE_CONTROLLER` default for Phase 02.1

**Decision:** **Keep the existing default of `unset`** (silent skip when controller not ready). The WR-09 contract established in Phase 2 Plan 13 (`test/integration/kind/suite_test.go:351-354`) stays intact: hard `Fail` only when `TIDE_REQUIRE_CONTROLLER=1`; soft-skip with `GinkgoWriter` warning by default.

**Operational rule:** Phase 02.1 CI runs MUST set `TIDE_REQUIRE_CONTROLLER=1` in the CI workflow's env block to ensure install failures fail the CI build loudly. Local dev iterations MAY leave it unset for partial-coverage debugging (e.g., when iterating on a non-install code path that doesn't need a healthy controller).

**Rationale:**
- The default-unset behavior was a deliberate choice in Phase 2 to keep dev-machine runs that only want CRD-level coverage from failing on environments without Docker.
- Inverting the default to `1` would break dev-machine `make test-int-fast` runs that intentionally skip Layer B.
- Wiring the gate in CI (rather than the code default) is a one-line env-var addition in the workflow file — Plan 04 territory. No code change.

**Rejected:** invert the default to `1` to make all runs strict — breaks the Phase 2 D-H3 contract that Layer B is a CI-strict, dev-soft gate.

**Implemented by:**
- **Plan 02** keeps the existing soft-skip logic intact — do not invert the default.
- **Plan 04 (`02.1-04-PLAN.md`)** wires `TIDE_REQUIRE_CONTROLLER=1` in the CI workflow `env:` block (file: `.github/workflows/test-int.yml` or equivalent).
