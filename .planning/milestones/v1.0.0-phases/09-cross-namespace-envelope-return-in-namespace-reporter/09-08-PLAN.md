<objective>
Close the two defects 09-07 live acceptance surfaced (evidence + design:
.planning/debug/09-07-premature-succession-evidence.md):

Defect B — premature-succession race. Option-C async materialization (plan 09-06)
broke the debug #9 guard: the Manager spawns a reporter Job and returns; the
reporter creates children asynchronously, so "no children visible yet" is
ambiguous (genuine leaf vs reporter-not-done). The four level controllers handle
this inconsistently (milestone: one-shot justMaterialized; phase: NO guard —
fell straight through to Succeeded while its child Plans were still Running;
plan: reporterSpawned early-return; project: separate). Result observed live:
Project=Complete while grandchild Plans=Running — an illegitimate Complete.

Defect C — planner-level cost rollup dropped. cmd/manager wires the correct
PodStatusEnvelopeReader (tiny status off the dispatch Job termination message),
but the four planner controllers call `if _, err := r.EnvReader.ReadOut(...)`
and DISCARD the EnvelopeOut, using it only for error classification. So the
planner's Usage (incl. defect #6's EstimatedCostCents) is never tallied;
Project.Status.Budget stays empty for planner spend. (Task executors already
roll up via budget.RollUpUsage in task_controller.go.)

Fix both with one data addition (ChildCount on the tiny status) + a uniform
succession gate + planner usage rollup. The reporter MECHANISM itself is correct
and unchanged.
</objective>
