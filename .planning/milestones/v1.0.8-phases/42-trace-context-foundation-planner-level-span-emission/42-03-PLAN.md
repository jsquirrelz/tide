<objective>
Add one durable span-emission idempotency marker field per planner level to the v1alpha3 CRD status types, and regenerate manifests. This is the storage half of 42-RESEARCH.md Architecture Pattern 2: span creation has no natural idempotency, the existing `*RolledUpUID` markers are envReadOK-gated by design (reusing them would re-emit degraded spans forever — Pitfall 2), and `resume --retry-failed` reuses the same terminal Job object (so one-span-per-Job-UID, durable-marker-gated, is the correct and sufficient contract today — research Open Question 3 acknowledged, no multi-attempt differentiation built).

Placement decision (research Open Question 2, resolved here): all four markers live DIRECTLY on the level's Status struct — including Project's. `Project.Status.Budget.PlannerRolledUpUID` is Budget-nested because budget rollup is cost bookkeeping; span emission is telemetry bookkeeping with no budget relationship, so `PlannerSpanEmittedUID` sits directly on `ProjectStatus`, consistent with the other three levels.

Explicitly NOT here: the `.status.trace` field (PROP-02) is Phase 43 — do not add it.

Purpose: Unblocks plans 42-04/42-05 (the wiring that gates on these markers). ATTR-01/02 spans depend on this exactly-once mechanism.
Output: Four additive optional scalar fields + regenerated deepcopy/CRD manifests.
</objective>
