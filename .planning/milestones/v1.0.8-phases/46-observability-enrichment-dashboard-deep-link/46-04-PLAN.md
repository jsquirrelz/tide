RESEARCH.md's "Task-level-only" recommendation (its Summary, Pitfall 2, and the anti-pattern bullet warning against a five-level drop) rests on a premise the planner disproved by direct code read:**

RESEARCH.md claims "only Task spawns the Phase-44 trace-only reporter that emits per-call LLM spans." The trace-only Job SHAPE is indeed Task-only — but the combined/materialization reporter Jobs spawned at all four planner-level completions ALSO synthesize LLM message spans. Verified evidence chain (all read this session):

1. `cmd/tide-reporter/main.go` line ~229: `synthesizeSpans(ctx, cfg, stderr)` is called on the combined-mode path too; its own doc comment says "Called from both the trace-only branch and the combined-mode path ... so a failed planner Job with no/partial out.json still gets its conversation spans (D-05)" — planner conversations emitting message spans is deliberate Phase 44 behavior.
2. `internal/subagent/anthropic/subagent.go` lines ~308-318: every dispatch (planner and executor alike) writes `events.jsonl` at `<WorkspaceRoot>/envelopes/<UID>/events.jsonl` — the synth input exists at all five levels.
3. `internal/controller/reporter_jobspec.go` lines ~244-250: the combined shape gets `OTEL_EXPORTER_OTLP_ENDPOINT` env whenever the manager has one, and `--workspace=/workspace --task-uid=<parentUID>` args — so planner-level LLM spans export whenever AGENT spans do.
4. Planner spawn sites pass `--traceparent` anchored at the level's own AGENT span (e.g. `milestone_controller.go:641`), making the LLM spans children of the AGENT span in the same trace.

Phoenix creates a SpanCost row per span carrying `llm.token_count.*` with no span-kind gate (RESEARCH's verified A2 — that finding stands). Therefore keeping rolled-up TokenCount on planner AGENT spans double-counts every level, violating CONTEXT.md D-03's LOCKED invariant ("Phoenix's session/trace cost views must not double-count a run"), whose locked decision rule is "per-call LLM counts are authoritative ... AGENT-level totals are adjusted/dropped." RESEARCH's fear that the drop "would silently zero out Phoenix's ONLY token/cost signal at those levels" is unfounded: the per-call spans carry the signal at every level, and in the self-instrumenting-runtime future (Phase 45) the runtime's own LLM spans do. Degradation case (events.jsonl reconstructs zero calls → that level shows no cost in Phoenix) is honest absence, already marked by the existing ParseDegraded/EnvelopeDegraded machinery — the locked invariant explicitly prefers absence over "a wrong number that looks authoritative."

Do not "correct" this back to Task-only when reading RESEARCH.md — the executor should cite this block in the code comment at the removal site.
