**Explicit plan decision — RESEARCH Pitfall 1 / Assumption A2, resolved here, not implicitly:**
Task's `handleJobCompletion` has FOUR terminal paths (EnvelopeReadFailed at ~938-957; OutputValidationError at ~962-998; OutputPathsViolation at ~999-1036; standard interpretation at ~1039-1121) versus the planner levels' one. We adopt **Option B, generalized**: TWO span-emission call sites positioned so every terminal path is covered —
1. inside the `EnvelopeReadFailed` branch with `envReadOK=false` (the only path where D-07's degraded-envelope span is reachable at Task level), and
2. immediately AFTER a successful envelope read but BEFORE the OutputValidationError/OutputPathsViolation/standard branching, with `envReadOK=true`.

Rationale: RESEARCH recommends Option B because TRACE-01 exists to close the "zero span" gap — and its rationale extends to branches 2/3: a Job that ran and consumed tokens but failed output validation would otherwise lose its cost data in Phoenix. Placing call site 2 before the branch divergence covers all three post-read paths with one line. The unit of instrumentation is the Job ATTEMPT (D-07 "one span per Job attempt"), so span status reflects Job success/failure via the existing `isJobFailed`, not TIDE's validation verdict.

Both call sites run BEFORE their branch's terminal status patch — if the terminal patch fails and the reconcile retries, the marker prevents a duplicate; if it succeeds and the handler never re-runs, the span already exists (span-loss-averse ordering).
