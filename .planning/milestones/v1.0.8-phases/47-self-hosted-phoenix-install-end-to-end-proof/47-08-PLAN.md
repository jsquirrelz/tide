<objective>
Close verification gap #3 (pre-existing run-integrity defect surfaced by the proof, 47-EVIDENCE §6.1 Defect #1): `Status.Git.LastPushedSHA` is set at the project-descent boundary push and later re-asserted verbatim on every `--force-with-lease` retry. When a wave-level push has since advanced the same run branch, every retry fails identically (non-fast-forward), flapping `medium-project.status.phase` forever — the controller keeps handing the tide-push Job a stale lease anchor.

Root fix at the binary, not the controller: before pushing, `cmd/tide-push` re-reads the ACTUAL remote tip of the branch and derives the effective lease with an ancestry guard — refresh the lease to the observed tip when the remote advance is already integrated in local history (TIDE's own wave-level push from the same PVC repo), fail closed with the existing exit-11/lease-rejected contract when the advance is external. Because the re-read executes inside every push attempt, both the controller's bounded retries AND the bypass-annotation re-dispatch path (which "clears state without refreshing the lease") are covered by one mechanism — no `project_controller.go` change is needed, and none may be made (that file is owned by plans 47-06/47-07 in this wave set).

Output: a generic `pkg/git` remote-tip primitive, the effective-lease derivation in tide-push, and regression tests proving both the refresh and the fail-closed arm.
</objective>
