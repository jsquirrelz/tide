# Architecture Patterns — v1.0.10 "King Tide" Integration

**Domain:** Kubernetes-native hierarchical agentic orchestrator — integrating a write-capable LangGraph authoring runtime, per-rung runtime selection, and three new loops (Product/System/Oversight) onto TIDE's existing envelope/dispatch seam.
**Researched:** 2026-07-21
**Confidence:** HIGH on all "today's state" claims (verified by file:line grep against `main`); MEDIUM on integration-point recommendations that extend a design note (`sounding-dynamic-orchestration-design.md`) not yet routed through GSD — flagged individually below.

**Supersedes** the prior `v1.0.9`-scoped `ARCHITECTURE.md` (verify-tier integration only). That content is now shipped, load-bearing code (Phase 48-53) and is treated here as "today's state," not a research target.

## Recommended Architecture

Nothing about the dispatch spine changes. Every new capability rides the same seam the read-only LangGraph verifier already proved (Phase 48-53): `EnvelopeIn`/`EnvelopeOut` JSON on the per-Project PVC, `pkg/dispatch.Subagent`, one K8s Job per dispatch, `podjob.BuildJobSpec` with a `Kind` discriminator (`JobKindPlanner`/`JobKindExecutor`/`JobKindVerifier`), and `ResolveLoopPolicy`/`ResolveProvider` as the config-resolution chokepoints. The six questions in scope are answered by extending three things, never by adding a fourth control plane:

1. **`ProviderSpec.Vendor`** — today hardcoded to `"anthropic"` in `ResolveProvider` (`internal/controller/dispatch_helpers.go:379-383`) and hardcoded to `"langgraph"` inline at three verifier dispatch call sites (`level_verify.go:410`, `plan_controller.go:1271`, `task_controller.go:2401`). This is the gap every "which runtime" question in this doc closes through.
2. **`podjob.BuildOptions.ReadOnly`** — today a boolean that flips a verifier Job's `/workspace` mount, `ReadOnlyRootFilesystem`, and git-credential omission (`internal/dispatch/podjob/jobspec.go:188-204`). A write-capable authoring image is the `ReadOnly: false` path through the SAME `JobKindPlanner`/`JobKindExecutor` build, not a new Job kind.
3. **`ResolveLoopPolicy`** (`internal/controller/dispatch_helpers.go:551`) — today resolves `MaxIterations`/`EscalationPolicy`/`EvaluatorRef`/`Level` from one precedence chain. `LoopPolicy.Autonomy` (`api/v1alpha3/loop_types.go:60`) is declared but never read or written by this function — confirmed by reading the full body: it stamps every other field, never `Autonomy`. This is the Oversight loop's entire integration point.

No new scheduling engine, no runtime DAG mutation, no wave list as input — every fan-out pattern below is Jobs *within* one dispatch (siblings sharing a `loopRunID`/parent CRD), never new DAG edges. This is the same discipline the (unrouted) Sounding design note calls "the wave-internal sub-scheduler behind Kahn," and it is the only shape consistent with CLAUDE.md's locked anti-patterns.

## Data Flow / Integration Points

### (a) Write-capable LangGraph authoring image

**What's identical to the read-only verifier:** envelope contract (`EnvelopeIn`/`EnvelopeOut`/`TerminationStub`, `pkg/dispatch/envelope.go`), the Python-side field-for-field port (`cmd/tide-langgraph-verifier/verifier/envelope.py`), `create_agent` + LangGraph tool-loop wiring, credproxy-gated LLM calls, `SelfInstruments`-keyed span emission.

**What changes, concretely:**

- **`ReadOnly: false`.** `BuildJobSpec` already branches on this flag (`internal/dispatch/podjob/jobspec.go:188-204, 481-507, 558-577`): read-only mounts `/workspace` RO + adds a scratch `emptyDir` + flips `ReadOnlyRootFilesystem`; the write path is simply not taking those branches — the SAME `JobKindPlanner`/`JobKindExecutor` code paths the CLI subagent already uses. No new Job kind.
- **Git write is a two-tier model already built for exactly this.** The executor's Claude CLI image never holds push credentials — `internal/harness/commit.go` commits locally to the isolated worktree using `resolveAgentIdentity`'s resolved name/email (`internal/controller/dispatch_helpers.go:706-740`), and a *separate* `tide-push` Job (which DOES hold the real git remote secret) pushes at level boundaries. A write-capable LangGraph image needs local `git commit` inside its container (same worktree-mount + agent-identity contract) and **nothing new for remote push** — the existing `tide-push` seam is vendor-agnostic (it consumes a HeadSHA off `EnvelopeOut.Git`, `envelope.go:355-366`, not a vendor-specific artifact).
- **Child-CRD emission: structured output replaces sanitize-and-parse.** Today's Claude CLI planner emits raw text the reporter must JSON-decode defensively (the Phase 10 cascade class the ladder note explicitly targets). LangGraph's `with_structured_output`/`ToolStrategy` pattern is already proven live for the verifier's `GateDecision` (`cmd/tide-langgraph-verifier/verifier/agent.py:37-55`: `response_format=ToolStrategy(GateDecision)`, and the fail-closed discipline that an unset `structured_response` never silently defaults to a passing verdict). The SAME mechanism, targeted at `pkg/dispatch.ChildCRDSpec` (`pkg/dispatch/childcrd.go:44`) as the Pydantic schema, makes the model emit a **validated** `ChildCRDSpec` list directly instead of free text — eliminating the parse-failure class structurally, not by better prompting. Downstream is **unchanged**: `EnvelopeOut.ChildCRDs` still carries the same wire shape, `MaterializeChildCRDs`'s Kind allowlist (`internal/reporter/materialize.go:198`) still gates server-side creation — only the emission-side risk is closed.
- **Commit protocol for planner dispatches:** planner Jobs don't currently commit at all (they only author child CRDs materialized server-side by the reporter) — a write-capable authoring image only needs the git-write path for the **executor** rung (which produces file diffs). Planner rungs (4 templates) migrate first per the ladder and need *no* git-write change at all — purely a structured-output + `Vendor` config change. This is corroborating evidence for the ladder's own sequencing (planner-first is not just "safer," it's architecturally cheaper).

### (b) Per-rung runtime selection + shadow/champion-challenger

**Where "planner level X uses runtime Y" lives today:** nowhere configurable. `ResolveProvider` (`internal/controller/dispatch_helpers.go:337`) pins `Vendor: "anthropic"` unconditionally for every planner/executor dispatch; the three verifier dispatch sites (`level_verify.go:410`, `plan_controller.go:1271`, `task_controller.go:2401`) each construct `pkgdispatch.ProviderSpec{Vendor: "langgraph", ...}` **inline**, bypassing `ResolveProvider` entirely. This is a real, verified gap, not a naming nuance: three call sites already hand-roll the exact override this milestone needs generalized.

**The integration point:** extend `LevelConfig` (the same struct `Model`/`Params`/`Image` already live on, resolved via `Levels.<overrideKey>` — `dispatch_helpers.go:341-353`) with a `Vendor` field, and thread it through `ResolveProvider`'s precedence chain the same way `resolveImage` already does (`dispatch_helpers.go:681-704`: `Levels.<key>.Image → Spec.Subagent.Image → helmDefaults.Image → ""`). **Vendor and Image must resolve together, not independently** — an `anthropic` vendor pointed at a LangGraph image (or vice versa) is a config error the subagent's own vendor sentinel check should reject (the doc comment on `EnvelopeIn.Provider` already promises this fail-fast behavior: "the provider implementation... rejects an envelope whose Provider.Vendor does not match its compiled-in sentinel"). This makes "planner level X uses runtime Y" a per-level `Levels.<level>.{Image, Vendor}` pair, evidence-gated per rung exactly as the ladder note specifies.

**Shadow/champion-challenger dispatch** is the SAME mechanism as the generate-and-filter pattern in (f): dispatch the identical envelope inputs through both the CLI/anthropic image and the candidate LangGraph image as sibling Jobs (N=2, one per runtime), let both complete, and materialize `ChildCRDs`/commits from only ONE (the champion) while feeding **both** results into `internal/eval`'s candidate-versioned evidence store (System loop). This is not a separate build — building the fan-out+reduce primitive once (see (f)) and parameterizing its "candidates" as {runtime A, runtime B} rather than {N prompt variants} is the whole of shadow-mode. Recommend building shadow-mode as the FIRST consumer of the shared fan-out primitive, since it has the simplest reduce step (no scoring/tournament logic needed — champion is whichever image the rung's current evidence-gate designates, loser is discarded, never materialized).

### (c) Product loop — outcome-level judgment

**Hook point:** the project-level verify seam ALREADY EXISTS and is already wired into `ResolveLoopPolicy`'s phase/milestone/project group (`dispatch_helpers.go:567-571`: `case "phase", "milestone", "project": maxIter = 0` — "no repair branch at these levels," D-07, deliberate). `resolveAuthoredVerification`'s switch already includes `level == "project"` via `projectLevelVerificationDefault` (`dispatch_helpers.go:394-412`). The Product loop's "judge the artifact tree against the outcome prompt, not just per-task specs" is this SAME seam, parameterized with an outcome-focused evaluator prompt instead of a mechanical gate-command — **not a new dispatch seam**.

**How its verdict drives re-planning without violating approve-at-descent:** the D-07 clamp (`MaxIterations = 0` at phase/milestone/project, unconditionally, "even when an authored contract OR the chart sets it higher" — `dispatch_helpers.go:520-526`) is locked and must NOT be reopened; post-execution rework at these levels discards paid work by design. The Product loop does not need to reopen it: "re-planning" in response to a Product-loop verdict is **authoring a new `Milestone` (or `Project`) child CRD** — exactly the `ChildCRDSpec`/`MaterializeChildCRDs` mechanism every other planner already uses. A new Milestone is a new top-level planning-DAG entry, not a mutation of the already-executing DAG, so this respects "no runtime DAG mutation" by construction. Its dispatch is gated by the SAME `checkParentApproval`/`gates.EvaluatePolicy` primitive every other Milestone already goes through (default `milestone: approve` — `internal/gates/policy.go:53-61`) — approve-at-descent is not something the Product loop needs to route around; it's the mechanism that already prevents an autonomous Product loop from spending unchecked on its own re-plan proposal.

**Structural note:** per `five-loop-model.md:67-68`, `Product`/`ProductLoop` is explicitly a **new CRD sitting above finite Projects** ("operates through Projects, never modifies Tasks directly... a current Project is an outcome-bound campaign, not a permanently-running backlog daemon"). This is a new peer/parent of `Project`, not a sixth level inserted into the locked Milestone→Phase→Plan→Task→Wave hierarchy — CLAUDE.md's "don't invent new levels" constraint is about that five-level chain, and Product sits outside it exactly as Oversight does above Product.

### (d) System loop — `internal/eval` → CI/controller wiring

**Current state (verified):** `internal/eval` is deliberately import-firewalled — its doc comment (`internal/eval/doc.go:45-52`) forbids importing `internal/controller`, `internal/budget`, `internal/metrics`, or `api/v1alpha3`, permitting only `internal/subagent/common` and `pkg/dispatch`. Two tiers exist today: (1) offline byte-ratchet tests (`render_test.go`, under plain `make test`, zero network) and (2) `cmd/tide-eval` / `make eval` (live Anthropic `count_tokens` call, needs `TIDE_PROXY_ENDPOINT`/`TIDE_SIGNED_TOKEN`, deliberately excluded from default CI because it costs money and needs credentials).

**Integration must be one-directional** to preserve the firewall: a new controller-side `SystemExperiment`/`SystemCandidate` reconciler (living in `api/v1alpha3` + `internal/controller`, per the five-loop-model's shape) **consumes `internal/eval` as a library**; `internal/eval` never gains a reverse dependency on the CRD types. This is the same "don't let the leaf grow upward" discipline the eval package's own doc comment already enforces.

**Two build-order-relevant stages, not one:**
1. **CI-gate stage (cheap, unblocks the authoring ladder immediately):** extend `cmd/tide-eval`/`make eval` from "measure one template's tokens" to "compare a candidate rung's output against the CLI baseline" and emit a pass/fail against a promotion threshold — this is directly what "each rung gated on eval evidence vs the CLI baseline" (PROJECT.md) requires, and it can run as a plain CI job with no new CRD.
2. **Controller-CRD stage (full System loop):** `SystemCandidate` (prompt-bundle version, model-routing policy, harness config, evaluator version, benchmark dataset version) + `SystemExperiment` (baseline, challenger, budget, promotion policy) as persisted, version-addressed CRDs, so a promotion decision is auditable and replayable rather than a one-off CI green checkmark. This stage depends on stage 1 existing (it operationalizes the same comparison, persistently) and is naturally later.

### (e) Oversight loop — risk/confidence/history + `ResolveLoopPolicy` extension

**The concrete, verified gap:** `LoopPolicy.Autonomy` (`api/v1alpha3/loop_types.go:57-60`, values `autonomous`/`supervised`) is declared on the shared contract but `ResolveLoopPolicy` (`dispatch_helpers.go:551-592`) never reads or stamps it — confirmed by reading the full function body, which resolves `Level`, `MaxIterations`, `EscalationPolicy`, `EvaluatorRef` and stops. This is exactly the "deferred `LoopPolicy.Autonomy` field is finally consumed" line from PROJECT.md's milestone framing — it is a real, already-scoped gap, not a hypothetical.

**Where risk/confidence/history should live — reuse existing precedent, don't invent a new persistence pattern:**
- **Static risk** (protected paths, high-risk-change classification) is a POLICY input, authored on `Project.Spec` — same shape as `FailureProfile`/`planAdmission.fileTouchMode`, which the codebase already treats as controller-read config, never derived (`five-loop-model.md:73-74` names `protectedPaths`/`highRiskChanges` explicitly).
- **Confidence** already exists as a per-run signal: `EvaluationSummary.FindingsCount`/`HighSeverityCount` (`api/v1alpha3/loop_types.go:170-180`) is the verifier's own confidence proxy today — Oversight's job is to fold this into a resolved autonomy/gate-strictness decision for the *next* dispatch, not invent a second confidence metric.
- **History must NOT accumulate in `LoopStatus`.** LOOP-03 is a structurally-guarded invariant (`loop_types.go:90-96`, "carries ONLY the current-iteration summary... never a slice/history," pinned by a compile-time guard test `TestLoopStatus_NoForbiddenFields`). Historical trend data belongs in the System loop's trace/eval store (candidate-versioned evidence, per `five-loop-model.md:71`: "every run records the exact prompt/model/harness/evaluator/policy versions used") — the Oversight resolver reads a pre-aggregated summary from there, the same way `LoopStatus.EffectiveMaxIterations` (`loop_types.go:107-118`) already stamps only the RESOLVED scalar from a precedence walk, never the walk's inputs.
- **The extension shape:** `ResolveLoopPolicy` gains an `Autonomy` resolution step alongside its existing `EscalationPolicy` step (same tier ordering: authored > chart > compiled default), but keyed additionally on a risk-tier input derived from `Project.Spec`'s policy fields + a bounded confidence/history summary — mirroring `EffectiveMaxIterations`'s "stamp the resolved scalar onto `LoopStatus`, not the inputs" pattern for a new `LoopStatus.EffectiveAutonomy`-shaped field.

### (f) Dynamic patterns — fan-out seams, ESC-04 bounding

**Which dispatch seams fan out N candidates/judges — mapped directly onto existing seams, not new ones:**

| Pattern | Dispatch seam | Mechanism |
|---|---|---|
| Adversarial verification (N independent refuters) | Verify seam — the SAME seam `ResolveLoopPolicy`/`LoopPolicy` already governs at every level | N sibling `JobKindVerifier` Jobs (mirroring `VerifierJobName`'s existing deterministic-name pattern, extended with a judge index) instead of one; each still counts individually against the existing `verifierInFlightCount` project-scoped pool (`dispatch_helpers.go:797-821`) — **additive, not new plumbing** |
| Generate-and-filter (N candidates judged) | Planner seams — the 4 `BuildPlannerEnvelope` call sites (`dispatch_helpers.go:632`) | N sibling `JobKindPlanner` Jobs with a divergence directive; a new merge/reduce step selects ONE candidate's `ChildCRDs` to materialize — **materializing more than one candidate's children would double-author the DAG, so the reduce step is load-bearing, not optional** |
| Tournament (cost-gated bracket) | Both seams combined (`N·(1+K)+1` dispatch shape) | Same fan-out+reduce primitive, gated by a budget **pre-flight** check before any sibling Job is created |

**The one genuinely new mechanism all three need:** today `1 CRD dispatch = 1 Job`. A fan-out pattern needs (1) a wait-for-fan-in sub-state that counts N sibling completions before the loop advances — this is a direct generalization of the EXISTING `ChildCount` race-free succession-gating pattern (`pkg/dispatch/envelope.go:301-309`, `TerminationStub.ChildCount`, "the controller compares observed owned children against this expected count and requeues until observed >= expected") applied to fan-out siblings instead of materialized children — and (2) a deterministic reduce (vote/quorum for judges; select-and-discard for generators). Neither requires a new CRD kind or a second scheduling engine; both compose with the existing informer-cache-based in-flight counting the codebase already uses for pool caps.

**ESC-04 bounding — reuse, don't duplicate:** `budget.ReservationStore` is already a single project-wide pool used by BOTH planner and verifier dispatch sites, explicitly documented as "the SAME `*budget.ReservationStore` instance... never a second counter that could undercount spend" (`dispatch_helpers.go:272-278`). A fan-out dispatch must **pre-charge N× the per-dispatch `ReserveEstimateCents` against this SAME store before creating any sibling Job** — for Tournament specifically, this means checking `HasHeadroom` (`internal/budget/reservation.go:127`) against the FULL `N·(1+K)+1` estimated cost as a single pre-flight gate, not per-Job as each sibling is created (a per-Job check could dispatch N-1 siblings before discovering the Nth has no headroom, leaving partial fan-out orphaned). Concurrency-wise, `verifierInFlightCount`/`plannerInFlightCount`'s existing project-scoped, role-labeled pool accounting (`dispatch_helpers.go:753-821`) already bounds how many of a fan-out's siblings can be in-flight simultaneously without any new pool type — the fan-out primitive just needs to request N slots from the existing pool instead of 1.

## Patterns to Follow

### Pattern: Extend a precedence-chain resolver, never add a parallel one
`ResolveProvider`/`resolveImage`/`ResolveLoopPolicy` are the three chokepoints. Every "where does config X live" question in this milestone (Vendor selection, Autonomy resolution) should extend one of these three, mirroring their existing `Levels.<key> → Project default → chart default → compiled fallback` shape. A fourth independent resolver is a maintenance smell this codebase has explicitly avoided at every prior phase (`dispatch_helpers.go`'s own package doc frames itself as the thing that prevents reconcilers from "drifting in lockstep").

### Pattern: New capability = existing seam + one new flag/field, not a new Job kind or CRD
The read-only verifier didn't need a new Job kind — it needed `BuildOptions.ReadOnly` on the existing `JobKindPlanner`/`JobKindExecutor`/new-`JobKindVerifier` triad. The write-capable authoring image needs the SAME triad with `ReadOnly: false`. Follow this precedent for every "does X need a new component" question in this milestone before reaching for one.

### Pattern: Structural enforcement over prompt enforcement
The read-only verifier's independence is structural — ReadOnly mount + credential omission, proven by a regression test (`TestBuildJobSpec_Verifier_NoGitCredsInAnyContainer`) — never prompt-enforced. Apply the same discipline to every new independence claim in this milestone (adversarial verifiers must not be able to see/influence each other's dispatch; a Product-loop evaluator must not hold the same credentials as the executor it judges).

## Anti-Patterns to Avoid

### Anti-Pattern: Materializing every fan-out candidate's ChildCRDs
Generate-and-filter's whole point is picking ONE candidate. Materializing all N candidates' `ChildCRDSpec` entries would silently multiply the DAG (N times the Tasks, N times the spend) and violate "waves are derived from a DAG the operator/planner authored," not from an ensemble. The reduce step that selects a champion is not an optimization — it's required for correctness.

### Anti-Pattern: Reopening the D-07 phase/milestone/project `MaxIterations` clamp for the Product loop
It is tempting to give the Product loop a repair branch at the project level so its verdict can drive iteration directly. Don't — D-07 is deliberate ("post-execution rework discards paid work"). Route Product-loop re-planning through NEW Milestone/Project authoring (a planning-DAG addition), not through reopening execution-level repair at a level that was locked shut for cost reasons.

### Anti-Pattern: Growing `LoopStatus` with history for Oversight's sake
The temptation is direct: "Oversight needs history, so let's add a `[]PastEvaluation` to `LoopStatus`." This is exactly what LOOP-03's compile-time guard test exists to prevent. History lives in the System loop's trace/eval store; `LoopStatus` gets one more RESOLVED scalar, never an accumulating slice.

### Anti-Pattern: Per-Job budget checks during fan-out creation
Checking `HasHeadroom` once per sibling Job as it's created (rather than once for the full fan-out width before creating any) can leave a partially-dispatched, partially-orphaned fan-out when budget runs out mid-creation. Pre-flight the WHOLE fan-out's estimated cost as one reservation before creating sibling #1.

## Scalability Considerations

| Concern | Today (single verifier per level) | Fan-out (Judge panel, width K) | Tournament (N·(1+K)+1) |
|---|---|---|---|
| Pod count per loop iteration | 1 | K | N·(1+K)+1 |
| Budget pre-charge | `ReserveEstimateCents` × 1 | `ReserveEstimateCents` × K, one pre-flight reservation | `ReserveEstimateCents` × (N·(1+K)+1), one pre-flight reservation |
| Pool accounting | `verifierInFlightCount`/`plannerInFlightCount`, project-scoped | Same counters, requesting K slots instead of 1 | Same counters, requesting up to N+K+1 slots across two role pools |
| Failure semantics | Loop escalates on the single verdict | Reduce step must define a partial-failure rule (quorum vs. unanimous) — a NEW policy surface, not free | Same, plus a tie-break rule for the merge stage |
| Cluster headroom | Proven safe (Phase 32 concurrency caps) | Needs a `maxFanOutWidth`-style ceiling (echoes the unrouted Sounding note's `soundingPolicy.maxShape`) so a single loop iteration can't OOM a single-node dev cluster the way Phase 32's dogfood run-2b defect did at ~60 concurrent planner pods | Same ceiling, sized more conservatively given the multiplicative cost |

## Suggested Build Order

Ordered by dependency, not by the milestone's letter order — several letters share one underlying mechanism and building it once first pays for the rest.

1. **`ProviderSpec.Vendor` precedence chain** (part of b). Prerequisite for everything else: without a real `Levels.<level>.Vendor` config seam, every subsequent rung migration repeats the verifier's inline-literal pattern a third and fourth time. Small, mechanical, mirrors `resolveImage` exactly.
2. **System loop CI-gate stage** (first half of d). "Each rung gated on eval evidence vs. the CLI baseline" is a hard dependency of the authoring ladder itself (per PROJECT.md's own framing) — build the candidate-vs-baseline comparison in `cmd/tide-eval` before migrating the first planner rung, not after.
3. **Fan-out + reduce dispatch primitive** (shared mechanism underlying b's shadow-mode and all of f). Build this ONCE — wait-for-N-siblings sub-state (generalizing the existing `ChildCount` gating pattern) + pre-flight N× budget reservation + a pluggable reduce function. This is the highest-leverage single build item in the milestone: it directly serves shadow/champion-challenger evaluation (b), adversarial verification, generate-and-filter, and tournament (f).
4. **Write-capable LangGraph authoring image, planner rungs first** (a). Depends on (1) for per-level selection and benefits from (2) being live so each rung's evidence gate has something real to check against. Planner rungs need no git-write path (they only emit `ChildCRDSpec`s) — structured-output child-CRD emission is the only new mechanism, and it's a direct reuse of the verifier's already-proven `ToolStrategy(PydanticModel)` pattern.
5. **Adversarial verification at verify seams** (first, simplest consumer of (3) in f). Verify seams already have per-level `LoopPolicy` + `verifierInFlightCount` pool accounting — extending width from 1 to K is close to purely additive.
6. **Generate-and-filter at planner seams** (second consumer of (3) in f). Needs the new "materialize only the champion's children" reduce step — genuinely new logic, not purely additive like (5).
7. **Executor rung of the authoring ladder** (closes a). Migrates last per the ladder's own stated reasoning (hardest parity bet); now needs the git-write path (local commit inside the LangGraph container, reusing `resolveAgentIdentity` + the existing `tide-push` remote-push seam unchanged).
8. **Tournament pattern** (hardest consumer of (3) in f — combines 5+6, adds budget pre-flight gating and a tie-break merge rule). Sequence last among the dynamic patterns so its budget/merge logic can reuse both simpler patterns' proven reduce steps.
9. **Product loop** (c).
