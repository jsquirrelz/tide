# Project Research Summary

**Project:** TIDE v1.0.9 — "Slack Tide": In-Cluster Verify Tier + LangGraph Beachhead
**Domain:** Adding a new read-only LLM verify tier (plan-check / level-verify / integration-check) and a second, Python/LangGraph subagent runtime behind an already-shipped Go/controller-runtime K8s orchestrator's existing `Subagent` seam
**Researched:** 2026-07-18
**Confidence:** HIGH (stack pins + TLS behavior verified against PyPI/source; architecture grep-confirmed against `main`; pitfalls grounded in TIDE's own incident history — Phase 25, Phase 32, Phase 34, Phase 45, the 2026-07-03 external run)

This is a subsequent-milestone synthesis. All four researchers converged on a shared architecture; this document exists to carry that convergence — plus the genuinely open calls — forward into requirements and roadmap without re-reading all four files.

## Executive Summary

Slack Tide closes a real, already-observed defect: the 2026-07-03 external run stamped a level `Complete` with a declared deliverable missing from the pushed branch and its pass criterion ("pytest green") never actually executed in-cluster — the only verifier in the loop was a human diffing `filesTouched` by hand. v1.0.7's Phase 34 already closed the *mechanical* half of this (git-verified merge completeness). Slack Tide ships the *semantic* half: three LLM-judged gates — plan-check (goal-backward, pre-dispatch), level-verify (real gate-command execution + deliverable/constraint check, post-execution), and integration-check (cross-level composition, milestone/project boundary) — each emitting a machine-enforced `gate_decision` (`APPROVED | REJECT | BLOCKED`) that the reconciler acts on directly, never a report a human might skip. It simultaneously seeds the LangGraph successor-runtime ladder: the verifier ships as a new, read-only Python/LangGraph container behind the **unchanged** `pkg/dispatch.Subagent` interface, proving a second concrete runtime works before any authoring role migrates.

All four research passes converge on the same shape with no material disagreement: `create_agent(response_format=GateDecision)` for the structured verdict; findings tagged with severity+confidence at generation time with filtering deferred to gate-policy config (the "coverage, not conservatism" split, directly countering the documented Opus-family failure mode of literally honoring "only flag high severity" and silently under-reporting); a new `ConditionVerifyHalt` project-level condition built by copying `failure_halt.go` file-for-file, including Phase 25's hard-won resume time-fence; prompts rendered orchestrator-side as a sixth Go template family (not ported into Python); and findings persisted via the same size×locality discipline already used everywhere else in TIDE — small `.status` summary, full findings on the git run branch via the v1.0.7 artifact-store mechanism, never a blob in etcd.

The primary risk is not the LLM-judging logic — it's the seams the new stage crosses that TIDE has been burned by before, each with a name and a prior incident: double span emission if the LangGraph image's vendor isn't registered in `SelfInstruments` before its first dispatch ships; an OOM if verify pods aren't counted against the Phase-32 concurrency gate (verify dispatches roughly double pod count at covered levels); a repeat of Phase 25's exact resume-ordering bug if `ConditionVerifyHalt` is written from a blank slate instead of copied from the existing halt-class pattern; and, worst of all, a fail-*open* interpretation of an unparseable `with_structured_output` result — which would silently reproduce the exact 2026-07-03 incident this milestone exists to close. One genuine unknown remains unresolved by any researcher and needs a dedicated early spike: whether `SSL_CERT_FILE` alone is sufficient to trust credproxy's CA through `langchain-anthropic`'s specific `ChatAnthropic`→`anthropic`-SDK→custom-`httpx`-transport construction path, given an open, unresolved upstream issue (`langchain#35843`) confirming there's no first-class custom-`httpx.Client` passthrough today.

## Key Findings

### Recommended Stack

Python 3.13-slim-bookworm (the highest version LangGraph 1.2.9's own PyPI classifiers declare support for) running `langgraph==1.2.9` + `langchain==1.3.14` + `langchain-anthropic==1.4.8` + `langchain-core==1.4.9` + `anthropic==0.117.0` + `pydantic==2.13.4` + `httpx==0.28.1` — every pin cross-checked directly against `requires_dist` PyPI metadata, zero version conflicts in the set. No checkpointer (stateless single-shot `.invoke()` per Job — resolves polyglot-doc assumption A4). No `deepagents`, no `GitPython`, no `langchain-mcp-adapters` — hand-authored `@tool`-wrapped `subprocess` calls for git-read and the bash gate-command are sufficient and match the existing Go-side "shell out to git" convention.

**Core technologies:**
- `langgraph` 1.2.9 — agent orchestration / conditional loop runtime — gives the plan→act→self-check→retry loop natively instead of hand-building one
- `langchain` 1.3.14, `create_agent(..., response_format=GateDecision)` — current (2026) idiomatic entry point combining the tool-using loop and the terminal structured verdict in one call
- `langchain-anthropic` 1.4.8 (`ChatAnthropic`) — exposes `effort`, `thinking`, `temperature`/`top_p`/`top_k`, and raw `cache_control` passthrough (including a purpose-built `AnthropicPromptCachingMiddleware`) — the exact levers the dead Go-side `Provider.Params` allowlist has been waiting for, now reachable from a different language
- `pydantic` 2.13.4 — `GateDecision`/`Finding` schema definitions, required transitively anyway

**A1 correction (load-bearing, carry forward verbatim):** `httpx` honors **only** `SSL_CERT_FILE` (or `SSL_CERT_DIR` for a cert directory), gated on `trust_env`/`verify` defaults (both `True`) — confirmed by reading `httpx/_config.py::create_ssl_context` directly. **`REQUESTS_CA_BUNDLE` is a `requests`-library convention that httpx does not read at all** — no such key anywhere in `_config.py`, and none of `anthropic`/`langchain-anthropic`/`langgraph` pull in `requests`. Drop `REQUESTS_CA_BUNDLE` from the contract-conformance table; it is not an "either/or" with `SSL_CERT_FILE`, it is dead. Set `SSL_CERT_FILE=/etc/tide/proxy/ca.crt` (same mount path pattern the CLI image already uses for `NODE_EXTRA_CA_CERTS`) — one env var, no code.

**However — flag this as an early de-risking spike, not a settled fact.** The Anthropic Python SDK (which `ChatAnthropic` wraps) constructs its own internal `httpx.Client` via a custom transport (`_DefaultHttpxClient`), and while STACK.md's direct source read found this transport still falls through to httpx's own `trust_env`/`verify` defaults (making `SSL_CERT_FILE` sufficient in principle), PITFALLS.md independently surfaced two corroborating signals that this exact path has a documented history of *not* honoring env-driven behavior cleanly: an open upstream report (`anthropic-sdk-python#923`) and an unresolved `langchain-ai/langchain` feature request (`#35843`) asking for `ChatAnthropic` to accept a custom `httpx.Client`/`httpx.AsyncClient` for exactly this class of problem — its own text calls today's workaround "monkey-patching... fragile." **Net: schedule a live pass/fail spike (real `ChatAnthropic(...).invoke()` call through the credproxy sidecar's self-signed CA) as the very first task of the image-build phase**, before any verify-stage business logic is written on top of outbound LLM calls. If it fails, the fallback is constructing the underlying `anthropic.Anthropic` client with an explicit `http_client=` and passing it into `ChatAnthropic(..., anthropic_client=...)` (if that constructor arg exists at the pinned version) — a real code path, not a config tweak, so budget schedule slack for it rather than discovering it mid-build.

### Expected Features — The Three Stages

All three stages share ONE verdict schema (`gate_decision`, `stage`, `level`, `findings[]` with `dimension`/`severity`/`confidence`/`finding`/`evidence`/`suggested_fix`, `summary`) — modeled on GSD's plan-checker/verifier YAML blocks plus the LLM-as-judge `{verdict, confidence, evidence, reason}` convention, extended with TIDE's own severity taxonomy (`BLOCKER`/`WARNING`/`INFO`).

- **`gate_decision` is 3-valued, not boolean:** `APPROVED` / `REJECT` / `BLOCKED` — because the two failure paths route to materially different reconciler actions. `REJECT` (plan-check only) routes to a **bounded re-plan loop**: re-dispatch the SAME planner template with findings appended verbatim, up to N attempts (config), with **stall detection** (if BLOCKER+WARNING count doesn't strictly decrease between attempts, halt immediately rather than burn remaining attempts) — directly modeled on GSD's Check-Revise-Escalate pattern. `BLOCKED` (level-verify / integration-check) is **terminal** — fires `ConditionVerifyHalt` immediately, no auto-retry, because post-execution rework discards already-paid-for work and that call belongs to the operator (Pillar 3).
- **Coverage-not-conservatism enforcement split (carry forward verbatim):** the verifier's prompt instructs it to emit a `finding` entry for *every* deviation it observes, tagged with `severity`+`confidence` — it never decides on its own to omit a low-severity finding. Gate **policy** (config, read by the reconciler) is what decides which severities force `REJECT`/`BLOCKED`. This is the explicit, deliberate fix for the documented Opus-family failure mode (CLAUDE.md's own subagent-tuning note): told "only flag high-severity," these models comply literally and silently drop real lower-severity findings. Splitting "observe and tag everything" (model) from "decide what blocks" (reconciler config) removes the model's ability to under-report by instruction-following too well.
- **plan-check** — goal-backward, pre-dispatch, Plan level only. Rubric: goal_alignment, file_touch_plausibility, dependency_correctness, verification_derivability (flags unfalsifiable `echo "done"`-style acceptance criteria as a BLOCKER). Cheapest place in the pipeline to reject — no task has spent yet.
- **level-verify** — post-execution, pre-Succeeded/pre-boundary-push, at Plan/Phase/Milestone. Runs the declared gate command for **real** via bash in a checked-out worktree and parses the actual exit code (the direct fix for the incident); confirms every declared deliverable exists on the run branch (not the ephemeral task PVC); confirms "do-not-touch" constraint paths held. Runs alongside/after Phase 34's mechanical `merge-base --is-ancestor` gate — that gate is structural (is the work merged), this stage is semantic (does the merged work satisfy the level).
- **integration-check** — milestone/project boundaries. Builds/tests the *entire* run branch as a whole, checks cross-level composition claims (does phase A's output actually get consumed by phase B) — the same bug class as the 2026-07-03 incident, one level up. Locked in scope for this milestone (Pillar 1 names all three stages) despite being the highest-complexity item.

**Explicitly out of scope for this milestone (anti-features):** auto-fixing findings, the verifier re-authoring a rejected plan directly, unbounded/silent retry-until-pass, auto-clearing a BLOCKED verdict to preserve throughput, flake-tolerant gate-command execution (repo doctrine is NO flake tolerance anywhere in verification), trusting the model's textual claim about test outcome instead of executing it, and scope creep into general code-review/security-audit (explicitly deferred to a later "quality tier" per the seed's staged sequencing).

Full detail: `.planning/research/FEATURES.md`.

### Architecture Approach — Resolved Recommendations (treat as strong priors)

All architecture questions the milestone doc left open were answered by direct grep against `main`, not left for requirements to re-litigate:

1. **Verifier prompt source = orchestrator-side Go template.** A new `role="verifier"` template class, rendered by the manager exactly like the 5 existing templates via `common.LoadPromptTemplate`, passed into `EnvelopeIn.Prompt` verbatim — the same channel `BuildPlannerEnvelope` already uses. Wins over an in-image Python template on every axis: zero prompt drift (one source of truth), no sixth authoring mechanism in a second language/toolchain, and it's what the milestone doc itself was already leaning toward. LangGraph's `create_agent`/`response_format` operates on the *output* schema, not input prompt construction — there's no LangGraph-side fit advantage to authoring prompts in Python.
2. **integration-check = level-verify parameterized, not a distinct template.** `gates.BoundaryDetected` is already the single function serving all 4 levels' "children done" check with `childKind` as the only parameter — the verify tier mirrors this. level-verify and integration-check differ in *scope* (one level's artifact vs. cross-sibling composition + whole-branch build), not in *where*/*when* they hook (both fire at the identical `BoundaryDetected==true` edge, before the terminal stamp). **Recommend exactly 3 template files, keyed by STAGE not CRD level:** `plan-check_verifier.tmpl`, `level-verify_verifier.tmpl` (serves Plan/Phase/Milestone), `integration-check_verifier.tmpl` (serves Milestone/Project) — avoiding a 4-level × 3-stage combinatorial explosion.
3. **Findings persistence follows the existing size×locality rule exactly, reusing the v1.0.7 git-artifact-store mechanism.** Tiny cross-namespace summary (`GateDecision`, `FindingsCount`/`HighSeverityCount`) on `TerminationStub` (≤4KB cap); full findings list on the per-Project PVC `out.json` (read in-namespace by the reporter Job, same actor that already materializes planner children); a small `VerifyResult` status struct (`Decision`, counts, `Stage`, `CompletedAt`) on the level CRD for dashboard/CLI queryability; and the **full findings artifact staged onto the run branch** at `.tide/planning/<kind>/<name>/verify-<stage>.json`, extending `collectStageEnvelopes` (not replacing it) — this surfaces on the dashboard for free via the existing `gitfetch`/artifacts API with zero new endpoint. Never a blob in etcd; never a new PVC path; never a bigger termination message.
4. **`ConditionVerifyHalt` mirrors `failure_halt.go` file-for-file, including Phase 25's time-fence, and gates BOTH the planner tier AND the task tier** — a deliberate deviation from `FailureHalt`, which is execution-only by design (its own doc comment explains why planning stays ungated for FailureHalt). VerifyHalt is different: a BLOCKED verify means the artifact tree itself is suspect, so new planning built on top of unverified artifacts compounds the problem. Add `checkVerifyHalt` to both `checkDispatchHolds` (planner tier) and `TaskReconciler.gateChecks` (task tier), and say so explicitly in the implementing plan — don't silently copy FailureHalt's planner-exemption.
5. **Config = a new chart-first `subagent.verify` block** (image/model/`stages.{planCheck,levelVerify,integrationCheck}.{enabled,maxAttempts,levels}`), following the existing `subagent.levels`/`resolveImage` precedence chain (Project override → chart default → off). **Default posture: milestone+project-only** — level-verify at Milestone, integration-check at Milestone+Project, plan-check off by default. Full `off` is the safe default for **existing installs upgrading in-place** (a verify tier that silently starts spending money on `helm upgrade` violates least-surprise as much as the "gate policy stays in config" principle); milestone+project-only is the recommended default for **new installs** — bounded, small dispatch count, still catches the exact bug class that motivated this milestone.
6. **Verifier pods MUST count against the Phase-32 concurrency gate before this ships.** Either extend `plannerInFlightCount`'s query to include the verifier Job label, or add a dedicated `verifierInFlightCount`/`VerifierPool.Capacity()` mirroring Phase 32's exact shape. Do not assume the existing planner cap "just also" covers a label it was never queried against — this is precisely the shape of the run-2b D3 defect (a dispatch path shipped before its concurrency accounting existed).

Full detail: `.planning/research/ARCHITECTURE.md`.

### Critical Pitfalls (see "Watch Out For" below for the top 4 with owning phases)

Full detail, all 11 pitfalls with prevention steps and phase mapping: `.planning/research/PITFALLS.md`.

## Watch Out For

The pitfalls researcher identified 11 concrete failure modes, each traceable to a specific TIDE architectural seam or prior incident. The four most consequential, distilled with prevention and owning phase:

1. **Double span emission** — `openinference-instrumentation-langchain` auto-instruments the LangGraph process and emits OpenInference spans natively; TIDE's reporter *also* synthesizes spans from `events.jsonl` unless `pkg/dispatch.SelfInstruments(vendor)` returns `true` for the verifier's vendor. Today it fails closed (`false`) for every known and unrecognized vendor — the right default for an unannounced vendor, but it means the verifier does **not** get skip-synthesis for free. Someone must add a real vendor literal and flip it to `true`, AND thread it through `ResolveProvider(...).Vendor` at the verifier's dispatch call site. **Prevention:** land the vendor-registration change in the SAME phase that first wires the verifier's dispatch call sites — never sequence "ship the verifier" before "register its `SelfInstruments` vendor." Verify with a live dispatch against a real OTLP collector asserting exactly one span tree (the Phase 45 stub test proves the seam compiles, not that the real third-party instrumentation behaves as assumed). Answered placement question: the verifier gets a **new, distinct `EVALUATOR` span kind** (OpenInference's own semantic convention for "a call to a function... performing an evaluation of outputs"), parented as a **sibling** to the checked level's own `AGENT` span — not nested inside it, since the verify dispatch is a genuinely separate, read-only, post-hoc Job.

2. **Read-only enforcement must be structural, not prompt-only** — the verifier ships with the same git+bash primitives an authoring image uses to commit/push; a prompt saying "never commit" is a request to an LLM, not an enforcement mechanism, and repo content the verifier reads is itself untrusted (prompt-injection surface). **Prevention (structural, not prompt):** mount the verifier's worktree `ReadOnly: true` (mirrors the existing `ReadOnly: true` pattern already used for the credproxy cert mount) with a separate ephemeral `emptyDir` for any incidental scratch writes; **never inject git-write/push credentials into the verifier's env at all** — omission as enforcement, not permission-checking; wire no child-CRD-authoring consumption path on the manager side at all, so an errant write is inert rather than merely against policy. Verify with an adversarial fixture test: have the LangGraph agent loop *attempt* `git commit`/push against a fixture worktree and assert it fails at the mount/credential layer. Owning phase: the same phase that authors the image's Job-spec contract (Pillar 2) — not a later hardening pass.

3. **Fail-CLOSED structured output, never fail-open** — `with_structured_output`/`create_agent`'s `response_format` can come back empty, partially validated, or with an unparseable/missing `gate_decision` field under transient LLM API issues or truncation. If the reconciler's fallthrough branch treats "no clear verdict" as "not BLOCKED, so proceed," that **exactly reproduces the 2026-07-03 silent-`Complete` incident this entire milestone exists to fix** — arguably worse, since it gives false confidence that verification ran when it didn't. **Prevention:** model the parse outcome as a tri-state (`APPROVED` / `BLOCKED` / `UNPARSEABLE-OR-MISSING`) and route the third state to the SAME `ConditionVerifyHalt` path as an explicit BLOCKED — never collapse it into APPROVED. Bound LangChain's own internal retry-on-validation-failure loop explicitly rather than letting it run unbounded (this compounds the cost/concurrency pitfall below). Test against fixture responses that are empty JSON, valid-but-missing-the-verdict-field, and genuinely malformed — all three must reach the halt path. Owning phase: lands with the reconciler-side `gate_decision` consumption logic — this is the core safety property of the entire tier, a DoD item, not a polish item.

4. **Cost × concurrency, not just cost × budget** — a verify stage at every covered lifecycle seam roughly *doubles* dispatch count for the levels it covers. The existing `BudgetConfig`/`ReservationStore` absorbs the $ side, but Phase 32's `plannerInFlightCount` concurrency gate was sized and scoped for the planner pool specifically — a new, uncounted dispatch class reproduces the exact run-2b D3 OOM class (single-node kind cluster OOM'd at ~60 in-flight planner pods) at the same scale, now doubled by concurrent authoring + verify pods in the same wave. **Prevention:** verify dispatches must be counted against a concurrency gate (extend `plannerInFlightCount` or add a dedicated `verifierInFlightCount`) in the SAME phase the verifier dispatch call sites land — not a follow-up patch — plus the conservative default posture (milestone+project-only, off at task tier) bounds the multiplier at the widest-fanout tier by default. Verify with a kind-cluster wave-load test dispatching authoring + verify pods concurrently, confirming pod count stays under the sized cap.

Two more worth naming explicitly since they compound with the above: **`ConditionVerifyHalt` must not repeat Phase 25's exact resume-ordering bug** (copy `failure_halt.go`/`billing_halt.go` file-for-file including the time-fence — this is the THIRD instance of this exact halt-condition pattern, and the bug is now a documented, known failure mode, not a novel risk); and **a BLOCKED verify must be a new halt class, never a reinterpretation of task/wave failure** (reusing the `Failed`-phase code path would incorrectly trigger sibling-continues/dependents-never-dispatch wave semantics for what is actually a project-wide gate — write a regression test asserting the checked level's phase, wave siblings, and conservative-profile propagation are NOT touched by a VerifyHalt).

## Implications for Roadmap

The research converges on a natural phase order driven by dependency, not by stage-number: de-risk the new runtime's trust seam first, lock the schema/persistence contract before any halt logic touches it, then land each stage with its concurrency/tracing/halt accounting *in the same phase* as its dispatch call sites (per the pitfalls researcher's repeated "don't sequence this as a follow-up" finding).

### Phase 1: LangGraph Image Beachhead + Credproxy TLS Spike
**Rationale:** Every other phase's stages dispatch through this image; the TLS trust question (A1/langchain#35843) and version-pin discipline are both genuine unknowns that gate everything downstream — de-risk before building stage logic on top of unverified plumbing.
**Delivers:** Minimal read-only Python/LangGraph container (git-read + bash-gate-command tools only, no file-edit/commit/push tools, no checkpointer) behind the unchanged `pkg/dispatch.Subagent` seam; live pass/fail proof that `SSL_CERT_FILE` alone trusts credproxy's CA through the actual `ChatAnthropic` construction path (or the `http_client=`/`anthropic_client=` fallback, if the spike fails); patch-exact dependency pins + a CI gate rejecting unpinned/range specifiers.
**Addresses:** FEATURES.md's "Read-only LangGraph specialist image" (P1); STACK.md §2/§3/§5.
**Avoids:** PITFALLS #2 (credproxy TLS), #3 (version churn), #4 (read-only enforcement — the `ReadOnly` volume mount + credential omission belong in this phase's Job-spec work, not a later hardening pass).

### Phase 2: Verify Schema, Envelope, and Findings Persistence Contract
**Rationale:** The `VerifyContext` envelope shape, the `gate_decision` schema, and the findings size×locality split are schema decisions that must be locked BEFORE any halt-condition or reconciler logic is written on top of them (PITFALLS #7 explicitly calls this out as a pre-halt-logic dependency).
**Delivers:** `VerifyContext` pointer field on `EnvelopeIn` (mirrors `DispatchMeta`/`Dev`); `GateDecision`/`Finding` Pydantic+Go schema pair; `TerminationStub` additions (`GateDecision`, counts, ≤4KB, size-tested like `TestNewTerminationStub_StaysSmall`); extended `collectStageEnvelopes` staging a `verify-<stage>.json` findings artifact onto the run branch via the existing v1.0.7 git-artifact-store path; small `VerifyResult` status struct per level CRD.
**Uses:** ARCHITECTURE.md recommendation #3 (findings persistence); STACK.md's `GateDecision`/`Finding` Pydantic shape.
**Implements:** the coverage-not-conservatism enforcement split (tag everything at generation time; filter at gate-policy/config time) — this is a schema/prompt-contract decision, lands here even though enforcement logic comes later.

### Phase 3: level-verify + `ConditionVerifyHalt`
**Rationale:** level-verify is the single highest-value stage (closes the headline incident directly — the declared gate command actually runs in-cluster) and is architecturally the simplest of the three (parameterizes cleanly across Plan/Phase/Milestone via one template). Building `ConditionVerifyHalt` here, by copying `failure_halt.go` file-for-file including the Phase 25 time-fence, means the halt-condition correctness contract lands with its first real consumer rather than being retrofitted.
**Delivers:** level-verify dispatch at the `BoundaryDetected==true` edge (before `patch*Succeeded`) for Plan/Phase/Milestone; real gate-command execution + deliverable-existence + do-not-touch constraint checks; `ConditionVerifyHalt` gating BOTH planner and task tiers; extended `tide resume` verb with the reset-then-clear-then-time-fence ordering; verifier dispatches counted against the (possibly newly extended) concurrency gate; `SelfInstruments` vendor registration for the LangGraph runtime, landed in this same phase as the first real dispatch call sites.
**Implements:** ARCHITECTURE.md recommendations #4 (VerifyHalt) and #6 (concurrency gate); the fail-closed tri-state verdict handling (PITFALLS #9).
**Avoids:** PITFALLS #1 (double span emission — vendor registration ships with the dispatch sites, not after), #5 (cost×concurrency — gate extension ships with the dispatch sites, not after), #8 (resume-ordering bug — copied, not reinvented), #9 (fail-open verdict handling), #11 (BLOCKED must not reuse `Failed`-phase wave semantics).

### Phase 4: plan-check + Bounded Re-Plan Loop
**Rationale:** Sequenced after level-verify/`ConditionVerifyHalt` exist, because plan-check's escalation-after-N-attempts path terminates into the SAME halt class — reusing rather than duplicating halt infrastructure. plan-check is architecturally distinct (Plan level only, a retry loop rather than a terminal halt on first BLOCKED) and needs its own attempt counter, so it's cleanly separable as its own phase.
**Delivers:** plan-check dispatch at the `handlePlannerJobCompletion`/`ValidationState` edge; goal-backward rubric (goal_alignment, file_touch_plausibility, dependency_correctness, verification_derivability); bounded re-plan loop with its OWN counter (`Plan.Status.PlanCheckAttempts`, `Project.Spec.MaxPlanCheckAttempts`, default 1) and stall detection (non-decreasing BLOCKER+WARNING count halts early); escalation to `ConditionVerifyHalt` on exhaustion.
**Addresses:** FEATURES.md's plan-check + bounded re-plan loop (P1).
**Avoids:** PITFALLS #6 (unbounded/ambiguous re-plan loop — own counter, not `maxAttemptsPerTask`).

### Phase 5: integration-check + Chart Config Surface + Dashboard Surfacing
**Rationale:** integration-check is the highest-complexity stage (whole-run-branch build/test, cross-sibling composition) and reuses level-verify's dispatch machinery almost entirely (per ARCHITECTURE.md's "parameterized, not distinct" finding) — sequencing it last lets it inherit a proven halt/concurrency/persistence stack rather than build its own. The chart-first `subagent.verify` config block and dashboard distinctions (VerifyHalt vs. Failed, findings display via existing gitfetch) are natural closing work once all three stages exist to configure/display.
**Delivers:** integration-check dispatch (Milestone/Project boundaries, whole-branch build/test); `subagent.verify` chart block with per-stage enable/attempts/levels and the milestone+project-only default posture (new installs) / off (in-place upgrades); dashboard surfacing of `VerifyHalt` as a visually distinct state from task-execution `Failed`; findings artifact browsing via the existing gitfetch/artifacts API (no new endpoint).
**Addresses:** FEATURES.md's integration-check (P1, locked in scope) and per-level stage-dispatch config (P1).
**Uses:** ARCHITECTURE.md recommendations #2 (parameterized template), #5 (chart config + default posture).

### Phase Ordering Rationale

- **Trust-seam and schema work come before any stage logic** — both the TLS spike and the findings/envelope schema are load-bearing dependencies every stage phase builds on; getting either wrong after stage logic exists means rework across all three stages, not one.
- **level-verify before plan-check** despite plan-check's cheaper cost-per-reject: level-verify is architecturally simpler (no retry loop, parameterizes trivially across 3 levels) and is where `ConditionVerifyHalt`'s correctness contract (Phase 25 time-fence, wave-semantics isolation) needs to be proven once, correctly, before plan-check's escalation path reuses it.
- **integration-check last** because it shares level-verify's machinery almost entirely (one stage-keyed template family, same dispatch edge) — building it after de-risks nothing new architecturally, it's mostly configuration + a heavier rubric.
- **Cross-cutting safety work (concurrency gate extension, `SelfInstruments` vendor registration, fail-closed verdict handling) is written into the phase that FIRST wires real dispatch call sites, never deferred to a follow-up phase** — this is the single most repeated instruction across all four research files, traced directly to the run-2b D3 and Phase 45-stub-vs-real-image precedents of "the seam existed, but a new consumer wasn't wired into it."

### Research Flags

Needs deeper research during `/gsd:plan-phase --research-phase <N>`:
- **Phase 1 (LangGraph image beachhead):** the TLS spike outcome is genuinely unknown until executed live — if `SSL_CERT_FILE` alone doesn't suffice through `ChatAnthropic`'s construction path, the fallback shape (`http_client=`/`anthropic_client=` or dropping to raw `anthropic-sdk-python`) needs to be planned as a real contingency, not discovered mid-build.
- **Phase 3 (level-verify + ConditionVerifyHalt):** the `GateCommand` schema source is a genuinely new requirement with no source in the codebase today — where a level declares its "pass criterion command" (a new `Plan.Spec`/`Project.Spec` field? a Makefile/package.json convention?) needs a requirements decision, not just an architecture recommendation.
- **Phase 3 (SelfInstruments vendor registration):** whether the LangGraph runtime needs a genuinely new vendor sentinel (e.g. `"langgraph"`) or can reuse `"anthropic"` with a runtime discriminator is unresolved — affects `pkg/dispatch/vendor_capabilities.go` and every `ResolveProvider(...).Vendor` call site.

Phases with standard, already-resolved patterns (research-phase can be skipped or kept light):
- **Phase 2 (schema/persistence):** the size×locality split, envelope pointer-field pattern, and `TerminationStub` size-testing convention are all direct reuse of proven v1.0.7/Phase-44 mechanisms — architecture research already fully specified the shape.
